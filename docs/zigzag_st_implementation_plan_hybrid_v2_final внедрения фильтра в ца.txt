## 2. Runtime audit and insertion point

### 2.1 Correct insertion point

Фильтр внедряется **внутри backtest execution path до returns/equity/trades/metrics**.

```text
BEFORE:
  trend -> generate_positions -> positions
        -> calculate_returns -> equity -> extract_trades -> metrics

AFTER (enabled):
  trend -> ST flips + ZigZag ST FSM -> filtered_positions
        -> calculate_returns -> equity -> extract_trades -> metrics

AFTER (disabled, baseline-compatible):
  trend -> generate_positions -> positions
        -> calculate_returns -> equity -> extract_trades -> metrics
```

Strictly forbidden: post-фильтрация `trades_df` после расчёта returns/equity. Это десинхронизирует bar-level экономику и trade-level rows (Appendix A v1.1 §10, §16).

### 2.2 Implementation invariant

Когда `trade_filter.enabled=true`, ZigZag ST FSM является behavioral source of truth для `filtered_positions`. Реализация как post-mask поверх raw `generate_positions` запрещена, потому что не выражает `ST_STOPPING` exit-only, freeze-counter, repeated-trigger suppression и lifecycle-aware position changes.

Acceptance gate в WP7: тест проверяет, что в enabled path `generate_positions` **не вызывается**.

`Spec reference: Appendix A v1.1 §10, §16, §17.13`

### 3.3 Shared close-only ZigZag formula (§3.3 plan)

Зафиксировано в plan (полностью соответствует Appendix A v1.1 §3.4):

1. **Inputs leg-detection**: single price stream `close`. В Phase 1 нет high/low wick-based pivots и нет `hlc3`/`ohlc4`.
2. **Pivot reversal**: новая ZigZag pivot подтверждается, когда close разворачивается от текущего candidate extremum минимум на `zigzag.reversal_threshold` (numeric fraction of price).
3. **Confirmed-leg height**:
   `height_pct = abs(end_pivot_price - start_pivot_price) / start_pivot_price`,
   единица — доля цены.
4. **Candidate-leg height**:
   `candidate_height_pct[t] = abs(current_candidate_extremum_price - last_confirmed_pivot_price) / last_confirmed_pivot_price`,
   та же шкала, что у explicit `candidate_trigger_threshold`.
5. **Median inputs**:
   - `global_median` — full-dataset confirmed-leg heights;
   - `local_median_N` — последние `N = zigzag.local_window` confirmed legs в **causal slice-local runtime history**, не ограниченные ногами после lifecycle start.
6. **Freeze counter**: `freeze_confirmed_legs` считает confirmed legs **после lifecycle start**, независим от `local_window`.
7. **Edge cases**:
   - `last_confirmed_pivot_price <= 0` unreachable после OHLC validation; если встретилось — fail-closed внутренней ошибкой;
   - identical consecutive close → zero candidate movement;
   - `reversal_threshold` никогда не auto-calibrated, берётся из config.
8. **Shared test fixture**: одна fixture определяет `close[]`, `reversal_threshold`, ожидаемые confirmed legs, `candidate_height_pct[]`, `global_median`. Используется в WP3, WP4, WP5.

`Spec reference: Appendix A v1.1 §3.1, §3.2, §3.4, §5, §6, §14.1..§14.5`

## 4. WF / OOS / prepend / force-flat risk gate

### 4.1 Canonical OOS path

```text
test_start, test_end <- WFWindowSlice
prepend_applied = min(prepend_bars_requested, test_start)
ext_start = test_start - prepend_applied
ext_arrays = full[ext_start:test_end]
run_single_backtest(ext_arrays, extract_trades_flag=True, trade_filter_config=..., zigzag_global_stats=..., global_offset=ext_start)
oos_boundary = prepend_applied
trim returns/equity/positions/filter_diagnostics [oos_boundary:]
force-flat carry-in trades on OOS-side bar economics; filter_diagnostics НЕ обнуляются
calculate_all_metrics(trimmed, warmup_period=0)
filter trades (entry_index >= oos_boundary), rebase indices
attach trade-level diagnostics (см. §4.4)
trade-level override metrics
```

### 4.2 Where the FSM lives

```text
FSM slice (OOS)   = extended slice = prepend + OOS
FSM slice (train) = train slice
```

FSM на post-trim OOS-only запрещён: создаст cold-start artifact для ZigZag и сломает alignment с SuperTrend warmup/prepend.

### 4.3 How full-dataset stats connect to slice execution

`ZigZagGlobalStats` — calibration baseline:
- `global_median` — full-dataset;
- `candidate_trigger_threshold` — config или materialized из full-dataset (§12);
- `local_median_N` и `candidate_height_pct` — causal внутри executed slice;
- `global_offset` — только metadata/diagnostics; запрещено использовать для копирования full-dataset confirmed legs в slice-local rolling state.

### 4.4 Diagnostics rebasing and force-flat

Per-bar diagnostics порождаются на extended slice:

```text
len(filter_diagnostics[key]) == len(ext_positions)
```

После OOS trim:

```text
filter_diagnostics_oos[key] = filter_diagnostics_ext[key][oos_boundary:]
```

Force-flat зануляет OOS bar-level returns/positions/equity у carry-in, **но** не модифицирует `filter_diagnostics`. Diagnostics остаются inspectable.

Trade-level diagnostics:
- attach происходит **до** OOS trim/rebase, на extended-slice индексах;
- после фильтрации (`entry_index >= oos_boundary`) и rebase колонки `entry_filter_state`, `entry_trigger_source`, `exit_reason` остаются прикреплёнными к trade rows;
- carry-in rows фильтруются вместе с обычными trade rows.

### 4.5 Desync risks and contracts

| Риск | Контракт |
|---|---|
| `filtered_positions` ≠ `returns` | `calculate_returns` всегда потребляет `filtered_positions`. |
| `trades_df` описывает другой decision stream | `extract_trades` всегда вызывается с теми же `positions=filtered_positions`. |
| Bar diagnostics и trade diagnostics не согласованы | Trade diagnostics берутся из bar diagnostics на entry/exit индексах extended slice. |
| Force-flat прячет filter decisions | Force-flat не модифицирует diagnostics arrays. |
| Train и OOS semantics drift | Train: те же FSM-правила, без prepend, без force-flat. |
| `global_median` становится per-WF-step | `ZigZagGlobalStats` считается один раз в orchestrator. |
| `ST_STOPPING` открывает реверс | `ST_STOPPING` exit-only по обратному ST flip; никаких новых entries/реверсов. |

`Spec reference: Appendix A v1.1 §10, §12, §13, §14, §15.7, §17`

## 5. FSM contract

### 5.1 States

```text
OFF
WAIT_FIRST_ST_FLIP
ST_ACTIVE_FREEZE
ST_ACTIVE_MONITORING
ST_STOPPING
```

### 5.2 Transitions

```text
OFF
  -- A: triggers.candidate_threshold.enabled AND candidate_height_pct >= candidate_trigger_threshold
  -- B: triggers.confirmed_median.enabled AND confirm_bar AND local_median_N >= global_median
  --> WAIT_FIRST_ST_FLIP

WAIT_FIRST_ST_FLIP
  -- next ST flip allowed by current trade_mode
       (trade_mode=long  -> only long flip;
        trade_mode=short -> only short flip;
        trade_mode=both/revers -> any flip;
        0 -> ±1 не считается flip)
  --> ST_ACTIVE_FREEZE

ST_ACTIVE_FREEZE
  -- accumulated freeze_confirmed_legs confirmed ZigZag legs after lifecycle start
  --> ST_ACTIVE_MONITORING

ST_ACTIVE_MONITORING
  -- on confirm_bar:
       valid local_median_N AND local_median_N <  global_median  -> ST_STOPPING
       unavailable / NaN / Inf local_median_N                    -> ST_STOPPING (fail-closed)
       valid local_median_N AND local_median_N >= global_median  -> stay

ST_STOPPING
  -- no open position OR open position closed by nearest opposite ST flip --> OFF
```

`Spec reference: Appendix A v1.1 §4, §5, §6, §7, §8, §9, §15.7`

### 5.3 State behavior table

| State | Новый ST entry | Reverse | Median stop check | Exit |
|---|---|---|---|---|
| `OFF` | запрещён | запрещён | нет | n/a |
| `WAIT_FIRST_ST_FLIP` | только первый ST flip, разрешённый `trade_mode` | n/a | нет | n/a |
| `ST_ACTIVE_FREEZE` | по обычной ST-логике, ограниченной `trade_mode` | ditto | только diagnostic, не выключает режим | normal ST |
| `ST_ACTIVE_MONITORING` | по обычной ST-логике, ограниченной `trade_mode` | ditto | confirm-bar only; unavailable/NaN/Inf median = fail-closed → `ST_STOPPING` | normal ST |
| `ST_STOPPING` | запрещён | запрещён | нет | только обратный ST flip закрывает текущую позицию |

`Spec reference: Appendix A v1.1 §8, §9, §14, §15.7`

### 5.4 Why entry mask is insufficient

`ST_STOPPING` нуждается в текущем filtered position state. Per-bar boolean mask не различает entry vs reverse, exit-only opposite flip vs реверс, no-position immediate `OFF`, freeze counters, repeated-trigger suppression. Поэтому `zigzag_st_filter.apply(...)` строит `filtered_positions` сам как stateful pass.

`Spec reference: Appendix A v1.1 §4, §8.5, §9, §14.7..§14.10`

### 5.5 Event-order rule for `OPEN_TO_OPEN`

Canonical Phase 1 rule (соответствует Appendix A v1.1 §10 и §15.4):

```text
Все ZigZag filter conditions и ST flip events оцениваются на close бара.

Если на decision bar t возникли trigger A или B и allowed ST flip:
  - filter trigger обрабатывается первым;
  - same-bar allowed ST flip МОЖЕТ быть использован как первый разрешённый flip;
  - resulting position change исполняется на open(t+1);
  - сделка не открывается на close(t).

Если same-bar ST flip запрещён текущим trade_mode:
  - flip пропускается;
  - lifecycle не стартует;
  - state остаётся WAIT_FIRST_ST_FLIP.

ST flip 0 -> ±1 считается initialization, не tradable flip:
  - не открывает позицию в WAIT_FIRST_ST_FLIP;
  - не закрывает позицию в ST_STOPPING.
```

Это правило обязательно покрыто unit-тестами, включая same-bar trigger+allowed-flip и same-bar trigger+disallowed-flip.

`Spec reference: Appendix A v1.1 §3.3, §10, §15.4, §17.13, §17.14`

### 6.3 Schema changes (`wf_grid.config.schema`)

Новые dataclasses:

```text
TradeFilterZigZagConfig
TradeFilterTriggersConfig
TradeFilterTriggerToggleConfig    # один enabled flag
TradeFilterLifecycleConfig
TradeFilterDiagnosticsConfig
TradeFilterConfig
GridConfig.trade_filter: Optional[TradeFilterConfig] = None
```

Поля `TradeFilterZigZagConfig`:
- `global_stats_source: str = "full_dataset"`
- `leg_height_mode: str = "pct"`
- `reversal_threshold: Optional[float] = None`
- `candidate_trigger_threshold: float | str | None = None`
- `candidate_trigger_quantile: Optional[float] = None`   # MUST be None by default; YAML-template value 0.80 is example, not dataclass default
- `global_median: str = "auto"`
- `local_window: int = 5`

**Important:** dataclass defaults для `candidate_trigger_threshold` и `candidate_trigger_quantile` обязаны быть `None`, а не материализованными числовыми значениями. YAML template в §6.2 показывает `0.80` только как пример. Если dataclass default превратит absent quantile в `0.80`, raw-key presence tracking в §6.5 / §6.4 перестанет работать и правило reject (numeric threshold + explicit quantile) станет либо ложно срабатывать, либо ложно пропускать.

### 6.4 Loader / strict schema (`wf_grid.config.loader`)

Расширение `_ALLOWED_KEYS`:

```text
__top__ += trade_filter
trade_filter            = enabled, type, zigzag, triggers, lifecycle, diagnostics
trade_filter.zigzag     = global_stats_source, leg_height_mode, reversal_threshold,
                          candidate_trigger_threshold, candidate_trigger_quantile,
                          global_median, local_window
trade_filter.triggers   = candidate_threshold, confirmed_median
trade_filter.triggers.candidate_threshold = enabled
trade_filter.triggers.confirmed_median    = enabled
trade_filter.lifecycle  = freeze_confirmed_legs, stop_check, stopping_exit
trade_filter.diagnostics = export_state_columns, export_trigger_columns
```

Unknown keys reject строгим schema.

#### 6.4.1 Raw-key presence tracking (требование к loader)

Loader обязан различать «ключ явно указан пользователем в YAML» и «ключ отсутствует / используется dataclass default». Это требование возникает в §6.5 для пары `candidate_trigger_threshold` (numeric) + `candidate_trigger_quantile` (explicit).

Контракт:

1. До материализации dataclass loader строит `raw_user_keys: set[tuple[str, ...]]` или эквивалентный `meta` snapshot, фиксирующий dotted-path всех ключей, реально присутствовавших в YAML (например `("trade_filter", "zigzag", "candidate_trigger_quantile")`).
2. Validation `_validate_config(...)` выполняется **до** потери этого raw-presence context, либо raw-presence сохраняется в transport / `meta`, доступном валидатору.
3. dataclass default не должен превращать absent ключ в “explicit” значение. Default `candidate_trigger_quantile` обязан быть `None` (см. §6.3).
4. Правило reject из §6.5 “`candidate_trigger_threshold` numeric и пользователь явно указал `candidate_trigger_quantile`” срабатывает **только** если `("trade_filter","zigzag","candidate_trigger_quantile") in raw_user_keys`. Если ключ отсутствует в YAML и dataclass дал `None`, reject не срабатывает.
5. Acceptance gate (см. WP2 / §12 test matrix): unit-тест проверяет три кейса:
   - YAML с `candidate_trigger_threshold: 0.012` без `candidate_trigger_quantile` → accept;
   - YAML с `candidate_trigger_threshold: 0.012` и явно `candidate_trigger_quantile: 0.80` → reject;
   - YAML с `candidate_trigger_threshold: auto` и `candidate_trigger_quantile: 0.80` → accept.

`Spec reference: Appendix A v1.1 §11.2, §11.3`

### 6.5 Validation rules (полная таблица)

| Правило | Failure |
|---|---|
| `trade_filter` присутствует, `enabled` отсутствует | `trade_filter.enabled is required` |
| `enabled` не bool | `must be bool` |
| `enabled=true` и `type` отсутствует | `type is required when enabled` |
| `enabled=true` и `type != zigzag_st_mode` | `unsupported type` |
| `enabled=false` и `type` присутствует, `!= zigzag_st_mode` | `unsupported type for disabled filter` |
| `enabled=false` и `type` отсутствует | OK (§11.1) |
| `enabled=true` и `zigzag` отсутствует | `zigzag block required when enabled` |
| `enabled=true` и `triggers` отсутствует | `triggers block required when enabled` |
| `enabled=true` и `lifecycle` отсутствует | `lifecycle block required when enabled` |
| `enabled=true`, оба triggers disabled | `at least one trigger must be enabled` |
| `triggers.candidate_threshold.enabled` не bool | reject |
| `triggers.confirmed_median.enabled` не bool | reject |
| `zigzag.global_stats_source != full_dataset` | `only full_dataset supported in v1` |
| `zigzag.leg_height_mode != pct` | `only pct supported in v1` |
| `reversal_threshold` отсутствует / `None` / `NaN` | `required when enabled` |
| `reversal_threshold` percent string `"0.5%"` | `must be numeric fraction` |
| `reversal_threshold <= 0` или `>= 1` | reject |
| `local_window` не int или `< 1` | reject |
| `freeze_confirmed_legs` не int или `< 0` | reject |
| `lifecycle.stop_check != confirm_bar_only` | reject |
| `lifecycle.stopping_exit != opposite_st_flip` | reject |
| `global_median != auto` | `only auto full_dataset median supported` |
| `candidate_trigger_threshold` numeric NaN/Inf/`<=0`/`>=1`/percent string | reject |
| `candidate_trigger_threshold` строка `!= "auto"` | reject |
| `candidate_trigger_threshold = auto` без `candidate_trigger_quantile` | reject |
| `candidate_trigger_quantile` не в `(0, 1)` | reject |
| `candidate_trigger_threshold` numeric и пользователь явно указал `candidate_trigger_quantile` (см. §6.4.1: проверяется по raw-key presence, не по dataclass значению) | reject (§11.3) |

**Note 1:** `freeze_confirmed_legs < local_window` — валидная конфигурация, не warning, не reject (Appendix A v1.1 §3.2, §17.20). Это явный non-rule в таблице.

**Note 2 (raw presence):** правило numeric+quantile reject обязано быть реализовано через raw YAML presence tracking из §6.4.1, а не через сравнение `candidate_trigger_quantile is not None` поверх материализованного dataclass. Если default dataclass значение когда-либо станет non-None (например при rebase / refactor), правило обязано продолжать работать корректно, потому что оно опирается на raw-presence flag, а не на runtime значение поля.

`Spec reference: Appendix A v1.1 §11, §11.1, §11.2, §11.3, §15.6, §17.2`

### 8.1.1 Migration scope для перехода на `RawBacktestArtifacts`

Переход с 7-tuple `(returns, equity, trend, positions, early_exit, exit_bar, exit_dd)` на dataclass `RawBacktestArtifacts` требует явного scope, чтобы grep-gate был tractable и не превращался в ложный блокер из-за inactive подкаталогов.

#### Active scope (обязательная миграция)

Active scope — все callers в активном runtime path Phase 1 + связанные тесты:

- `donor/**` (включая `donor/supertrend_optimizer/**`);
- `wf_grid/**` (orchestrator, runner, step_executor, collectors, exports);
- `tests/**` для активного runtime (unit / integration / regression).

В active scope:

- ни один stale 7-value unpack `run_backtest_fast` не остаётся;
- каждый caller мигрируется явно на `RawBacktestArtifacts`;
- grep-gate (CI) проверяет отсутствие pattern'ов вида `returns, equity, trend, positions, early_exit, exit_bar, exit_dd = run_backtest_fast` и эквивалентных positional unpack'ов в active scope.

#### Inactive / parallel scope

Если в репозитории есть параллельные копии или legacy snapshots (например `donor TESTER/**`, `donor zigzag/**` (reference-only), архивные snapshots), план обязан принять explicit decision per copy:

- **Mode A — sync migrate**: подкаталог приводится к новому contract'у одновременно с active scope, попадает под grep-gate.
- **Mode B — quarantine**: подкаталог явно исключается из active grep-gate scope, помечается как inactive в плане / README, и для него добавляется отдельный smoke-test (см. ниже).

Запрещено: «silently skip» — то есть оставить дубликат с stale 7-tuple unpack без явной декларации mode и без smoke-test. Это создаёт latent breakage при будущих refactor / reuse.

#### Tester disabled-path smoke-test (post-migration)

После миграции на `RawBacktestArtifacts` обязателен smoke-test для donor TESTER:

1. Import smoke: `import` всех тестерных модулей не падает.
2. Один минимальный disabled backtest: `BacktestResult(...)` без `filter_diagnostics`, `extract_trades` сигнатура неизменна, return type совместим с тестерным caller'ом. Это покрывает breakage от silent contract changes.

Acceptance gate (WP7 / WP10):

- grep-gate чист в active scope;
- per inactive subtree принят explicit mode (A или B), задокументирован;
- tester disabled-path smoke зелёный.

### 8.2 `run_backtest_fast` return contract

`run_backtest_fast` сейчас возвращает 7-tuple `(returns, equity, trend, positions, early_exit, exit_bar, exit_dd)`. В Phase 1 переход на dataclass `RawBacktestArtifacts`:

```text
RawBacktestArtifacts(
    returns, equity_curve, trend, positions,
    early_exit, exit_bar, exit_drawdown,
    filter_diagnostics: dict[str, np.ndarray] | None = None,
)
```

Правила:
1. `run_backtest_fast` возвращает `RawBacktestArtifacts`.
2. `run_single_backtest` собирает `BacktestResult` из `RawBacktestArtifacts`.
3. `filter_diagnostics=None` на disabled path.
4. Disabled path: positions / returns / equity / trades / metrics / exports — bit-identical baseline.
5. Migration scope grep-gate (см. §8.1.1 ниже): ни один stale 7-value tuple unpack `run_backtest_fast` не остаётся в active scope; каждый caller мигрируется явно.
6. `BacktestResult` получает optional `filter_diagnostics: dict[str, np.ndarray] | None = None`. `__post_init__` добавляет инвариант `len(arr) == len(positions)` для каждого ключа, если `filter_diagnostics is not None`. **Этот инвариант проверяется уже после любой early_exit truncation (правило 7).**
7. **Early-exit synchronous truncation.** Если в каком-либо runtime path `early_exit=True` приводит к усечению `positions` / `returns` / `equity` / `trend` до `exit_bar`, тогда внутри `run_backtest_fast` **до возврата `RawBacktestArtifacts`** соответствующие массивы из `filter_diagnostics` усекаются до той же длины, по тем же индексам. Truncation выполняется синхронно для всего keyset.
   - Это операция внутри `run_backtest_fast`, **не** в `BacktestResult.__post_init__`. `__post_init__` остаётся чистой проверкой инварианта длин.
   - WF OOS path в Phase 1 hardcoded `early_exit_enabled=False` (см. §9.2), поэтому в canonical Phase 1 runtime truncation никогда не активируется. Правило существует как defensive contract для shared donor API: тестер, debug runs, future tooling могут включить early_exit, и broken length invariant в этом случае недопустим.
   - Если `filter_diagnostics is None` (disabled path), правило n/a.

Acceptance gates:
- baseline parity тест проходит на disabled path;
- tester import smoke-test проходит;
- grep-gate на 7-value unpack чист в active scope (§8.1.1);
- early_exit truncation: synthetic unit test с `early_exit_enabled=True`, `trade_filter.enabled=true` и принудительным early_exit подтверждает `len(filter_diagnostics[k]) == len(positions)` после возврата из `run_backtest_fast`, и `BacktestResult.__post_init__` не падает.

`Spec reference: Appendix A v1.1 §10, §13, §17.6`

#### 8.3.1 Close-only invariant в ZigZag pivot / height path

Plan §3.3 и Appendix A v1.1 §3.4 фиксируют, что ZigZag в Phase 1 — close-only. В сигнатуре `zigzag_st_filter.apply(...)` всё ещё принимаются `open_prices`, `high`, `low` ради signature symmetry с `run_backtest_fast`, future diagnostics и общих transport needs (например, `open_prices` нужен `calculate_returns` ниже по pipeline).

Жёсткое правило для Phase 1:

1. ZigZag pivot detection, confirmed-leg height formula, candidate-leg height formula, `local_median_N`, `global_median` используют **только** `close`.
2. `open_prices` / `high` / `low` могут быть переданы в `apply(...)` для signature symmetry, future diagnostics или общих transport needs, но **не имеют права** появляться в коде ZigZag pivot detection или height formula в Phase 1.
3. `hlc3` / `ohlc4` / любые derived OHLC streams в ZigZag pivot/height path запрещены в Phase 1.

Acceptance gates (WP4 / WP5 / WP9 / §12 test matrix):

- **Grep-gate**: внутри `donor/supertrend_optimizer/core/zigzag_st_filter.py` нет обращений к `high` / `low` (или к их synonyms) в функциях ZigZag pivot detection и leg-height formula. Допустимы только signature parameters и transport-only передача без участия в формуле.
- **Invariance unit gate**: запуск `apply(...)` с искусственно искажёнными `high` / `low` (например `high = close * 10`, `low = close * 0` при сохранении `close` и `open_prices`) даёт **bit-identical** `filtered_positions` и ZigZag-relevant поля `filter_diagnostics` (`candidate_height_pct`, `local_median_N`, `global_median`, `confirmed_legs_since_start`, `trade_filter_state`) относительно запуска с консистентными OHLC. Этот тест ловит косвенные утечки лучше, чем grep.

`Spec reference: Appendix A v1.1 §3.4, §17.1.3`

### 8.4 Trade-level diagnostics indexing

Donor использует `OPEN_TO_OPEN`. Signal decision принимается на close бара `t`, execution исполняется на open бара `t+1`. В `extract_trades` `entry_index` и `exit_index` — это **execution bars** (бары, на open которых произошёл вход / выход), а не decision bars.

Правило lookup для **OPEN_TO_OPEN** (Phase 1 canonical):

```text
# Entry side
entry_signal_idx       = max(entry_index - 1, 0)        # PINNED via WP6 unit test
entry_filter_state     = filter_diagnostics["trade_filter_state"][entry_signal_idx]
entry_trigger_source   = filter_diagnostics["trade_filter_trigger_source"][entry_signal_idx]

# Exit side — symmetric to entry under OPEN_TO_OPEN
exit_signal_idx        = max(exit_index - 1, 0)         # PINNED via WP6 unit test (§8.4.1)
fsm_state_at_exit_decision    = filter_diagnostics["trade_filter_state"][exit_signal_idx]
median_stop_at_exit_decision  = filter_diagnostics["median_stop_triggered"][exit_signal_idx]

# exit_reason ДОЛЖЕН определяться по bar-level diagnostics / FSM state на decision bar
# (exit_signal_idx), а не по execution bar (exit_index) без проверки.
exit_reason:
  - если fsm_state_at_exit_decision == ST_STOPPING и closed by opposite ST flip
        -> "filter_stopping_opposite_flip"
  - иначе нормальный ST exit / reversal
        -> "st_flip"
  - donor-style pending open trade (trade всё ещё открыт на последнем доступном баре slice'а)
        -> "pending_open_trade_at_end"
        # Edge case: для этого reason exit_signal lookup НЕ выполняется,
        # exit_reason проставляется напрямую без обращения к diagnostics.
```

#### 8.4.1 Acceptance gate

Unit test (WP6 / WP7) обязан **в одном тесте** пинить **обе** indexing rules под `OPEN_TO_OPEN`:

1. На синтетическом ряду с известными ST flips демонстрирует, что donor `extract_trades` ставит `entry_index` = open bar входа, `exit_index` = open bar выхода.
2. Подтверждает, что `entry_signal_idx = entry_index - 1` соответствует close decision bar для entry.
3. Подтверждает, что `exit_signal_idx = exit_index - 1` соответствует close decision bar для exit.
4. Конструирует сценарий, где FSM находится в `ST_STOPPING` ровно на `exit_signal_idx` и уже в `OFF` на `exit_index` (то есть decision bar и execution bar дают разный state). Тест требует, чтобы `exit_reason == "filter_stopping_opposite_flip"`. Если бы exit_reason читался по execution bar, он был бы неверным — этот тест ловит такой regression.
5. Покрывает edge cases:
   - `entry_index == 0` → `entry_signal_idx = max(0 - 1, 0) = 0` (теоретически unreachable под OPEN_TO_OPEN, но правило `max(..., 0)` защищает invariant);
   - `pending_open_trade_at_end` → exit signal lookup не выполняется, exit_reason проставляется напрямую.

**Если donor `extract_trades` indexing test покажет другое соответствие** (например `entry_index` оказался decision bar, а не execution bar), план **должен быть обновлён до** старта реализации `attach_trade_filter_diagnostics`. Без зелёного pinning теста §8.4 trade-level diagnostics в `BacktestResult` не подключаются.

Lookup делается на extended-slice индексах **до** OOS trim / rebase. После attach диагностические колонки следуют за trade rows через rebase и фильтрацию (см. §4.4).

`Spec reference: Appendix A v1.1 §10, §13, §15.4, §17.13`

**Clarification к пункту (a), чтобы не конфликтовать с §10.5.2:** «присутствует в стандартном экспорте» означает, что **набор** диагностических полей и их согласованность с Appendix A v1.1 §13 соблюдены в **стандартном** trade- / step-level export pipeline (включая summary-колонки, trade-level attach, при необходимости optional листы вроде `WF_FilterDiagnostics_Top1` по флагам) — **без** internal-only полей (b). Это **не** требует **полной** per-bar materialization **всех** §13 массивов в **каждом** XLSX-листе для **всех** grid points. Полный bar-level keyset **всегда** в runtime — §10.5.1; что именно **сохраняется** в pickle/xlsx и на каких grid points — §10.5.2, §10.5.3 (по умолчанию full bar в XLSX — rank-1 / debug, не весь grid).

### 10.5 Diagnostics availability vs storage / memory policy

В §10.5 явно разделены **два независимых слоя**, чтобы избежать silent decision о retention:

#### 10.5.1 Runtime result invariant (всегда)

Для любого enabled backtest (`trade_filter.enabled=true`) runtime result содержит **полный Appendix A v1.1 §13 bar-level keyset** в `BacktestResult.filter_diagnostics` к моменту возврата из `run_single_backtest`. Это invariant, не option.

- Полный keyset перечислен в §10.1 и в Appendix A v1.1 §13.
- Длины массивов согласованы с `positions` (см. §8.2 правило 6 + early-exit truncation §8.2 правило 7).
- Этот invariant покрывает rank-1, rank-N, debug, on-demand inspections — всё, что читает live `BacktestResult` сразу после backtest.
- Disabled path: `filter_diagnostics is None`. Это не нарушение invariant'а — keyset relevant только when filter enabled.

Acceptance gate: §12 матрица содержит test, проверяющий, что для enabled backtest все §13 ключи присутствуют в `BacktestResult.filter_diagnostics` и имеют корректную длину.

#### 10.5.2 Storage policy (что попадает в long-term artifacts)

Default production grid run:
- step-level summary diagnostics (см. §10.6 `_OOS_COLUMNS` summary block) сохраняются для всех grid points;
- trade-level diagnostics приклеены к `trades_df` rows если фильтр enabled;
- full bar-level diagnostics **не** сохраняются для всех grid points в pickle / xlsx, чтобы не раздувать память (footprint `n_grid × n_steps × n_bars × n_keys`).

Rank-1 / explicit debug / on-demand path:
- материализует full Appendix A v1.1 §13 bar-level keyset из ещё-живого runtime result, **не** пытается “восстановить” его из summary;
- именно эта материализация попадает в `WF_FilterDiagnostics_Top1` лист и в debug-only artifacts.

`StepResult.filter_diagnostics_oos` может быть очищен / суммирован после извлечения нужных summary и trade-level diagnostics для не-rank-1 grid points, если debug-флаг не выставлен. Это разрешено именно потому, что runtime invariant из §10.5.1 уже сработал — все decisions, нужные для trade-level attach и summary, уже извлечены до cleanup.

#### 10.5.3 Full-grid bar diagnostics retention

Включается **только** explicit debug flag / debug mode. Это отдельное memory-policy решение, не silent implementation detail. Если owner потребует full-grid bar diagnostics always-on (без debug флага), это требует явного approval и обновления §10.5.2, а не молчаливого включения. До этого момента default остаётся rank-1-only.

XLSX:
- `WF_FilterDiagnostics_Top1` — опциональный лист, gated `diagnostics.export_state_columns: true`;
- full-grid bar diagnostics в стандартном экспорте отсутствуют.

### 10.6 Collectors and XLSX writer

#### 10.6.1 Conditional trade diagnostic columns

Trade-level filter diagnostic columns (`entry_filter_state`, `entry_trigger_source`, `exit_reason`) добавляются в trade-level outputs **только** при выполнении обоих условий:

1. `trade_filter.enabled=true` в активной конфигурации;
2. `BacktestResult.filter_diagnostics is not None` и trade-level attach (§8.4) фактически выполнен.

Если хотя бы одно условие не выполнено, эти колонки **не появляются** в df / collector / экспорте. Disabled baseline artifacts / exports **не должны** получать пустые / NaN / placeholder filter diagnostic columns. Это hard parity rule с baseline.

#### 10.6.2 Collector / export schema preservation

`wf_grid.collect.trades_collector` и downstream writers обязаны:

- сохранять любые extra diagnostic columns, присутствующие в input df, **without silently dropping them**;
- не «нормализовать» столбцы df к фиксированному баseline-only списку;
- не превращать absent колонки (disabled path) в empty/NaN columns (enabled-look-alike).

#### 10.6.3 Deterministic column order

Финальный порядок колонок в trade-level export:

```text
[ known donor trade columns in canonical order ]
+ [ optional filter diagnostic columns in fixed order, ONLY if present in df ]
```

Канонический порядок optional блока (если присутствует):

```text
entry_filter_state
entry_trigger_source
exit_reason
```

`wf_grid.export.xlsx_writer._TRADES_COLUMN_ORDER` расширяется условно по наличию колонок в DataFrame; **deterministic snapshot тест** фиксирует exact column sequence для двух базовых сценариев:

- disabled run (no filter columns at all);
- enabled run (filter columns present, в фиксированном порядке после donor columns).

#### 10.6.4 Step-level summary (всегда conditional)

`wf_grid.collect.step_collector._OOS_COLUMNS` расширяется summary-блоком только когда фильтр был enabled на данной grid point / step:

```text
filter_states_visited
n_bars_in_off
n_bars_in_freeze
n_bars_in_monitoring
n_bars_in_stopping
n_filter_blocked_entries
lifecycle_starts_count
median_stop_triggered_count
```

Bar-level массивы в `step_oos_long` не льются (см. §10.5.2).

#### 10.6.5 Optional rank-1 sheet

Опциональный лист `WF_FilterDiagnostics_Top1` с bar-level массивами для rank-1 grid_point добавляется по флагу (см. §10.5.3).

#### 10.6.6 Golden tests (acceptance)

В §12 test matrix добавляются два golden test'а:

1. **Disabled baseline export parity**: pipeline на disabled config производит trade-level export, который bit-identical baseline. Проверяется отсутствие любых filter diagnostic columns (`entry_filter_state`, `entry_trigger_source`, `exit_reason`) и фиксированный snapshot column order.
2. **Enabled diagnostic column preservation**: pipeline на enabled config производит trade-level export, в котором filter diagnostic columns присутствуют в фиксированном порядке после donor columns, ни одна колонка не дропнута collector / writer цепочкой.

### 10.7 Module location и state model

`donor/supertrend_optimizer/core/zigzag_st_filter.py` — модуль реализующий **deterministic single-pass FSM with no module-level / global hidden state between calls**. В Phase 2 reused тестером без перемещения, потому что contract стабилен.

#### 10.7.1 Точный state model

Внутри одного вызова `apply(...)` FSM **stateful по определению**: `WAIT_FIRST_ST_FLIP` ждёт flip, `ST_ACTIVE_FREEZE` считает confirmed legs, `ST_ACTIVE_MONITORING` обновляет rolling window, и т.д. Это ядро §5 / §5.4 / §5.5: per-bar boolean mask недостаточен.

Между вызовами `apply(...)` модуль **не сохраняет** никакого state:

- никаких module-level globals / mutable defaults;
- никаких class-level mutable defaults (никаких `class X: cache = []` или эквивалентов);
- никаких lru_cache / ad-hoc dict cache на module / class scope, которые могут пережить вызов;
- `ZigZagGlobalStats` приходит как input аргумент, а не вычисляется лениво внутри модуля и не кэшируется в нём;
- все per-call structures (FSM state, rolling window, diagnostics buffers) создаются на входе в `apply(...)` и disposed на выходе.

Корректная формулировка для документации и review: **«deterministic single-pass FSM with no module / global state between calls; pure per-call state, no module-level state retained»**.

Слово **stateless** к этому модулю **не применяется** и не должно использоваться в плане, тестах, или PR descriptions, потому что вводит в заблуждение и может прочитаться как лицензия на per-bar boolean mask реализацию (что прямо нарушает §2.2 invariant и §5.4).

#### 10.7.2 Acceptance gate

Unit test (WP9 / WP10): два последовательных вызова `apply(...)` с **идентичными** входными аргументами производят **bit-identical** `ZigZagSTFilterResult` (positions, filter_diagnostics, internal_legs). Если бы модуль имел скрытое module-level state, второй вызов мог бы дать другой результат — этот тест ловит такое regression.

`Spec reference: Appendix A v1.1 §13, §16, §17.2`

### WP6. ST flip detection and event ordering

**Goal.** ST flip detection и `OPEN_TO_OPEN` semantics.

## 14. Definition of Done

План считается выполненным, когда:

1. Appendix A v1.1 приложен и каждая WP содержит `Spec reference`.
2. Disabled path (`trade_filter.enabled=false` или отсутствие блока) bit-identical baseline по positions, returns, equity, trades, metrics, exports, `bucket_matrix_*`, `PipelineDiagnostics`.
3. Все 24 functional + 12 technical критериев Appendix A v1.1 §17 проходят.
4. Все тесты из матрицы §12 (актуальная нумерация v2.1.1: строки #1…#57) зелёные; при добавлении строк в матрицу DoD ссылается на **текущую** таблицу §12, а не на устаревшее число.
5. ZigZag thresholds и height metrics — fractions; нет percent-string путей.
6. Close-only ZigZag formula совпадает в WP3, WP4, WP5 (одна shared fixture).
7. `np.quantile method="linear"` pinned для auto threshold.
8. Init failure (`§12.3`) фиксируется до WF execution; нет частично-материализованных stats.
9. ST_ACTIVE_MONITORING на confirm-bar с unavailable / NaN / Inf median → ST_STOPPING (true fail-closed).
10. WAIT_FIRST_ST_FLIP × `trade_mode`: disallowed flip пропускается, lifecycle не стартует.
11. `0 -> ±1` не tradable flip ни в WAIT_FIRST_ST_FLIP, ни в ST_STOPPING.
12. Same-bar trigger + allowed ST flip → entry на `t+1`; same-bar trigger + disallowed flip → state остаётся.
13. `freeze_confirmed_legs < local_window` валиден без warning/reject.
14. `local_median_N` использует causal slice-local историю, не ограничен legs after lifecycle start.
15. `RawBacktestArtifacts` используется как transport; нет stale 7-value tuple unpack `run_backtest_fast`.
16. `BacktestResult` имеет `filter_diagnostics: Optional[...] = None`; tester и существующие callers совместимы.
17. `extract_trades` сигнатура неизменна; trade-level diagnostics приклеиваются post-`extract_trades` на extended-slice индексах через unit-test-pinned indexing rules: `max(entry_index - 1, 0)` и `max(exit_index - 1, 0)` (§8.4), включая edge cases из §8.4.1.
18. Force-flat не модифицирует `filter_diagnostics`; bar-level economics и diagnostics инвариантны по длине.
19. Стандартный export pipeline соблюдает Appendix A v1.1 §13 keyset (без internal-only); трактовка «стандартный экспорт» vs full per-bar XLSX для всего grid — §10.4 clarification + §10.5.2. Internal-only поля (`candidate_trigger_source`, `global_offset`, ...) — только в debug-only листе.
20. Default memory footprint: full bar diagnostics — только для rank-1 / debug grid points; debug флаг включает full-grid retention.
21. Tester import smoke-test passes; `donor TESTER` regression на disabled path стабилен.
22. **Raw YAML presence tracking** (§6.4.1) реализован в loader; правило numeric+quantile reject не зависит от dataclass default; default `candidate_trigger_quantile = None`.
23. **Close-only invariance** (§8.3.1): grep-gate чист, invariance unit gate зелёный (искажённые `high`/`low` не меняют ZigZag-relevant outputs).
24. **Trade-level indexing pinned** для entry **и** exit под `OPEN_TO_OPEN` одним unit-тестом (§8.4.1); `pending_open_trade_at_end` edge case обработан.
25. **Early-exit synchronous truncation** (§8.2 правило 7) реализован внутри `run_backtest_fast`; `__post_init__` length-инвариант стабилен под `early_exit_enabled=True`.
