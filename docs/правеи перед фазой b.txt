# 3. File-by-File Audit (A1–A13)

### A1 — Config / schema / loader  
**Файлы:** `config/schema.py`, `config/loader.py`  
- **Корректно:** валидация полей, `resolve_periods_per_year` через донор, `effective_step_min_trades`, deepcopy при resolve.  
- **Подозрительно:** `annualization_basis`/рынок по умолчанию легко **не совпасть** с реальностью инструмента; нет предупреждения в отчёте при `periods_per_year` explicit vs auto mismatch.  
- **Недотестировано:** экстремальные YAML, смешанные типы в вложенных dict.  
- **Хрупкость:** смена донорного resolver’а ломает годовую нормализацию без явного теста паритета.  
**Phase verdict:** **Needs hardening**

### A2 — Grid enumeration  
**Файлы:** `grid/enumeration.py`  
- **Корректно:** тики для multiplier, детерминированный порядок.  
- **Подозрительно:** `grid_point_id` = `f"...{canonical_mult:.2f}..."` — теоретический **коллизионный** риск при очень мелком шаге и округлении (низкая вероятность, не ноль).  
- **Недотестировано:** границы `tick_min`/`tick_max` vs `mult_max` (off-by-one в сетке).  
**Phase verdict:** **OK** с оговоркой

### A3 — Status model  
**Файлы:** `status/status_model.py`  
- **Корректно:** приоритет `insufficient_bars` → `no_trades` → `ok`; гейты отдельно.  
- **Подозрительно:** `assign_candidate_status` в коде агрегатора **не используется** для гейтов/ранга — статус кандидата в tiering **пересобран** из `n_ok`/`n_total` (согласовано, но дублирование семантики).  
- **Недотестировано:** float `num_trades` ≈ 0.  
**Phase verdict:** **OK**

### A4 — WF runner + step executor  
**Файлы:** `wf/runner.py`, `wf/step_executor.py`  
- **Корректно:** prepend, `oos_boundary`, trim, trade filter `entry_index >= boundary`, override трейд-метрик, defensive fallback с инвалидацией метрик.  
- **Подозрительно:** **`early_exit`** (см. Critical); `net_pnl_pct := avg_trade` в override — дублирование смысла, риск путаницы с compound PnL; train в `execute_train_step` кладёт сделки в `oos_trades_df` (имя поля вводит в заблуждение при межмодульном чтении).  
- **Недотестировано:** граничные случаи `test_start=0`, prepend=0, OOS из одного бара, согласованность `early_exit` с длиной OOS.  
**Phase verdict:** **Risky**

### A5 — Step collectors  
**Файлы:** `collect/step_collector.py`  
- **Корректно:** инварианты полноты/уникальности; runtime_error через `error_message`.  
- **Подозрительно:** нет статуса `invalid` в ветке collector (в enum есть) — «мёртвый» путь.  
**Phase verdict:** **Needs hardening**

### A6 — Trades collectors  
**Файлы:** `collect/trades_collector.py`  
- **Корректно:** инвариант E `entry_index >= 0`.  
- **Подозрительно:** экспорт только rank-1 в оркестраторе — **полная сетка сделок не в артефакте**; человек не видит «хвост» сетки без доработок.  
**Phase verdict:** **OK** для контракта, **Risky** для исследовательской честности

### A7 — Aggregation  
**Файлы:** `aggregate/aggregator.py`  
- **Корректно:** ok-only, sentinel→NaN, cap на MAX для выбранных метрик, `sum_pnl_pct_Sum`, `total_oos_trades`.  
- **Подозрительно:** агрегация **только по ok** = **survivorship на уровне шагов**; предупреждения есть частично (`survivorship_warning`), но **не заменяют** статистику множественных сравнений.  
- **Недотестировано:** смесь ok с разными длинами OOS при early exit.  
**Phase verdict:** **Needs hardening**

### A8 — Gates  
**Файлы:** `gates/gates.py`  
- **Корректно:** шаговые гейты до агрегата; кандидатные — после; явные колонки `total_oos_trades`, `ok_ratio`.  
- **Подозрительно:** при малых `n_ok_steps` гейты на **медиане/мин** статистически **шумовые**; `worst_segment` off → гейт «всегда True» для непустых ok.  
**Phase verdict:** **Needs hardening**

### A9 — Scoring  
**Файлы:** `ranking/scoring.py`  
- **Подозрительно:** см. Critical (импутация медианой, молчаливое перераспределение весов, N=1/нулевой разброс → «тихий» score).  
**Phase verdict:** **Risky**

### A10 — Ranking  
**Файлы:** `ranking/ranker.py`, `ranking/tiering.py`  
- **Корректно:** детерминизм `grid_point_id`, NA last, legacy с приоритетом `seed_gate_passed`.  
- **Подозрительно:** Tier 2 сортируется по сырому PnL — **сильные кандидаты с провалом гейтов** могут выглядеть «высоко» в таблице; Tier 1 требует все шаги ok + гейты + `min_segments` — слово **«proven»** в комментарии **перегружено** для финансового смысла.  
**Phase verdict:** **Risky**

### A11 — Summary builder  
**Файлы:** `export/summary_builder.py`

- **Подозрительно:** `n_passed_for_scoring` **одинаков для всех строк** — легко **перепутать** с per-row метрикой; fallback парсинга `grid_point_id` если нет identity-колонок.

### A12 — XLSX export  
**Файлы:** `export/xlsx_writer.py`

### A13 — Orchestrator + diagnostics + E2E  
**Файлы:** `pipeline/orchestrator.py`, `logging_/diagnostics.py`, `tests/test_e2e_real_data.py`  
- **Корректно:** deepcopy конфига, полный проход, диагностика top-5.

# 4. Cross-Module Contract Drift

- **Оркестратор (комментарий) ↔ код:** порядок gates vs collection (см. выше).  
- **`StepResult.oos_trades_df` в train:** одно поле для разных смыслов (OOS vs train).  
- **Два источника констант INVALID/MAX:** `wf_grid.config.schema` и `supertrend_optimizer.utils.constants` — должны совпадать; дрейф версий = яд для агрегатора.  
- **Скоринг:** частичное автодобавление `abs_max_drawdown_*` только для части весов; остальное — тихое удаление весов.  
- **Phase B:** `quantile_gates_status` = stub `None` — следующая фаза **сломает** ожидания потребителей summary, если они начнут полагаться на колонку.  
- **Строковый `grid_point_id`** vs структурные поля: fallback-парсинг при отсутствии колонок — риск **тихой** деградации identity в экспорте.

# 5. Math / Stats / Research Integrity Risks

- **Нет коррекции на множественные сравнения** — главный источник ложной устойчивости top-1.  
- **Survivorship:** ok-only агрегация + `ok_ratio` отражают отсев, но **не делают** выводы о значимости alpha.  
- **Малые выборки:** медиана/Std по 2–3 ok-шагам — **неустойчивые** порядки ранга.  
- **Min-max / rank на 2–5 passed:** «точность» скора **иллюзорна**; `score_discrimination_status` смягчает, но **не блокирует** ранжирование.  
- **Капы MAX_VALID_METRIC:** обнуление в агрегаторе убирает экстремумы — корректно для робастности, но **скрывает** «идеальные» PF (все прибыли) из средних.  
- **Отключённые гейты** (`worst_segment_pnl_threshold: null`): худший сегмент **не ограничен** — типичный путь к **хвостовому риску** при красивой медиане.  
- **Train vs OOS:** train в отчёте **не** должен использоваться как подтверждение OOS — смешение в голове пользователя = исследовательская ошибка.

---

# 6. Human-Risk / Real-Money Risks

- Человек видит **`grid_rank == 1`**, Tier 1, `reliability_flag == HIGH`, аккуратный XLSX → **переоценка** причинности.  
- **`tester_seed_score`** выглядит как объективный «балл», хотя это **внутренняя нормализация когорты**, не ожидаемая доходность.  
- **`ok_ratio` 0.9** даёт **пустой** survivorship warning — при длинном WF всё ещё **10%** сегментов отброшено/не ok.  
- **Убытки:** выбор top-1 после сетки без форвард-теста; игнорирование комиссий/проскальзывания вне донора; смена режима рынка; хвостовые сценарии при отключённом worst-segment gate.  
- **Нет guardrails:** внешний hold-out, лимиты на размер сетки в UI, обязательный блок «multiple testing» в summary, запрет торговли по одному прогону без форварда.

# 7. Test Coverage Gaps (конкретика)

- Дифф-тест: **`run_single_backtest` на полном срезе vs `execute_oos_step`** на том же окне (значения метрик).  
- Путь **`early_exit=True`** с усечением внутри OOS: ожидаемый статус/флаг, а не «ok» с полным горизонтом.  
- **Один прошедший гейт кандидат** — стабильность `tester_seed_score` и порядка ранга.  
- **Sparse / неравномерный** календарь intraday + `periods_per_year` auto.  
- **Золотые снимки чисел** для одного фиксированного мини-грида (не только форма колонок).  
- **Fuzz** конфигов и статусов шагов (property: aggregator не падает, инварианты).  
- Схема **версионирования** YAML (неизвестные ключи — fail или warn).  
- Тест: отсутствующая колонка в весах скоринга → **явная ошибка**, не тихое перераспределение.

# 8. Mandatory Preconditions Before Real Use

**A) Research use**  
- Зафиксировать конфиг, seed, версии кода и данных; явный дисклеймер multiple testing.  
- Проверить `early_exit` (для WF — предпочтительно **off** на OOS).  
- Просмотреть `score_discrimination_status`, `survivorship_warning`, `has_defensive_fallback_steps`.

**B) Paper trading**  
- Всё из (A) + forward/OOS **вне** сетки; сравнение top-3 vs случайные baseline из сетки.  
- Мониторинг фактической длины OOS vs заявленной (диагностика early exit / defensive fallback).

**C) Real money**  
- Независимая валидация на **чистых** данных; лимиты капитала; проскальзывание/комиссии реальные; **никаких** решений только по Phase A XLSX.

# 9. Final Go / No-Go

**Итог: NO-GO** для real-money и для «решений на основе одного прогона Phase A».

**GO ONLY FOR RESEARCH** — **условно**, если одновременно выполнено:

1. OOS-путь без искажения горизонта (**`early_exit` отключён** или жёстко валидирован неполный OOS).  
2. Пользователь **не** использует Tier 1 / rank 1 как доказательство — только как гипотезу для следующей фазы.  
3. Устранены или явно зафиксированы контракты скоринга (отсутствующие колонки весов → ошибка, не молчание).  
4. В отчёт вынесены **предупреждения** про множественные сравнения и неполноту когорты для скора.

Для **paper trading** — только как **следующий этап** после отдельного форвард-дизайна, не как прямое продолжение «зелёных» тестов Phase A.

---

**Главный ответ на ваш контрольный вопрос:** система может привести к риску реальных денег, потому что **выдаёт ранжированный, оформленный как аналитика результат сеточного поиска без статистической честности top-1**, с **скрытыми степенями свободы** в скоринге и с **потенциально неверным OOS-горизонтом** при включённом early exit — всё это легко читается человеком как «готовый к торговле» вывод.


Если брать приоритет фиксов перед любым paper/live use, я бы поставил так:
(1) запретить или явно инвалидировать OOS-steps с early_exit,
(2) сделать scoring fail-fast для любой отсутствующей взвешенной колонки,
(3) убрать медианную импутацию из score path,
(4) вынести hard warning про multiple testing в summary/XLSX на уровень “не торговать по одному прогону”.
