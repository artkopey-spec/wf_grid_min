ТЗ v3 final: Volatility Breakout Upgrade для ZigZag-ST Filter
==============================================================

Статус документа
----------------
Документ является самостоятельным финальным ТЗ для реализации v3.
Реализация ведётся по этому документу. Чтение v2 не требуется.

v2 (`docs/ТЗ volatility breakout upgrade для ZigZag-ST v2.txt`) остаётся
в репозитории только как исторический артефакт. При любом расхождении v2 и
v3 приоритет имеет v3. Будущие изменения требований оформляются отдельным
ТЗ, а не правками этого документа.

Каноническая модель v3:

```yaml
trade_filter:
  enabled: true
  type: zigzag_st_mode
  zigzag:
    mode: A | B | C | A+B | C+B
```

Других режимов ZigZag-ST filter в v3 нет.

Legacy `trade_filter.triggers.*` разрешён только как backward-compat input
для миграции старых конфигов, когда `trade_filter.zigzag.mode` отсутствует.
В каноническом v3 config блок `trade_filter.triggers` не требуется.

Legacy v2-форма `trade_filter.zigzag.candidate_entry` в любом виде не
поддерживается и отвергается валидацией.


1. Цель
-------
Расширить существующий ZigZag-ST filter без изменения OPEN_TO_OPEN execution
model и без изменения post-entry lifecycle.

Режимы:

- `A` - существующий candidate-threshold WAIT-режим:
  candidate-нога достигает порога высоты, FSM переходит в
  `WAIT_FIRST_ST_FLIP`, первая сделка открывается по первому допустимому
  SuperTrend flip.
- `B` - существующий confirmed-median WAIT-режим:
  confirmed-median condition становится истинным, FSM переходит в
  `WAIT_FIRST_ST_FLIP`, первая сделка открывается по первому допустимому
  SuperTrend flip.
- `C` - новый volatility breakout / immediate entry:
  candidate-нога достигает порога высоты, FSM открывает позицию сразу по
  направлению candidate-ноги, без ожидания SuperTrend flip.
- `A+B` - одновременно активны A-ветка и B-ветка.
- `C+B` - одновременно активны C-ветка и B-ветка.

Дополнительно добавить optional candidate duration gate:

- включается из config;
- `max_bars` задаётся из config;
- применяется только к candidate-веткам режимов `A`, `C`, `A+B`, `C+B`;
- не применяется к B-ветке;
- при чистом `mode: B` gate валиден, но runtime ведёт себя bit-identical
  режиму B без gate; в логах появляется ровно одна INFO-строка на запуск.

После входа в позицию lifecycle такой же, как в текущем A/B:

```text
ST_ACTIVE_FREEZE -> ST_ACTIVE_MONITORING -> ST_STOPPING -> OFF
```


2. Термины
----------
Candidate-нога
    Текущая незавершённая ZigZag-нога, вычисленная causal close-only pass.

Confirmed-нога
    Уже подтверждённая ZigZag-нога.

Decision bar
    Бар `t`, на close которого FSM принимает решение. Позиция пишется в
    `filtered_positions[t + 1]`, если `t + 1` находится в границах массива.

Candidate height (`candidate_height_pct[t]`)
    Относительная высота текущей candidate-ноги. Значение хранится как
    fraction of price, не 0..100 percent. UNKNOWN = `NaN`.

Candidate direction (`candidate_leg_direction[t]`)
    Направление текущей candidate-ноги:

```text
+1 = candidate-нога вверх
-1 = candidate-нога вниз
 0 = UNKNOWN
```

`candidate_leg_direction[t]` отражает close-only ZigZag direction и не
является синонимом `held_pos`.

Candidate age (`candidate_age_bars[t]`)
    Возраст текущей candidate-ноги:

```text
candidate_age_bars[t] = t - last_pivot_bar + 1
```

UNKNOWN = `-1`. Значение `0` не используется как UNKNOWN.

Confirmed median condition
    Существующее per-bar условие B-ветки. В реализации оно может называться
    `confirmed_median`, `global_median`, `B-trigger` и т.п.; v3 фиксирует
    поведение, а не переименовывает helper.

Daily reset event (`daily_reset_event[t]`)
    Per-bar boolean: `true` на баре, где выполняется reset-обработка новой
    сессии / нового календарного дня.

Trade mode
    Существующее значение:

```text
long | short | both | revers
```

Whitelist `trade_mode` в v3 не расширяется.


3. Config
---------
Каноническая форма v3:

```yaml
trade_filter:
  enabled: true
  type: zigzag_st_mode

  zigzag:
    mode: C+B                       # A | B | C | A+B | C+B
    global_stats_source: full_dataset
    leg_height_mode: pct
    reversal_threshold: 0.001
    candidate_trigger_threshold: auto
    candidate_trigger_quantile: 0.80
    global_median: auto
    local_window: 5
    daily_reset: true

    candidate_duration_gate:
      enabled: true
      max_bars: 10                  # int >= 1, required when enabled=true

  lifecycle:
    freeze_confirmed_legs: 5
    stop_check: confirm_bar_only
    stopping_exit: opposite_st_flip

  diagnostics:
    export_state_columns: true
    export_trigger_columns: true
```

Defaults:

```text
trade_filter.zigzag.mode = A
trade_filter.zigzag.candidate_duration_gate.enabled = false
trade_filter.zigzag.candidate_duration_gate.max_bars = absent
```

`trade_filter.type` остаётся `zigzag_st_mode`. Новый тип вроде
`zigzag_st_mode_v3` не вводится.

3.1. Backward compatibility: legacy triggers -> mode

Legacy `trade_filter.triggers.*` используется только если
`trade_filter.zigzag.mode` отсутствует.

Mapping:

```text
нет mode и нет legacy triggers                 -> A
candidate_threshold=true, confirmed_median=false -> A
candidate_threshold=false, confirmed_median=true -> B
candidate_threshold=true, confirmed_median=true  -> A+B
candidate_threshold=false, confirmed_median=false -> ConfigError
```

Если `trade_filter.zigzag.mode` явно задан и одновременно присутствует
`trade_filter.triggers`, config является mixed canonical/legacy schema и
отвергается `ConfigError`.

Legacy `trade_filter.zigzag.candidate_entry` в любом виде -> `ConfigError`.


4. Config Validation
--------------------
Все ошибки trade_filter validation агрегируются в один `ConfigError`, как
в существующем `validate_trade_filter`.

4.1. `trade_filter.zigzag.mode`

Допустимые литералы, строго case-sensitive:

```text
A
B
C
A+B
C+B
```

Любое другое значение:

```text
trade_filter.zigzag.mode must be one of {A, B, C, A+B, C+B}; got <value>
```

4.2. `trade_filter.zigzag.candidate_duration_gate`

`enabled`:

- должен быть `bool`;
- строки `"true"`, `"false"`, числа и `null` отвергаются.

`max_bars`:

- обязателен при `enabled: true`;
- должен быть `int >= 1`;
- `bool`, `float`, `string`, `null` отвергаются;
- при `enabled: false` ключ `max_bars` должен полностью отсутствовать в
  YAML. Явное `max_bars: null` тоже ошибка.

Ошибки:

```text
candidate_duration_gate.max_bars is required when enabled is true
candidate_duration_gate.max_bars must be absent when enabled is false; remove the key
candidate_duration_gate.max_bars must be int >= 1; got <value>
```

4.3. Mode B + enabled gate

`mode: B` + `candidate_duration_gate.enabled: true` + valid `max_bars`
является валидным config. Gate сохраняется в metadata/config snapshot, но
не участвует в B runtime decisions.

4.4. Strict schema whitelist

В shared whitelist и loader whitelist добавить:

```text
trade_filter.zigzag.mode
trade_filter.zigzag.candidate_duration_gate
trade_filter.zigzag.candidate_duration_gate.enabled
trade_filter.zigzag.candidate_duration_gate.max_bars
```

Legacy `trade_filter.triggers` остаётся в whitelist только для migration.
При explicit `zigzag.mode` наличие `trade_filter.triggers` отвергается
валидатором как mixed schema.

4.5. Init-failure keyset

Расширить existing init-failure pipeline ключами:

```text
mode_invalid_literal
mode_conflicts_with_legacy_triggers
candidate_entry_deprecated
duration_gate_enabled_invalid_type
duration_gate_max_bars_missing
duration_gate_max_bars_present_when_disabled
duration_gate_max_bars_invalid_type
duration_gate_max_bars_below_one
```


5. Global Stats / Materialization
---------------------------------
Materialization выполняется один раз на полном валидированном датасете до
WF execution.

`ZigZagGlobalStats` расширяется полями:

```text
zigzag_mode: Literal["A", "B", "C", "A+B", "C+B"]
candidate_duration_gate_enabled: bool
candidate_duration_max_bars: Optional[int]
```

Disabled / absent gate:

```text
candidate_duration_gate_enabled = False
candidate_duration_max_bars = None
```

Enabled gate:

```text
candidate_duration_gate_enabled = True
candidate_duration_max_bars = int(max_bars)
```

Representation:

| Layer | Disabled value | Enabled value |
| --- | --- | --- |
| Dataclass | `None` | `int(max_bars)` |
| metadata.config_snapshot | `None` | `int(max_bars)` |
| numpy diagnostics | `-1` | `int(max_bars)` |
| XLSX | `-1` | `int(max_bars)` |

`metadata.config_snapshot` обязан содержать:

```python
{
    "zigzag_mode": "A" | "B" | "C" | "A+B" | "C+B",
    "candidate_duration_gate": {
        "enabled": bool,
        "max_bars": int | None,
    },
    "reversal_threshold": float,
    "local_window": int,
    "global_stats_source": "full_dataset",
    "leg_height_mode": "pct",
    "candidate_trigger_threshold": raw_value,
    "candidate_trigger_quantile": raw_value_or_none,
}
```


6. Per-Bar ZigZag Fields
------------------------
`ZigZagPerBar` и `filter_diagnostics` должны гарантировать массивы:

```text
candidate_height_pct     float64
candidate_age_bars       int64
candidate_leg_direction  int8
```

Все массивы имеют длину `len(close)`.

6.1. UNKNOWN representation

На pre-bootstrap, reset-баре, pathological close и UNKNOWN direction:

```text
candidate_height_pct[t] = NaN
candidate_age_bars[t] = -1
candidate_leg_direction[t] = 0
```

6.2. Confirm-bar semantics

На confirm-баре новая candidate-нога уже началась после подтверждённой ноги.
Если подтверждённая нога имела direction `D`, то:

```text
candidate_leg_direction[t] = -D
candidate_age_bars[t] >= 2
```

6.3. Reset semantics

На reset-баре in-progress candidate state стирается до вычисления
candidate/runtime primitives для этого бара. Первое реальное candidate
trigger после reset возможно только после нового bootstrap.


7. Runtime Primitives
---------------------
На каждом decision bar `t` runtime формирует immutable primitives.

Snapshot fields фиксируются в начале bar step и не меняются внутри step:

```text
state_at_bar_start
held_pos_at_bar_start
confirmed_legs_at_bar_start
```

7.1. `candidate_threshold_ok[t]`

```text
candidate_threshold_ok =
    not daily_reset_event[t]
    and isfinite(candidate_height_pct[t])
    and candidate_height_pct[t] >= candidate_trigger_threshold
```

7.2. `duration_ok[t]`

```text
duration_ok =
    true, если gate disabled
    candidate_age_bars[t] > 0 and candidate_age_bars[t] <= max_bars, если gate enabled
```

Boundary:

```text
age == max_bars     -> PASS
age == max_bars + 1 -> FAIL
age == 0            -> FAIL при enabled gate
age == -1           -> FAIL при enabled gate
reset bar           -> FAIL при enabled gate
```

7.3. `candidate_component_ok[t]`

```text
candidate_component_ok = candidate_threshold_ok and duration_ok
```

7.4. `confirmed_median_ok[t]`

Существующее B-condition:

```text
confirmed_median_ok =
    not daily_reset_event[t]
    and confirmed_median_condition[t]
```

7.5. `b_component_ok[t]`

```text
b_component_ok = confirmed_median_ok
```

7.6. `immediate_allowed[t]`

```text
immediate_allowed =
    candidate_leg_direction[t] in {-1, +1}
    and trade_mode_allows(candidate_leg_direction[t])
```

`trade_mode_allows`:

```text
dir +1 allowed by long, both, revers
dir -1 allowed by short, both, revers
```

7.7. Reset guard summary

На reset-баре:

```text
candidate_threshold_ok = false
confirmed_median_ok = false
b_component_ok = false
candidate_component_ok = false
immediate_allowed = false
```


8. Mode Semantics
-----------------
Mode-specific logic выполняется только если `state_at_bar_start == OFF` и
`daily_reset_event[t] == false`.

Если state не OFF, repeated triggers не стартуют новый lifecycle:

```text
trade_filter_trigger_source[t] = "none"
immediate_candidate_entry_used[t] = 0
immediate_candidate_entry_block_reason[t] = "state_not_off"
```

8.1. Mode A

Start:

```text
candidate_component_ok == true
```

Transition:

```text
OFF -> WAIT_FIRST_ST_FLIP
trigger_source = "candidate_threshold"
```

First trade direction берётся из первого допустимого ST flip, не из
candidate-ноги.

8.2. Mode B

Start:

```text
b_component_ok == true
```

Transition:

```text
OFF -> WAIT_FIRST_ST_FLIP
trigger_source = "confirmed_median"
```

Duration gate не влияет на runtime decisions Mode B.

8.3. Mode C

Start:

```text
candidate_component_ok == true
and immediate_allowed == true
```

Transition:

```text
OFF -> ST_ACTIVE_FREEZE
held_pos = candidate_leg_direction[t]
confirmed_legs_since_start = 0
trigger_source = "candidate_threshold"
immediate_candidate_entry_used = 1
```

Если candidate component прошёл, но immediate заблокирован direction/trade_mode,
state остаётся OFF. Fallback в WAIT в чистом Mode C запрещён.

8.4. Mode A+B

Компоненты:

```text
A component = candidate_component_ok
B component = b_component_ok
```

Таблица:

| A component | B component | transition | trigger_source | used |
| --- | --- | --- | --- | --- |
| false | false | OFF stays OFF | none | 0 |
| true | false | OFF -> WAIT | candidate_threshold | 0 |
| false | true | OFF -> WAIT | confirmed_median | 0 |
| true | true | OFF -> WAIT | both | 0 |

Gate-blocked A не считается сработавшей A-веткой для `trigger_source`.

8.5. Mode C+B

Компоненты:

```text
C component = candidate_component_ok
B component = b_component_ok
```

Приоритет C над B:

| C component | immediate_allowed | B component | transition | trigger_source | used |
| --- | --- | --- | --- | --- | --- |
| false | - | false | OFF stays OFF | none | 0 |
| false | - | true | OFF -> WAIT | confirmed_median | 0 |
| true | true | false | OFF -> FREEZE | candidate_threshold | 1 |
| true | true | true | OFF -> FREEZE | both | 1 |
| true | false | false | OFF stays OFF | none | 0 |
| true | false | true | OFF -> WAIT | both | 0 |

Если C не прошёл duration gate, но B сработал:

```text
OFF -> WAIT
trigger_source = "confirmed_median"
used = 0
immediate_candidate_entry_block_reason = "duration_gate_failed"
```

Gate-blocked C не считается candidate source для `trigger_source`, но
учитывается в immediate-blocked diagnostics.


9. FSM Step Ordering
--------------------
Canonical bar step:

```text
0. Capture immutable snapshots:
   state_at_bar_start, held_pos_at_bar_start, confirmed_legs_at_bar_start

1. If daily_reset_event[t]:
   state = OFF
   held_pos = 0
   confirmed_legs_since_start = -1
   no mode-specific OFF transition on this bar

2. If state_at_bar_start == OFF and not reset:
   run mode-specific OFF transition

3. WAIT_FIRST_ST_FLIP -> ST_ACTIVE_FREEZE on allowed ST flip
   Direction comes from ST flip.
   This step does not overwrite same-bar Mode C immediate held_pos.

4. Confirm-event counter increment only if state_at_bar_start was already
   ST_ACTIVE_FREEZE or ST_ACTIVE_MONITORING.
   Same-bar Mode C immediate entry does not count same-bar confirm.

5. FREEZE -> MONITORING when confirmed_legs_since_start >= freeze_confirmed_legs.
   freeze_confirmed_legs == 0 may move FREEZE -> MONITORING on the same
   decision step as entry.

6. MONITORING -> STOPPING on confirm-bar median check.
   If step 5 moved FREEZE -> MONITORING on this same bar, step 6 must not fire.

7. Update held_pos from ST flips only if state_at_bar_start was already
   ST_ACTIVE_FREEZE or ST_ACTIVE_MONITORING.
   Same-bar opposite ST flip on Mode C immediate entry bar does not rewrite
   candidate-direction held_pos.

8. Write filtered_positions[t + 1] if in bounds.
```

OPEN_TO_OPEN remains unchanged:

```text
filtered_positions[t + 1] = held_pos
```

Last-bar immediate entry is allowed in diagnostics, but no out-of-bounds
position write occurs.


10. Diagnostics
---------------
10.1. Existing `filter_block_reason`

Existing `filter_block_reason` whitelist is not changed:

```text
none
local_median_unavailable
stopping_mode_no_new_entries
filter_off
trade_mode_disallowed_flip
daily_reset
```

This field describes existing FSM/ST-level block reasons. It does not encode
candidate immediate-entry block reasons.

10.2. New immediate diagnostics

Add per-bar arrays:

```text
zigzag_mode                               object/string
candidate_age_bars                       int64
candidate_leg_direction                  int8
candidate_duration_gate_enabled          int8
candidate_duration_max_bars              int64     # -1 when disabled
candidate_duration_gate_passed           int8
candidate_threshold_ok                   int8
candidate_component_ok                   int8
confirmed_median_ok                      int8
b_component_ok                           int8
immediate_allowed                        int8
immediate_candidate_entry_used           int8
immediate_candidate_entry_block_reason   object/string
```

Disabled filter path:

- no new diagnostic arrays are created;
- no new XLSX columns are emitted;
- disabled baseline remains bit-identical.

10.3. `candidate_duration_gate_passed`

```text
candidate_duration_gate_passed = 1 if duration_ok else 0
```

Special cases:

- gate disabled -> always `1`;
- pure Mode B -> always `1`, even when gate is enabled in config.

10.4. `immediate_candidate_entry_block_reason`

Whitelist:

```text
daily_reset
filter_off
state_not_off
mode_not_c
height_gate_failed
duration_gate_failed
unknown_candidate_direction
trade_mode_disallows_direction
none
```

Priority, highest first:

```text
1. daily_reset
2. filter_off
3. state_not_off
4. mode_not_c
5. height_gate_failed
6. duration_gate_failed
7. unknown_candidate_direction
8. trade_mode_disallows_direction
9. none
```

`none` is allowed only when immediate entry was actually used:

```text
immediate_candidate_entry_used[t] == 1
<=>
immediate_candidate_entry_block_reason[t] == "none"
```

In modes `A`, `B`, `A+B`:

```text
immediate_candidate_entry_used = 0
immediate_candidate_entry_block_reason = "mode_not_c"
```

Reset has highest priority in every mode.

10.5. Trigger source invariant

Allowed values:

```text
none
candidate_threshold
confirmed_median
both
```

Strict invariant:

```text
trade_filter_trigger_source[t] != "none"
<=>
FSM реально ушёл из OFF на этом баре and not daily_reset_event[t]
```

`ZigZag_Trigger_Events` строится только из строк, где
`trade_filter_trigger_source != "none"`.

11.2. `ZigZag_Trigger_Events`

Строки создаются только для `trade_filter_trigger_source != "none"`.

Лист должен корректно поддерживать:

- legacy WAIT starts (`A`, `B`, `A+B`);
- same-bar immediate starts (`C`, `C+B`);
- C+B B-rescue starts, где immediate заблокирован, но B запускает WAIT.

Существующую reconstruction formula нельзя оставлять завязанной только на
переход WAIT -> FREEZE. Для Mode C lifecycle start может быть same-bar
`OFF -> ST_ACTIVE_FREEZE` без WAIT.

Добавить колонки:

```text
ZigZag Mode
Immediate Candidate Entry Used
Immediate Candidate Entry Block Reason
Candidate Age Bars
Candidate Leg Direction
Candidate Duration Gate Passed
```

11.3. `filters_summary`

Params section добавить:

```text
ZigZag Mode
Candidate Duration Gate Enabled
Candidate Duration Max Bars
```

Period section добавить:

```text
Immediate Entries Count
Immediate Entries Blocked Count
```

Formulas:

```text
Immediate Entries Count =
    sum(immediate_candidate_entry_used == 1)

Immediate Entries Blocked Count =
    sum(immediate_candidate_entry_block_reason in {
        duration_gate_failed,
        unknown_candidate_direction,
        trade_mode_disallows_direction,
    })
```

Blocked Count не растёт от:

```text
height_gate_failed
mode_not_c
state_not_off
daily_reset
filter_off
none
```

Existing summary counters based on `filter_block_reason` remain separate and
do not become immediate-entry counters.


12. Bit-Identity / Anti-Drift
-----------------------------
Must remain unchanged:

- OPEN_TO_OPEN execution model;
- `trade_mode` whitelist;
- `_is_first_flip_allowed`;
- `_update_held_pos`;
- `filter_block_reason` whitelist;
- `ZigZagFSMState` IntEnum codes;
- disabled filter path outputs;
- Mode A without gate vs existing candidate-threshold behavior;
- Mode B without gate vs existing confirmed-median behavior;
- Mode A+B without gate vs existing both-trigger behavior;
- Mode B + enabled gate vs Mode B without gate, except one INFO log and
  metadata/config echo.


13. Acceptance Criteria
-----------------------
A. Config / schema

A1. `mode: A/B/C/A+B/C+B` валидны.
A2. Lowercase or unknown mode -> ConfigError.
A3. Нет mode и нет legacy triggers -> materialized mode A.
A4. Legacy candidate-only -> mode A.
A5. Legacy confirmed-only -> mode B.
A6. Legacy both -> mode A+B.
A7. Explicit mode + any `trade_filter.triggers` -> ConfigError.
A8. `candidate_entry` anywhere under `trade_filter.zigzag` -> ConfigError.
A9. `candidate_duration_gate.enabled` non-bool -> ConfigError.
A10. `enabled: true` без `max_bars` -> ConfigError.
A11. `max_bars` bool/float/string/null/<1 -> ConfigError.
A12. `enabled: false` + explicit `max_bars` -> ConfigError.
A13. Absent gate block -> disabled gate.
A14. `mode: B` + enabled gate + valid max_bars -> valid.

M. Materialization / metadata

M1. `zigzag_mode` materialized in `ZigZagGlobalStats`.
M2. `candidate_duration_gate_enabled/max_bars` materialized.
M3. Disabled gate uses `None` in dataclass and metadata.
M4. Diagnostics/XLSX representation uses `-1` for disabled max bars.
M5. `metadata.config_snapshot` contains `zigzag_mode` and gate object.

C. Per-bar candidate state

C1. `candidate_age_bars == -1` on pre-bootstrap, reset, pathological close,
    UNKNOWN direction.
C2. `candidate_leg_direction == 0` on the same UNKNOWN cases.
C3. `candidate_height_pct == NaN` on the same UNKNOWN cases.
C4. Confirm-bar `candidate_leg_direction` is opposite confirmed leg direction.
C5. Confirm-bar `candidate_age_bars >= 2`.

P. Runtime primitives

P1. Reset bar forces candidate/B/immediate primitives false.
P2. Gate disabled -> `duration_ok == true`.
P3. Gate enabled age `max_bars` -> pass.
P4. Gate enabled age `max_bars + 1`, `0`, `-1` -> fail.
P5. Mode B ignores duration gate in runtime decisions.
P6. Mode B + enabled gate emits exactly one INFO line per run.
P7. Snapshots are immutable during one FSM step.

T. Mode transition tables

T1. Mode A table produces OFF/WAIT and trigger_source exactly as spec.
T2. Mode B table produces OFF/WAIT and trigger_source exactly as spec.
T3. Mode C success opens FREEZE immediately with candidate direction.
T4. Mode C blocked by unknown direction or trade_mode stays OFF, no WAIT.
T5. Mode A+B table matches §8.4.
T6. Mode C+B table matches §8.5.
T7. C+B immediate success has priority over B WAIT.
T8. C+B B-rescue enters WAIT when C is blocked and B fired.

F. FSM ordering

F1. Same-bar opposite ST flip on Mode C entry bar does not rewrite held_pos.
F2. Same-bar confirm on Mode C entry bar does not increment confirmed counter.
F3. Repeated triggers outside OFF do not start new lifecycle.
F4. Last-bar immediate entry records FREEZE in diagnostics and does not write
    out of bounds.
F5. `freeze_confirmed_legs == 0` can move FREEZE -> MONITORING same step, but
    MONITORING -> STOPPING cannot fire on that same bar.

D. Diagnostics

D1. `used == 1 <=> immediate_reason == "none"`.
D2. `used == 1` only in `C` or `C+B`.
D3. `trigger_source != none` iff actual OFF departure and not reset.
D4. A/B/A+B immediate reason is `mode_not_c`.
D5. Reset reason priority is `daily_reset`.
D6. `filter_block_reason` whitelist unchanged.
D7. C+B B-rescue by direction/trade_mode has immediate blocked reason and
    trigger_source `both`.
D8. C+B B-rescue by duration has immediate reason `duration_gate_failed` and
    trigger_source `confirmed_median`.

X. XLSX / summary

X1. New diagnostics columns appear only when filter enabled.
X2. Disabled filter path emits no new diagnostics/XLSX columns.
X3. Trigger events include Mode C same-bar immediate starts.
X4. Trigger events include C+B B-rescue WAIT starts.
X5. `Immediate Entries Count == sum(used == 1)`.
X6. `Immediate Entries Blocked Count` counts only the three immediate block
    reasons listed in §11.3.
X7. Existing blocked-entry counters remain based on `filter_block_reason`.

I. Regression / inherited behavior

I1. Mode A without gate bit-identical existing candidate-threshold mode.
I2. Mode B without gate bit-identical existing confirmed-median mode.
I3. Mode A+B without gate bit-identical existing both-trigger mode.
I4. Mode B + enabled gate bit-identical Mode B without gate except INFO and
    metadata/config echo.
I5. `_is_first_flip_allowed` unchanged.
I6. `_update_held_pos` unchanged.
I7. OPEN_TO_OPEN unchanged.
I8. `ZigZagFSMState` codes unchanged.
