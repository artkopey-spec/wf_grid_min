# XLSX Export Spec for WF Grid Search

## 1. Назначение

Этот документ задаёт контракт экспорта результатов WF grid search в XLSX.

Спецификация описывает:

- **состав книги** (сколько листов, какие, что на каждом) — §2;
- промежуточный long-формат;
- правила построения wide-таблицы;
- правила ранжирования;
- обязательный порядок колонок;
- UX-настройки Excel.

Этот spec предполагает, что вычислительное ядро, walk-forward и сбор long-данных уже выполнены, а export-layer получает готовые long-записи.

---

## 2. Итого: листы книги экспорта

Экспорт по умолчанию — **полная книга** (как в референсе 2.0: `excel_walk_forward` / вырезка `REFERENCE_excel_wf_sheets_excerpt.py`). Отдельного режима «только `summary`» **нет**: альтернативные варианты состава книги не поддерживаются.

### 2.0 Сводная таблица

| | Число листов | Имена листов |
|---|----------------|--------------|
| **Полная книга (единственный контракт)** | **4 + N** (+ **1**, если создан лист `WF_Gates`) | `WF_Config` + `WF_Gates` (условно) + `WF_01`…`WF_N` + `WF_Trades` + `WF_Train_Trades` + **`summary`** |

- **N** — число шагов walk-forward (один лист на шаг: `WF_01` … `WF_N` с ведущим нулём).
- Лист **`summary`** всегда входит в книгу; остальные листы — по правилам §2.2–2.6.

### 2.1 Лист `summary` (обязателен)

- **Назначение:** step-level / **wide** представление для отчёта: одна строка wide = одна точка сетки (`atr_period`, `multiplier`, `trade_mode`); сегментные колонки `S1_*`…`SN_*`; ранжирование и decision columns.
- **Содержимое:** только **OOS**-данные; train-only не смешиваются (см. §3.1.1).
- **Детальная схема колонок, long→wide, блоки A/B/C, сортировка:** §4–§12 этого документа.

### 2.2 Лист `WF_Config`

- **Назначение:** снимок конфигурации эксперимента для воспроизводимости.
- **Содержимое:** три колонки — `Section`, `Parameter`, `Value`; вложенный YAML развёрнут в плоские строки; сортировка по `Section`, `Parameter` (стабильная).
- **Реализация:** референс `_write_config_sheet` / `_flatten_config_to_rows`.

### 2.3 Лист `WF_Gates` (условно)

- **Назначение:** человекочитаемая сводка проверок gate на уровне прогона (как в 2.0 — см. `gates_result.checks`).
- **Содержимое (минимум):** `Name`, `Passed` (`PASS`/`FAIL`), `Value`, `Threshold`, `Message`.
- **Условие:** лист создаётся, если в конфиге включены соответствующие гейты и есть объект результата гейтов; иначе лист можно не писать или писать пустой — **зафиксировать одно поведение в билде**.
- **Примечание:** в 2.0 после записи возможно **дополнение** строк диагностикой canonical/coverage (`_append_cag_diagnostics_to_gates`); при переносе в grid-проект либо воспроизвести, либо явно исключить в ТЗ.

### 2.4 Листы `WF_01` … `WF_N`

- **Назначение:** детализация **по шагам** WF (в 2.0 — таблица TOP-K кандидатов на шаг).
- **Содержимое (референс 2.0):** колонки в строгом порядке: `rank`, `atr_period`, `multiplier`, `entry_train_objective`, `step_best_train_objective`, `train_sum_pnl_pct`, `robust_score`, `oos_sortino`, `oos_sum_pnl_pct`, `worst_oos_max_dd_pct`, `oos_num_trades`, `in_consensus`.
- **Grid-проект:** если на шаг приходится **одна** выбранная точка сетки, схему колонок адаптировать (например добавить `grid_point_id`), сохранив смысл «один лист = один wf_step».

### 2.5 Лист `WF_Trades`

- **Назначение:** все **OOS**-сделки, собранные по шагам (long-таблица сделок).
- **Содержимое:** колонки как у `trades_df` из бэктеста; обязательно иметь идентификаторы шага и точки сетки (`wf_step`, `grid_point_id` или эквивалент), чтобы не смешивать шаги; datetime без timezone для Excel.

### 2.6 Лист `WF_Train_Trades`

- **Назначение:** все **train**-сделки по тем же правилам, что `WF_Trades`, для окон train.

### 2.7 Порядок листов

1. `WF_Config`  
2. `WF_Gates` (если есть)  
3. … прочие служебные листы продукта (если появятся) …  
4. `WF_01` … `WF_N`  
5. `WF_Trades`  
6. `WF_Train_Trades`  
7. **`summary`** — удобно **в конце** или **в начале**; зафиксировать в реализации (во всех экспортах присутствуют все листы из §2.0, кроме опциональных по условиям §2.3).

## 3. Общие правила (лист `summary`)

### 3.1. Лист результата
- Имя листа: `summary`

Книга XLSX всегда включает **полный набор листов** по §2.0; лист `summary` — часть этой книги, не отдельный экспорт.

### 3.1.1. Содержимое `summary`: только OOS

- Long-вход и wide-выход на листе `summary` строятся **исключительно из OOS-сегментов** walk-forward (одна строка long = один OOS-сегмент для точки сетки).
- Train-only метрики и train-колонки **не** смешиваются с этим листом; train и сделки train — на листах `WF_Train_Trades` и в шаговых листах по §2.

### 3.1.2. Полный перебор и ранжирование

- Внутри **одной точки сетки** отдельной «оптимизации» нет: для каждой пары параметров считается полный WF по OOS.
- Задача эксперимента — **перебор сетки + ранжирование/отбор сильных точек** на выходе, а не поиск внутри точки.

### 3.1.3. `objective_metric` и отчётные агрегаты

- Поле `objective_metric` в конфиге задаёт, **какая метрика считается главной для смысла эксперимента** (например `sortino` или `pnl`).
- Для **сравнения точек в отчёте** при `objective_metric = sortino` сводный показатель по сегментам — **медиана Sortino по OOS-сегментам** (`sortino_Median` в wide).
- **`tester_seed_score`** (и правила `tester_grid_selection`) **не связаны** с какими-либо иными внутренними скорами структурной устойчивости; при включённом score ранжирование по `grid_rank` в режиме `gates_score` опирается на **`tester_seed_score` по убыванию** (см. §9.2).

### 3.2. Гранулярность wide-строки
- Одна строка wide-таблицы соответствует одной точке сетки:
  - `atr_period`
  - `multiplier`
- При этом `trade_mode` фиксирован в рамках строки и хранится как часть идентичности.
- Строки после ранжирования не удаляются: в итоговом листе остаются все точки сетки.

### 3.3. Сегменты
- Сегменты соответствуют OOS-окнам walk-forward.
- `segment_label` имеет вид `S1`, `S2`, ..., `SN`.
- `n_segments` определяется конфигурацией `validation.walk_forward` и не задаётся вручную в export-layer.
- В рамках одного эксперимента `n_segments` считается фиксированным для всех точек сетки.
- Wide-схема должна иметь единый набор колонок `S1_* ... SN_*`.

### 3.4. Sentinel policy
- Невалидные значения метрик кодируются sentinel-значением, например `-999.0`.
- При расчёте агрегатов по сегментам sentinel должен быть предварительно заменён на `NaN`.
- Sentinel не должен интерпретироваться как валидное числовое значение для статистик и profitable-count.

## 4. Входной формат: long-table

Экспорт в XLSX должен строиться из промежуточной long-таблицы.

Одна строка long-таблицы соответствует одной паре:

- `(grid_point, segment)`  
или эквивалентно:
- `(atr_period, multiplier, trade_mode, segment_index)`

### 4.1. Обязательные поля long-формата

#### Идентичность и сегмент
- `atr_period` (int)
- `multiplier` (float)
- `trade_mode` (str)
- `segment_label` (str: `S1`, `S2`, ..., `SN`)
- `segment_index` (int, `1..N`)
- `range_label`
- `n_bars`
- `start_date`
- `end_date`
- `prepend_bars`

#### Warmup fields
- `warmup_used`
- `warmup_effective`

Эти поля обязательны в long, если они нужны downstream-диагностике или экспорту.

#### Метрики сегмента
- `sum_pnl_pct`
- `sharpe`
- `sortino`
- `max_drawdown`
- `cagr`
- `win_rate`
- `num_trades`
- `profit_factor`
- `avg_trade`
- `max_losing_streak`
- `max_losing_trade`
- `worst_streak_pnl`

### 4.2. Типы
- даты допускаются как datetime / pandas Timestamp / ISO-like string;
- числа сохраняются как numeric values до этапа Excel-formatting;
- `trade_mode` обязателен в каждой строке long.

---

## 5. Построение wide-таблицы из long

Для каждой уникальной комбинации:

- `atr_period`
- `multiplier`
- `trade_mode`

должна быть построена одна строка wide-таблицы.

### 5.1. Группировка
1. Сгруппировать long по:
   - `atr_period`
   - `multiplier`
   - `trade_mode`
2. Внутри группы отсортировать строки по `segment_index` по возрастанию.

### 5.2. Базовые колонки wide-строки
В wide-строку должны быть записаны:

- `atr_period`
- `multiplier`
- `trade_mode`
- `n_segments`

где `n_segments` — число OOS-окон в рамках эксперимента.

### 5.3. Сегментные блоки
Для каждого сегмента с меткой `L = segment_label` должны добавляться колонки с префиксом `{L}_`.

Для сегмента `L` обязательны колонки:

- `{L}_segment_label`
- `{L}_range_label`
- `{L}_n_bars`
- `{L}_start_date`
- `{L}_end_date`
- `{L}_prepend_bars`

- `{L}_sum_pnl_pct`
- `{L}_sharpe`
- `{L}_sortino`
- `{L}_max_drawdown`
- `{L}_cagr`
- `{L}_win_rate`
- `{L}_num_trades`
- `{L}_profit_factor`
- `{L}_avg_trade`

- `{L}_max_losing_streak`
- `{L}_max_losing_trade`
- `{L}_worst_streak_pnl`

Если в схеме long присутствуют `warmup_used` и `warmup_effective`, допускается экспортировать также:
- `{L}_warmup_used`
- `{L}_warmup_effective`

### 5.4. Порядок сегментов
Сегменты должны идти в wide строго по порядку:
- `S1`
- `S2`
- ...
- `SN`

Сортировка должна опираться на `segment_index`, а не на строковую сортировку `segment_label`.

## 6. Сводные статистики по сегментам

Для каждой метрики из набора:

- `sum_pnl_pct`
- `sharpe`
- `sortino`
- `max_drawdown`
- `cagr`
- `win_rate`
- `num_trades`
- `profit_factor`
- `avg_trade`
- `max_losing_streak`
- `max_losing_trade`
- `worst_streak_pnl`

должны быть вычислены сводные статистики по сегментам.

### 6.1. Подготовка ряда
Для каждой метрики `m`:
1. взять значения по сегментам;
2. заменить sentinel-значения на `NaN`;
3. считать статистики только по валидным числам.

### 6.2. Набор статистик
Для каждой метрики `m` должны быть созданы колонки:

- `{m}_Mean`
- `{m}_Std`
- `{m}_Min`
- `{m}_Max`
- `{m}_Median`

### 6.3. Правила расчёта
- `Std` рассчитывается только если валидных значений не меньше 2.
- Если валидных значений меньше 2, `{m}_Std = NaN`.
- Если валидных значений нет, все агрегаты по этой метрике должны быть `NaN`.

### 6.4. Просадка: семантика `max_drawdown_Min` (блок B)

Имя `{m}_Min` для метрики `max_drawdown` **не означает** «самую маленькую просадку в абсолютном значении без контекста знака».

- **`max_drawdown_Min` в блоке B (§10.2)** — это **наихудшая просадка среди OOS-сегментов** (наиболее тяжёлое падение / убыток от пика в принятой в ядре семантике).
- Численно это тот же агрегат `{m}_Min`, что и в §6.2, при **соглашении проекта** о знаке просадки (например отрицательные значения → минимум = наихудший сегмент). Если в ядре просадка задаётся положительной величиной, правило «наихудший сегмент» должно быть явно сопоставлено с парой `Min`/`Max` в коде агрегации и **единым образом** для long, wide и блока B.

---

## 7. profitable_segments_count

Должна быть вычислена колонка:

- `profitable_segments_count`

Правило:
- считается число сегментов, где `{L}_sum_pnl_pct` валиден и строго больше 0.

Не считаются прибыльными:
- sentinel-значения;
- `NaN`;
- нулевой PnL.

---

## 8. Ранжирование и decision columns

Wide-таблица должна проходить этап ранжирования.

### 8.1. Общие требования
- `grid_rank` должен быть первой колонкой итогового DataFrame.
- Ранг должен быть плотным: `1 ... G`
- `1 = лучший`

### 8.2. Обязательные decision columns
Минимально должны присутствовать колонки:

- `grid_rank`
- `ranking_mode`
- `score_contract_status`
- `quantile_gates_status`

- `seed_gate_passed`
- `tester_seed_score`

- `gate_ok_positive_median`
- `gate_ok_min_trades`
- `gate_ok_worst_segment`
- `gate_ok_drawdown`

- `seed_gate_fail_reason`

### 8.3. Допустимые значения ranking_mode
В коде должен быть зафиксирован ограниченный список допустимых режимов, например:
- `legacy`
- `gates_score`

При необходимости список может быть расширен, но значения должны быть централизованно определены, а не генерироваться произвольно.

---

## 9. Правила сортировки

### 9.1. Legacy mode
Если score/gates выключены, используется fallback-ранжирование:

1. сортировка по убыванию `sort_by`
   - по умолчанию: `sum_pnl_pct_Median`
2. затем по убыванию tiebreaker
   - по умолчанию: `sum_pnl_pct_Min`
3. затем по возрастанию `sum_pnl_pct_Std`
4. `NaN` должны уходить в конец

### 9.2. Gates/score mode
Если gates/score включены:
1. логически выделить точки, прошедшие гейты;
2. прошедшие ранжировать по `tester_seed_score` по убыванию;
3. непрошедшие оставлять ниже;
4. для непрошедших использовать fallback-сортировку, совместимую с legacy.

### 9.3. Сохранение всех строк
Независимо от режима:
- строки не удаляются;
- провалившиеся по gates остаются в wide-листе;
- меняется только порядок и decision columns.

## 10. Порядок колонок в итоговом Excel

Перед записью в XLSX должна выполняться явная перестановка колонок.

### 10.1. Блок A — identity + rank
Колонки должны идти в таком порядке, если они присутствуют:

- `grid_rank`
- `ranking_mode`
- `score_contract_status`
- `quantile_gates_status`
- `atr_period`
- `multiplier`
- `trade_mode`
- `n_segments`
- `profitable_segments_count`
- `seed_gate_passed`
- `tester_seed_score`
- `gate_ok_positive_median`
- `gate_ok_min_trades`
- `gate_ok_worst_segment`
- `gate_ok_drawdown`
- `seed_gate_fail_reason`

### 10.2. Блок B — key decision summaries
После блока A должны идти ключевые сводные колонки, если они существуют:

- `sum_pnl_pct_Median`
- `sum_pnl_pct_Min`
- `max_drawdown_Min` — **наихудшая просадка** среди OOS-сегментов (см. §6.4)
- `max_losing_streak_Max`
- `max_losing_trade_Min`
- `profit_factor_Median`
- `num_trades_Median`
- `sum_pnl_pct_Std`

### 10.3. Блок C — остальные колонки
Все остальные колонки должны идти после блоков A и B в исходном порядке.

Сюда попадают:
- все `S*_*` колонки;
- остальные агрегаты `{metric}_Mean/_Std/_Min/_Max/_Median`;
- дополнительные диагностические поля.

## 11. Excel UX

Следующие UX-улучшения рекомендуются как часть export layer.

### 11.1. Автофильтр
На строку заголовков должен быть установлен auto-filter.

### 11.2. Freeze panes
Слева должны быть закреплены все подряд идущие колонки из множества:

- `grid_rank`
- `ranking_mode`
- `score_contract_status`
- `quantile_gates_status`
- `atr_period`
- `multiplier`
- `trade_mode`
- `n_segments`
- `profitable_segments_count`
- `seed_gate_passed`
- `tester_seed_score`
- все `gate_ok_*`

Закрепление должно выполняться до первой колонки, которая не входит в это множество.

### 11.3. Optional conditional formatting
Допускается:
- heatmap по числовым summary-колонкам;
- отдельная заливка колонок, совпадающих с regex `^S\d+_sum_pnl_pct$`.

Эти UX-улучшения являются желательными, но не обязательными, если это отдельно не закреплено в основном ТЗ.

---

## 12. Минимальные инварианты

Экспорт считается корректным, если выполняются следующие условия:

1. В итоговом XLSX присутствует **полная книга** по §2.0 (включая `summary` и `WF_*`-листы); см. §2 и §3.1.
2. Данные на `summary` отражают **только OOS**-сегменты WF (см. §3.1.1).
3. В wide-таблице одна строка соответствует одной уникальной точке:
   - `atr_period`
   - `multiplier`
   - `trade_mode`
4. `grid_rank` — первая колонка.
5. Все строки grid остаются в таблице после ранжирования.
6. Сегментные колонки строятся по фиксированной схеме `S1_* ... SN_*`.
7. Агрегаты по метрикам считают статистику только по валидным значениям.
8. `profitable_segments_count` не учитывает sentinel и `NaN`.
9. Колонки блоков A и B стоят в обязательном порядке.
10. Остальные колонки следуют после них без потери данных.

## 13. Рекомендация по включению в ТЗ

Этот документ рекомендуется использовать как отдельное приложение к основному ТЗ:

- основной ТЗ содержит краткое требование об XLSX-экспорте;
- данный spec-файл прикладывается как подробный контракт export-layer.

Рекомендуемое имя файла:
- `xlsx_export_spec.md`

При реализации экспорта листов `WF_*` (§2.2–2.6) рекомендуется приложить вырезку референсного кода 2.0, например `donor/REFERENCE_excel_wf_sheets_excerpt.py`, к ТЗ или к репозиторию — чтобы зафиксировать колонки и форматирование записи.
