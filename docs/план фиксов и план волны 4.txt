## Волна 1 — БЛОКЕРЫ (без них система опасна даже для research)

### FIX-1.1. Defensive fallback: effective_oos_bars contamination

**Severity**: Critical
**Файлы**: `wf/step_executor.py`
**Проблема**: `_defensive_fallback` устанавливает `effective_oos_bars = len(ext_result.returns)` — длина extended массива, включая prepend-бары. Downstream `assign_step_status` видит завышенное число баров, может присвоить `ok` шагу с нулём реальных OOS-баров. Метрики при этом содержат prepend-данные.

**Решение**:

* В `_defensive_fallback` вычислять `effective_oos_bars = max(0, len(ext_result.returns) - prepend_bars_applied)` если returns не None, иначе 0.
* Альтернативный (более безопасный) вариант: при defensive fallback всегда ставить `effective_oos_bars = 0`. Это гарантирует статус `insufficient_bars` для любого defensive fallback шага.
* Добавить warning-лог с указанием `actual_oos_bars_estimate` и `ext_total_bars` для диагностики.

**Критерий готовности**: Defensive fallback шаг никогда не получает статус `ok`. `effective_oos_bars` отражает только OOS-бары, не prepend.

**Тесты**:

* Early exit в prepend-зоне → defensive fallback → `effective_oos_bars == 0` → статус `insufficient_bars`.
* Boundary case: `oos_boundary == len(ext_returns)` → `effective_oos_bars == 0`.
* None arrays → `effective_oos_bars == 0`.
* Defensive fallback шаг → aggregation не включает его (ok mask фильтрует).

**Тесты**:

### FIX-1.3. resolved_periods_per_year silent fallback 252

**Severity**: High
**Файлы**: `wf/step_executor.py` (2 места: `execute_oos_step` line 89, `execute_train_step` line 220)
**Проблема**: `config.resolved_periods_per_year or 252.0` — если resolver не вызван (config без data), молча используется 252. Crypto/FX данные требуют 365 / 8760. Все ratio-метрики (sharpe, sortino, cagr) рассчитываются с неправильной annualization.

**Решение**:

* Заменить fallback на явную проверку: если `resolved_periods_per_year is None` — бросать `ConfigError("resolved_periods_per_year must be set before execution")`.
* В orchestrator убедиться, что resolve_periods_per_year вызывается ДО WF execution (уже так, но добавить assert).

**Критерий готовности**: Невозможно запустить execution без resolved periods_per_year. Молчаливый fallback 252 удалён.

**Тесты**:

* `execute_oos_step` с config где `resolved_periods_per_year=None` → `ConfigError`.
* `execute_train_step` с config где `resolved_periods_per_year=None` → `ConfigError`.
* Orchestrator: resolve вызывается до execution loop (integration test).

### FIX-1.4. NaN в step-level gate метриках проходит gate

**Severity**: High
**Файлы**: `gates/gates.py`
**Проблема**: `df["max_drawdown"] < dd_threshold` → `NaN < -0.50` = False → step проходит gate. Шаг с NaN max_drawdown (неизвестная просадка) не блокируется drawdown gate.

**Решение**:

* В `apply_step_gates`: для каждого gate, NaN в метрике трактовать как failure. Добавить `& df[metric].notna()` в pass-condition, или `| df[metric].isna()` в fail-condition.
* Конкретно:

  * `min_trades_fail = ok_mask & (df["num_trades"].isna() | (df["num_trades"] < threshold))`
  * `dd_fail = ok_mask & (df["max_drawdown"].isna() | (df["max_drawdown"] < dd_threshold))`

**Критерий готовности**: Шаг с NaN в любой gate-метрике получает `gate_failed`.

**Тесты**:

* `max_drawdown=NaN`, `num_trades=10` → `gate_failed`.
* `max_drawdown=-0.10`, `num_trades=NaN` → `gate_failed`.
* Оба NaN → `gate_failed`.
* Не-NaN, проходящие threshold → `ok` (regression).

### FIX-4.6. Percentile-based scoring

* При `normalization="percentile"`:

  * `norm(x) = percentile_rank(x) / (n_passed - 1)` для higher_is_better.
* При `n_passed < 5` и `normalization="minmax"` → auto-switch на percentile + warning.

**Критерий готовности**: Percentile scoring доступен. Auto-switch при малых n. Оба mode тестированы.

**Тесты**:

* 2 candidates + minmax → auto-switch to percentile + warning.
* 10 candidates + percentile → scores ∈ [0, 1], rank-order preserved.
* 10 candidates + minmax → same rank-order as percentile (consistency).
