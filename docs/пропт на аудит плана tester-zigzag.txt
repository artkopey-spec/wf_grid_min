Проверь особенно:

1. Source of truth / governance
- Все ли решения плана согласованы с Appendix A v1.1.
- Нет ли мест, где план переопределяет spec вместо того, чтобы ссылаться на него.
- Есть ли work-package/design decisions без точной Spec reference.
- Не противоречит ли план обновлённому TZ v0.4 legacy-only.

2. Legacy-only scope / equal_blocks rejection
- Нет ли остаточных обещаний enabled equal_blocks integration.
- Нет ли `SegmentResult.filter_diagnostics` / `SegmentResult.filter_diagnostics_summary` как Phase 2 requirement.
- Нет ли `FilterDiagnostics_S1`, `ZigZag_Trigger_Events_S1`, equal_blocks `filters_summary` как обязательных артефактов.
- Нет ли `oos_boundary` trim / prepend warning / rebase diagnostics как обязательных Phase 2 пунктов.
- Нет ли enabled equal_blocks golden tests или acceptance gates.
- Достаточно ли явно описан reject: `equal_blocks + trade_filter.enabled=true` -> ConfigError до stats/backtest/export.
- Есть ли defensive guard для direct `run_equal_blocks(..., trade_filter_config.enabled=true)`.
- Сохраняется ли disabled equal_blocks path bit-identical baseline.

3. Engine reuse / Mode C
- Реалистичен ли Mode C как единственный путь.
- Нет ли скрытого дублирования active donor engine.
- Не остаётся ли stale/dead path в `donor TESTER/`, который может быть случайно использован.
- Достаточно ли явно описаны import/runtime boundaries между tester-only кодом и active donor engine.
- Не строятся ли global stats для rejected equal_blocks enabled config.

4. Contracts and diagnostics
- Bar-level `BacktestResult.filter_diagnostics` должен содержать только массивные поля длины `positions`.
- Disabled path должен быть строго `filter_diagnostics is None`.
- Все aggregates/scalars должны жить только в `PeriodResult.filter_diagnostics_summary`, не в bar-level dict.
- Нет ли протаскивания 2.0-TESTER keyset: `mode`, `thresholds`, `counters`, `allow_entry`, `filtered_reason`, `zz_st_*` в bar-level dict.
- Проверь dtype-контракты против active donor implementation.
- Проверь, что spec §13 keyset + служебные ключи active donor описаны без drift.

5. Runner / stats / indexing
- Правильно ли описан legacy path: `run_period` / `run_all_periods`.
- Корректен ли `global_offset` для legacy period slices.
- Правильно ли строится `ZigZagGlobalStats`: один раз на full validated dataset для legacy run, а не per-period.
- Нет ли cold-start FSM на post-trim segment-only данных.
- `OPEN_TO_OPEN` entry/exit diagnostic indexing pinned: `entry_index - 1`, `exit_index - 1`.

6. Signal events
- Не добавляет ли план `0 -> ±1` raw initialization в signals list.
- Сохраняется ли baseline bit-identity для disabled path.
- Достаточно ли определены новые columns только для enabled filter path.
- Signals sheet не должен выглядеть как raw ST trades, когда фактические trades отфильтрованы.

7. Config / CLI
- Строгая schema validation: unknown keys reject.
- Tester не должен импортировать неподходящий `load_grid_config`, если root schema другая.
- Проверь open question про место жительства `TradeFilter*` dataclass + `_validate_trade_filter`: является ли это blocker до WP-T2.
- Проверить, что current `config_tester.yaml` без `trade_filter` остаётся backward-compatible.
- Проверить, что silent no-op для unsupported `equal_blocks + enabled` невозможен.

8. Excel/export
- Условное добавление diagnostics sheets/columns только при enabled legacy path.
- Disabled legacy и disabled equal_blocks exports должны быть bit-identical baseline.
- Детерминированный column order.
- Нет ли смешения bar-level diagnostics и summary aggregates.
- Правильно ли обрабатываются close rows, `_NA` sentinel, trade -> trigger linking.
- Нет ли обязательных equal_blocks enabled sheets/columns после scope v0.4.

9. Tests / acceptance / DoD
- Достаточна ли legacy-only test matrix.
- Нет ли старой нумерации и ссылок: `#26`, `#27`, “27 tests”, старые equal_blocks enabled tests.

- WP-T7 должен быть legacy export integration, а не equal_blocks export integration.
- Есть ли golden/baseline tests для disabled path.
- Есть ли тесты на dtype/keyset/length invariant.
- Есть ли explicit regression gates против 2.0-TESTER keyset drift.
- Есть ли acceptance criteria, по которым можно точно сказать “реализация завершена”.
