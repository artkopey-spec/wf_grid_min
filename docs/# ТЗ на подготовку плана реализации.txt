# ТЗ на подготовку плана реализации WF Grid Search

---

## 1. Цель

Подготовить **план реализации** оптимизатора walk-forward с **полным перебором (grid search)** по заданной сетке гиперпараметров.

Документ должен описывать:

* этапы внедрения;
* зависимости;
* архитектуру модулей;
* артефакты;
* контракты данных;
* требования к логированию и диагностике;
* требования к автоматическим тестам;
* критерии готовности этапов;
* сопоставление со существующим кодом;
* явное разделение:

  * reuse;
  * wrapper;
  * adapter;
  * refactor;
  * new module.

Документ **не должен содержать реализацию кода**.

---

## 2. Scope

### 2.1. In scope

* Grid search по заданной сетке гиперпараметров WF.

* Walk-forward pipeline.

* Раздельная реализация:

  * **Phase A (обязательная и завершённая)**
  * **Phase B (расширение поверх Phase A)**

* Формирование:

  * **candidate-level aggregate** (canonical слой);
  * **step-level / long** (вспомогательный слой).

* OOS gates.

* Canonical ranking.

* Полный XLSX export contract:

  * `WF_Config`
  * `WF_Gates` (условно)
  * `WF_01 ... WF_N`
  * `WF_Trades`
  * `WF_Train_Trades`
  * `summary`

* Логирование, диагностика и автоматические тесты.

* Подготовка Phase A под Phase B.

---

### 2.2. Out of scope

* Optuna.
* Байесовские / adaptive оптимизаторы.
* Оптимизация внутри одной точки сетки.
* UI / визуализация.
* Distributed orchestration (если отдельно не оговорено).
* Переписывание существующих canonical модулей без необходимости.

---

## 3. Основные ограничения

* Только полный перебор (grid search).
* Один grid-point = один полный WF.
* Ranking только по candidate-level aggregate.
* Step-layer не используется напрямую для ranking.
* XLSX не является source of truth.
* Единый execution contract для всех шагов.

---

## 4. Phase A и Phase B (ключевой принцип)

### 4.1. Общий принцип

* Phase A и Phase B реализуются **отдельно**.
* Phase A должна быть:

  * полностью завершена;
  * пригодна к использованию;
  * независима от Phase B.
* Phase B — **расширение**, а не часть MVP.

---

### 4.2. Требование к Phase A

Phase A должна:

* полностью выполнять WF Grid Search;
* давать usable ranking;
* формировать полный экспорт;
* быть эксплуатационно полезной.

---

### 4.3. Требование к Phase B

Phase B:

* работает поверх Phase A;
* не ломает её контракты;
* добавляет enrichment:

  * plateau;
  * topology;
  * FIR;
  * дополнительные каналы.

---

### 4.4. Подготовка Phase A под Phase B

Phase A обязана заранее обеспечить:

* стабильный `grid_point_id`;
* стабильный candidate aggregate;
* полный step-layer;
* machine-readable статусы;
* достаточные диагностические поля;
* расширяемость схем.

---

### 4.5. Запрет на архитектурный тупик

Запрещено:

* менять identity grid points;
* ломать candidate aggregate;
* пересобирать базовые артефакты для Phase B.

Допустимо:

* добавлять поля;
* расширять схемы;
* добавлять каналы.

---

## 5. Канонический слой данных

### 5.1. Candidate-level aggregate (главный слой)

* одна строка = одна точка сетки;
* используется для:

  * ranking;
  * selection;
  * Phase B;
  * export decision layer.

---

### 5.2. Step-level / long (вспомогательный слой)

Используется для:

* диагностики;
* explainability;
* aggregation;
* построения:

  * `summary`
  * `WF_*`
  * trades layers.

---

### 5.3. Приоритет

candidate-level aggregate — authoritative слой.

---

## 6. Reuse policy

### 6.1. Обязательное переиспользование

Нельзя переписывать:

* config loader
* CSV loader
* validator
* backtest executor
* BacktestResult
* warmup logic
* timeframe logic
* WF slicer
* trades extraction
* metrics layer
* sentinel policy

---

### 6.2. Допустимые формы интеграции

* reuse
* wrapper
* adapter
* refactor (обоснованный)
* new module

---

### 6.3. Запрет дублирования

Нельзя создавать альтернативные реализации:

* backtest;
* trades;
* metrics;
* slicing;
* warmup.

---

## 7. XLSX export contract

### 7.1. Полная книга

Обязательные листы:

1. `WF_Config`
2. `WF_Gates` (если есть)
3. `WF_01 ... WF_N`
4. `WF_Trades`
5. `WF_Train_Trades`
6. `summary`

---

### 7.2. `summary`

* только OOS;
* wide формат;
* одна строка = grid point;
* содержит:

  * rank;
  * decision columns;
  * `S1...SN`;
  * агрегаты;
* не удаляет строки.

---

### 7.3. `WF_Config`

* flattened config;
* deterministic order.

---

### 7.4. `WF_01...WF_N`

* один шаг WF;
* step-level представление;
* включает train + OOS.

---

### 7.5. `WF_Trades`

* все OOS trades;
* содержит идентичность точки и шага.

---

### 7.6. `WF_Train_Trades`

* все train trades.

---

### 7.7. `WF_Gates`

* run-level gates;
* условный лист.

---

### 7.8. UX

* autofilter;
* freeze panes;
* deterministic column order.

---

---

## 25. Финальное требование

План реализации должен:

* использовать candidate aggregate как основу;
* использовать step-layer как вспомогательный слой;
* переиспользовать существующий код;
* реализовывать Phase A как законченный продукт;
* готовить архитектуру под Phase B;
* поддерживать полный XLSX export contract;
* быть полностью тестируемым и воспроизводимым.
