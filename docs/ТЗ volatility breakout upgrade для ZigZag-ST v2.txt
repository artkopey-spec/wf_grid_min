ТЗ v2: Volatility Breakout Upgrade для ZigZag-ST Filter
========================================================

Документ полностью заменяет v1 (`docs/ТЗ volatility breakout upgrade
для ZigZag-ST.txt`). Реализация ведётся строго по v2.

Изменения относительно v1
-------------------------
1. Duration gate упрощён до единственного режима — fixed integer.
   Удалены поля `max_bars_source`, `min_population_size`,
   `inclusive_bars`, ветка `median_above_ctt`, reference fixture
   с ожидаемой медианой 24.
2. Закреплён внутренний порядок FSM-шагов на одном баре
   (раздел "FSM Step Ordering").
3. Введена единая таблица представлений disabled / unavailable
   значений (раздел "Disabled / Unavailable Representations").
4. Унифицированы display-имена колонок XLSX между
   FilterDiagnostics_100 и ZigZag_Trigger_Events.
5. Описана семантика `candidate_leg_direction` на confirm-баре
   (направление новой ноги, противоположной только что
   подтверждённой).
6. Описана daily_reset interaction с продолжающейся
   candidate-ногой.
7. `immediate_candidate_entry_block_reason` whitelist согласован
   с существующим `filter_block_reason` там, где смысл совпадает
   (`filter_off`, `daily_reset`).
8. Зафиксировано поведение pathological bar.
9. Удалён избыточный `candidate_duration_fixed_max_bars`
   (тождественен `candidate_duration_max_bars`).
10. Allowed-keys whitelist (strict-schema validator) дополнен
    явным перечнем dotted-paths.
11. Same-bar comb-таблица trigger_source описана явно
    для всех (a_trig, b_trig, immediate_allowed) комбинаций.

Revision history (внутри v2)
----------------------------
Initial v2: переход с v1 (см. список 1–11 выше).

Audit-fix pass (после ревью v2):
- Trigger source combinations: разделены строки `immediate_allowed=false`
  по причине блокировки. mode=wait_st_flip и mode=immediate с
  invalid direction/trade_mode дают разные FSM transitions.
- `trigger_source[t]` явно ограничен правилом "не равно none только
  если FSM покинул OFF на этом баре" — устраняет утечку blocked
  баров в `ZigZag_Trigger_Events`.
- block_reason whitelist реструктурирован: `entry_mode_wait_st_flip`
  поднят в приоритете до позиции 4; добавлена гарантия, что КАЖДЫЙ
  бар получает ровно одну причину; `none` зарезервирован
  исключительно для `used == 1` — invariant `used==1 ⇔ reason==none`
  становится строгим.
- `Immediate Candidate Entries Blocked Count` сужен до содержательных
  блокировок (`duration_gate_failed`, `unknown_candidate_direction`,
  `trade_mode_disallows_direction`); `height_gate_failed` исключён
  как noise (срабатывает на большинстве баров).
- B-priority при A-blocked-by-direction зафиксирован: trigger_source
  = "both" с явным комментарием о raw-conditions семантике.
- `filter_off` явно помечен как defensive-only (нормальный disabled-
  path не создаёт ключи).
- Config rule про `max_bars: null` явно закреплён.
- Wording "candidate per-bar fields ← UNKNOWN" в FSM ordering
  заменён на "observed as UNKNOWN" — per-bar поля считаются до
  apply().
- AC F-группа дополнена тестами F6/F7 для blocked-immediate cases.

Цель
----
Добавить в ZigZag-ST фильтр опциональный режим volatility breakout:

1. Candidate-нога должна достигнуть `candidate_trigger_threshold`
   (auto-quantile или численный threshold из config — без изменений
   относительно текущей логики).
2. Опционально: ограничить максимальный возраст candidate-ноги
   на decision bar через `candidate_duration_gate.max_bars`
   (явный fixed integer).
3. При включенном immediate entry первая сделка открывается сразу
   по направлению candidate-ноги, без ожидания первого ST flip.
4. После входа дальнейшая FSM-логика остаётся прежней:
   `ST_ACTIVE_FREEZE -> ST_ACTIVE_MONITORING -> ST_STOPPING -> OFF`.

Бизнес-смысл
------------
Режим присоединения к резкому расширению волатильности:

   большая амплитуда + короткое время = импульс / volatility breakout

Текущий режим `wait_st_flip` остаётся консервативным: ZigZag даёт
сигнал, но направление подтверждается SuperTrend flip.

Новый режим `immediate_leg_direction` агрессивнее: если candidate-нога
быстро достигла высокого threshold, стратегия входит по направлению
этой ноги.

Версионирование контракта
-------------------------
`trade_filter.type` остаётся прежним:

   type: zigzag_st_mode

Новые блоки — backward-compatible расширение существующего режима.
Отдельный `zigzag_st_mode_v2` НЕ вводится.


Config
------
Добавляются два независимых блока внутри `trade_filter.zigzag`:

trade_filter:
  zigzag:
    candidate_duration_gate:
      enabled: true
      max_bars: 10                # int >= 1, required when enabled=true

    candidate_entry:
      mode: wait_st_flip          # wait_st_flip | immediate_leg_direction

Дефолты при отсутствии блоков:

   candidate_duration_gate.enabled = false
   candidate_duration_gate.max_bars = None
   candidate_entry.mode            = wait_st_flip

Правила интерпретации:

- отсутствие `candidate_duration_gate` или `enabled: false` сохраняет
  текущее поведение duration-gate bit-identical (gate тождественно
  возвращает true);
- absence `candidate_entry` означает `mode: wait_st_flip`;
- `wait_st_flip` полностью сохраняет текущую FSM-логику:
  `OFF -> WAIT_FIRST_ST_FLIP -> ST_ACTIVE_FREEZE`;
- `immediate_leg_direction` включает вход по направлению
  candidate-ноги:
  `OFF -> ST_ACTIVE_FREEZE`;
- оба блока применяются только к trigger-A
  (`candidate_threshold`);
- trigger-B (`confirmed_median`) сохраняет текущую логику ожидания
  ST flip, кроме явно описанного same-bar A+B приоритета (см.
  "Trigger Priority").


Config Validation
-----------------
1. `candidate_entry.mode` принимает только литералы:

   wait_st_flip
   immediate_leg_direction

   Любое другое значение — ConfigError.

2. `candidate_duration_gate.enabled` должен быть bool.

3. `candidate_duration_gate.max_bars`:

   - **обязателен** при `enabled: true`; отсутствие — ConfigError
     с сообщением:
     `candidate_duration_gate.max_bars is required when enabled is true`;
   - должен быть int >= 1; bool отвергается явно (как уже сделано
     для `local_window` / `freeze_confirmed_legs`);
   - **не должен присутствовать в YAML** при `enabled: false` —
     ConfigError:
     `candidate_duration_gate.max_bars must not be specified when
     enabled is false; remove the key or set enabled to true`.
     Проверка через raw_user_keys (по аналогии с numeric-threshold +
     explicit-quantile rule §11.3), не через значение поля.
     Уточнение: даже `max_bars: null` (или `max_bars: ~`)
     считается explicitly present и отклоняется. Единственный
     разрешённый при `enabled: false` вариант — полное отсутствие
     ключа `max_bars` в YAML.

4. Валидация коллапсирует все ошибки и поднимает один ConfigError —
   как существующий `validate_trade_filter`.

Whitelist updates (strict-schema):

- `_ALLOWED_KEYS["trade_filter.zigzag"]` (loader) и
  `TRADE_FILTER_ALLOWED_KEYS["trade_filter.zigzag"]` (shared)
  дополняются ключами:

      candidate_duration_gate
      candidate_entry

- Новые dotted paths:

      "trade_filter.zigzag.candidate_duration_gate":
          {"enabled", "max_bars"}
      "trade_filter.zigzag.candidate_entry":
          {"mode"}

  Любой ключ вне этих whitelist-ов отклоняется как
  `unknown config key:` ещё до валидации значений.


Duration Gate: Global Stats
---------------------------
`candidate_duration_max_bars` материализуется один раз на полном
валидированном датасете до WF execution, как `candidate_trigger_threshold`
и `global_median`. Это пробрасывание user-supplied integer без
зависимости от prices.

Шаг 0 — disabled / absent block:

   candidate_duration_gate_enabled = False
   candidate_duration_max_bars     = None    (dataclass)
                                    -1        (filter_diagnostics)

   ConfigError на population / max_bars НЕ возбуждается.

Шаг 1 — enabled == true:

   candidate_duration_gate_enabled = True
   candidate_duration_max_bars     = int(max_bars)   # validated >= 1

Снимок в `ZigZagGlobalStats` (top-level fields):

   candidate_duration_gate_enabled: bool
   candidate_duration_max_bars:     Optional[int]   # None when disabled

Снимок в `metadata.config_snapshot.candidate_duration_gate`:

   {"enabled": <bool>, "max_bars": <int|None>}

Init-failure keyset для §12.3 расширяется на:

   - `duration_gate_max_bars_missing`     (enabled=true, key absent)
   - `duration_gate_max_bars_present_when_disabled`
                                          (enabled=false, key present)
   - `duration_gate_max_bars_invalid_type`(не int / bool)
   - `duration_gate_max_bars_below_one`   (int < 1)


Per-Bar ZigZag (WP4 расширение)
-------------------------------
Добавить в `ZigZagPerBar` и `filter_diagnostics`:

   candidate_age_bars      int64
   candidate_leg_direction int8

Массивы выровнены с close.

Контракт candidate_age_bars:

   Когда direction текущей candidate-ноги известен:
       candidate_age_bars[t] = t - last_pivot_bar + 1

   UNKNOWN value:
       candidate_age_bars[t] = -1

`-1` используется ВО ВСЕХ unknown-кейсах:

   (a) pre-bootstrap (direction == UNKNOWN до первого reversal-move);
   (b) reset bar (`daily_reset_event[t] == true`);
   (c) pathological close (NaN / inf / non-positive close[t]);
   (d) explicit clear после daily_reset до новой bootstrap-ноги.

`0` НЕ используется как unknown — чтобы не путаться с "нулевым возрастом".

Поведение на confirm-баре close-only pass:

- на confirm-баре `t` last_pivot_bar становится `end_bar` только что
  подтверждённой ноги; новая candidate-нога противоположного
  направления заводится от этого pivot на том же баре;
- следствие: `candidate_age_bars[t]` на confirm-баре >= 2;
- это поведение закрепляется отдельным unit-тестом
  (см. acceptance #C2).

Контракт candidate_leg_direction:

   +1 = текущая ZigZag candidate-нога вверх
   -1 = текущая ZigZag candidate-нога вниз
    0 = направление неизвестно (UNKNOWN)

`candidate_leg_direction` всегда отражает close-only ZigZag direction
и не зависит от FSM. В активных FSM-состояниях это НЕ направление
сделки; направление сделки остаётся `held_pos` / `filtered_positions`.

Поведение на confirm-баре:

- старая нога направления `D` подтверждается;
- внутри того же бара pass выполняет flip: `direction = -D`,
  `last_pivot_bar = end_bar`, `run_ext_bar = t`;
- end-of-bar snapshot:
      `candidate_leg_direction[t] = -D`   (направление НОВОЙ ноги)
      `candidate_height_pct[t]` относится к НОВОЙ ноге
      `candidate_age_bars[t] = t - end_bar + 1` (>= 2)

Следствие для immediate entry: same-bar immediate entry на confirm-баре
открывает позицию по направлению **новой** candidate-ноги, т.е.
**противоположному** только что подтверждённой. Закрепляется
отдельным acceptance-тестом (#I4).

Pathological bar:

- `candidate_age_bars[t] = -1`            (всегда unknown, не freeze)
- `candidate_leg_direction[t] = 0`
- `candidate_height_pct[t] = NaN`
- a_trig = false, b_trig следует существующей логике
- FSM-state не мутирует на этом баре
- `confirmed_leg_idx_at_t[t]` и `last_confirmed_leg_height_pct[t]`
  сохраняют предыдущее значение (consistent с текущим контрактом WP4).


Daily Reset
-----------
При `daily_reset_event[t] == true`:

   close-only candidate state    -> UNKNOWN
   candidate_height_pct[t]       =  NaN
   candidate_leg_direction[t]    =  0
   candidate_age_bars[t]         =  -1
   a_trig                         =  false
   immediate entry               невозможен (см. block_reason priority)

Семантика для in-progress candidate-ноги:

- candidate-нога, не достигшая threshold к последнему бару
  предыдущего дня, **полностью стирается** на reset-баре нового
  дня — её height не переносится;
- первый "реальный" candidate в новой сессии возникает только
  после нового bootstrap (накопление min/max и достижение
  reversal_threshold);
- `candidate_duration_max_bars` (global stat) на это **не влияет** —
  он материализуется один раз на полном датасете и не
  пересчитывается;
- confirmed-leg history НЕ сбрасывается (как в текущем контракте).


Duration Gate: Runtime Logic
----------------------------
Trigger-A проходит duration gate только когда gate включён:

   duration_ok = (
       not candidate_duration_gate_enabled
       or (
           candidate_age_bars[t] > 0
           and candidate_age_bars[t] <= candidate_duration_max_bars
       )
   )

Базовый trigger-A:

   height_ok = (
       math.isfinite(candidate_height_pct[t])
       and candidate_height_pct[t] >= candidate_trigger_threshold
   )

   a_trig = a_enabled and height_ok and duration_ok

Boundary semantics:

- `candidate_age_bars[t] == max_bars`        -> gate PASS  (`<=`, не `<`)
- `candidate_age_bars[t] == max_bars + 1`    -> gate FAIL
- `candidate_age_bars[t] == 0` или `-1`      -> gate FAIL  (требование `> 0`)
- `candidate_age_bars[t] == -1` на reset-баре -> gate FAIL,
  но reason имеет более высокий приоритет: `daily_reset`.

Принципиальные ограничения:

- duration gate использует только текущий per-bar `candidate_age_bars[t]` —
  никакого ожидания будущих confirmed-leg событий, никакого lookahead;
- если candidate-нога превысила duration cap, trigger-A не запускает
  ни immediate entry, ни legacy `OFF -> WAIT`; независимый trigger-B,
  если он true на этом же баре, сохраняет legacy-поведение;
- repeated triggers в WAIT/FREEZE/MONITORING/STOPPING остаются
  подавленными (текущий контракт §15.3 не меняется).


Candidate Entry Modes
---------------------
Текущий режим (default):

   mode: wait_st_flip

Поведение (без изменений относительно baseline):

   candidate threshold trigger
   OFF -> WAIT_FIRST_ST_FLIP
   wait allowed ST flip
   WAIT_FIRST_ST_FLIP -> ST_ACTIVE_FREEZE

Новый режим:

   mode: immediate_leg_direction

Поведение:

   candidate threshold trigger
   OFF -> ST_ACTIVE_FREEZE
   held_pos = candidate_leg_direction
   confirmed_legs_since_start = 0

Проверка immediate entry:

   candidate_dir = candidate_leg_direction[t]

   immediate_allowed = (
       candidate_entry_mode == "immediate_leg_direction"
       and candidate_dir in (-1, +1)
       and trade_mode_allows(candidate_dir)
   )

Правила trade_mode для immediate entry:

   trade_mode == "long"            -> разрешено только candidate_dir == +1
   trade_mode == "short"           -> разрешено только candidate_dir == -1
   trade_mode in ("both", "revers") -> разрешены оба направления

Если mode == immediate_leg_direction, но направление неизвестно
(`candidate_dir == 0`) или запрещено trade_mode, trigger-A не запускает
lifecycle (даже legacy ветку OFF -> WAIT не запускает — это явное
отличие от старой семантики, где a_trig единолично переводил OFF -> WAIT).

Корректное поведение в этом случае:

- если только a_trig и immediate_allowed=false из-за direction/trade_mode
  — FSM остаётся в OFF, reason = `unknown_candidate_direction` или
  `trade_mode_disallows_direction`;
- если a_trig и mode == wait_st_flip — FSM идёт OFF -> WAIT
  (legacy путь), reason = `entry_mode_wait_st_flip`.

Это разделение явно отражено в trigger_source comb-таблице ниже.


FSM Step Ordering
-----------------
Внутри одного FSM-step проверки выполняются строго в этом порядке:

1. `daily_reset_event[t]` обработка (FSM-уровень):
      state                     <- OFF
      held_pos                  <- 0
      confirmed_legs_since_start <- -1
      _stopping_start            <- -1

   На этом же баре per-bar поля
   (`candidate_age_bars[t]`, `candidate_leg_direction[t]`,
   `candidate_height_pct[t]`) уже **вычислены WP4 close-only pass
   как UNKNOWN** (см. Daily Reset / Per-Bar ZigZag) — FSM-шаг
   их не мутирует, а только наблюдает значения.

2. **immediate entry check** (новый шаг):

   Если все условия:
      state == OFF
      a_trig == true
      immediate_allowed == true

   То:
      state                       = ST_ACTIVE_FREEZE
      held_pos                    = candidate_dir
      confirmed_legs_since_start  = 0
      immediate_candidate_entry_used[t] = 1
      <шаг 3 пропускается на этом баре>

3. legacy `OFF -> WAIT_FIRST_ST_FLIP`:

   legacy_wait_allowed =
       b_trig
       or (
           a_trig
           and candidate_entry_mode == "wait_st_flip"
       )

   Если шаг 2 не сработал и `state == OFF and legacy_wait_allowed`:
      state = WAIT_FIRST_ST_FLIP

   Следствия:
   - A-only + wait_st_flip -> legacy WAIT;
   - A-only + immediate_leg_direction, но direction/trade_mode заблокировали
     immediate -> OFF stays OFF;
   - B-only -> legacy WAIT;
   - A+B + immediate_leg_direction, но direction/trade_mode заблокировали
     immediate -> legacy WAIT за счет B fallback.

4. legacy `WAIT_FIRST_ST_FLIP -> ST_ACTIVE_FREEZE`:

   Если `state == WAIT_FIRST_ST_FLIP and flip_dir != 0
   and _is_first_flip_allowed(flip_dir, trade_mode)`:
      state                      = ST_ACTIVE_FREEZE
      held_pos                   = flip_dir
      confirmed_legs_since_start = 0

5. confirm-bar increment (gated на state_at_bar_start):

   Если `confirm_event[t] == 1 and state_at_bar_start in
   {ST_ACTIVE_FREEZE, ST_ACTIVE_MONITORING}`:
      confirmed_legs_since_start += 1

6. `ST_ACTIVE_FREEZE -> ST_ACTIVE_MONITORING`:

   Если `state == ST_ACTIVE_FREEZE
   and confirmed_legs_since_start >= freeze_confirmed_legs`:
      state = ST_ACTIVE_MONITORING

7. `ST_ACTIVE_MONITORING -> ST_STOPPING`:

   Текущий контракт §15.7 без изменений (gated на state_at_bar_start
   == ST_ACTIVE_MONITORING).

8. ST flip apply held_pos (gated на state_at_bar_start):

   Если `flip_dir != 0 and state in {FREEZE, MONITORING}
   and state_at_bar_start in {FREEZE, MONITORING}`:
      held_pos = _update_held_pos(held_pos, flip_dir, trade_mode)

   Это автоматически обеспечивает правило "same-bar ST flip
   на immediate-entry баре не переписывает held_pos", потому что
   на immediate-entry баре `state_at_bar_start == OFF` и шаг 8
   не срабатывает.

9. `ST_STOPPING -> OFF` нормализация и t+1 position write
   (без изменений относительно baseline).

Same-bar same-step priority — A immediate beats B trigger:

- если `a_trig=true, b_trig=true, immediate_allowed=true` — выигрывает
  immediate (state становится ST_ACTIVE_FREEZE на шаге 2; шаг 3
  пропускается);
- `trigger_source[t]` = `"both"`;
- FSM НЕ уходит в WAIT_FIRST_ST_FLIP.


Trigger Source Combinations
---------------------------
Базовое правило для `trigger_source[t]`:

   trigger_source[t] != "none"
   <=>
   state_at_bar_start == OFF AND state в конце шага != OFF

То есть `trigger_source` сигнализирует **факт** перехода FSM из
`OFF` на этом баре. На баре, где state остался `OFF` (включая
случаи, когда a_trig был блокирован direction/trade_mode и нет
независимого B-trigger), `trigger_source[t] = "none"`. Это
гарантирует, что `ZigZag_Trigger_Events` не получает ложных
строк — лист строится по `trigger_source != "none"` и содержит
только реальные FSM transitions.

Семантика "candidate_threshold" / "confirmed_median" / "both"
отражает **raw conditions**, которые были истинны на момент
перехода:

- `candidate_threshold` — переход вызван только A;
- `confirmed_median`    — переход вызван только B;
- `both`                — на момент перехода a_trig=T и b_trig=T,
                          независимо от того, какой именно из них
                          фактически продвинул FSM (см. строку
                          T-T-immediate-invalid ниже).

Полная таблица для `state_at_bar_start == OFF`:

| # | a_trig | b_trig | mode                     | dir/trade valid | FSM transition                          | trigger_source[t]    | block_reason[t]                                 | used |
|---|--------|--------|--------------------------|-----------------|-----------------------------------------|----------------------|-------------------------------------------------|------|
| 1 | F      | F      | wait_st_flip             | n/a             | OFF stays OFF                           | none                 | entry_mode_wait_st_flip                         | 0    |
| 2 | F      | F      | immediate_leg_direction  | n/a             | OFF stays OFF                           | none                 | height_gate_failed                              | 0    |
| 3 | T      | F      | wait_st_flip             | n/a             | OFF -> WAIT_FIRST_ST_FLIP               | candidate_threshold  | entry_mode_wait_st_flip                         | 0    |
| 4 | T      | F      | immediate_leg_direction  | T               | OFF -> ST_ACTIVE_FREEZE (immediate)     | candidate_threshold  | none                                            | 1    |
| 5 | T      | F      | immediate_leg_direction  | F (dir=0)       | OFF stays OFF                           | none                 | unknown_candidate_direction                     | 0    |
| 6 | T      | F      | immediate_leg_direction  | F (trade-mode)  | OFF stays OFF                           | none                 | trade_mode_disallows_direction                  | 0    |
| 7 | F      | T      | wait_st_flip             | n/a             | OFF -> WAIT_FIRST_ST_FLIP               | confirmed_median     | entry_mode_wait_st_flip                         | 0    |
| 8 | F      | T      | immediate_leg_direction  | n/a             | OFF -> WAIT_FIRST_ST_FLIP (legacy by B) | confirmed_median     | height_gate_failed                              | 0    |
| 9 | T      | T      | wait_st_flip             | n/a             | OFF -> WAIT_FIRST_ST_FLIP (legacy)      | both                 | entry_mode_wait_st_flip                         | 0    |
|10 | T      | T      | immediate_leg_direction  | T               | OFF -> ST_ACTIVE_FREEZE (immediate)     | both                 | none                                            | 1    |
|11 | T      | T      | immediate_leg_direction  | F (dir=0)       | OFF -> WAIT_FIRST_ST_FLIP (B fires)     | both                 | unknown_candidate_direction                     | 0    |
|12 | T      | T      | immediate_leg_direction  | F (trade-mode)  | OFF -> WAIT_FIRST_ST_FLIP (B fires)     | both                 | trade_mode_disallows_direction                  | 0    |

Ключевые наблюдения по таблице:

- Строки 5, 6 — это содержательное **отличие** от legacy: в режиме
  `immediate_leg_direction` blocked-by-direction trigger-A НЕ
  падает в legacy `OFF → WAIT`. Семантика выбранного режима — "только
  immediate ИЛИ ничего"; смешанный fallback на WAIT поломал бы
  бизнес-смысл нового режима.
- Строки 11, 12 — A заблокирован direction/trade_mode, но B
  независимо фиксирует переход OFF → WAIT. trigger_source = "both"
  отражает raw conditions; "both" здесь следует читать как "оба
  raw-условия были истинны на момент перехода; собственно перевод
  FSM выполнен B, потому что A immediate был заблокирован".
- Строки 1, 2, 5, 6 — все случаи "state stays OFF" дают
  `trigger_source = none`. ZigZag_Trigger_Events не получает этих
  строк.
- Строка 8 — необычный случай: a_trig=false (height_gate_failed),
  но b_trig=true в режиме immediate. B-trigger фиксирует переход.
  block_reason берётся по полному priority порядку — height_gate_failed
  стоит выше unknown/trade_mode и считается основной причиной,
  по которой immediate entry не использовался.

Для `state_at_bar_start != OFF` (любая комбинация a_trig/b_trig/mode):

   trigger_source[t] = "none"     (repeated triggers подавлены, §15.3)
   block_reason[t]   = "state_not_off"
   used              = 0

Reason priority и полная whitelist описаны в "Diagnostics".


Execution Timing
----------------
"Сразу" означает без ожидания ST flip, но без нарушения OPEN_TO_OPEN
контракта:

   decision принимается на close бара t
   позиция применяется с open бара t+1

   filtered_positions[t]   == 0 (ещё нет позиции)
   filtered_positions[t+1] == candidate_dir (immediate entry)

Lookahead не появляется. На последнем баре датасета
(`t == n-1`) immediate entry приводит к state == ST_ACTIVE_FREEZE
в diagnostics, но `filtered_positions[t+1]` не существует — позиция
просто не открывается (как любая другая FSM-ветка на последнем баре).


FSM Semantics
-------------
При immediate entry:

   OFF -> ST_ACTIVE_FREEZE

на том же decision-баре, где произошёл candidate trigger.

Дальше логика не меняется:

   ST_ACTIVE_FREEZE
   -> ST_ACTIVE_MONITORING после freeze_confirmed_legs confirm-events
   -> ST_STOPPING при stop condition (§15.7)
   -> OFF на close через opposite ST flip или fail-closed

Same-bar confirmed leg, совпавший с immediate lifecycle start, НЕ
засчитывается в `confirmed_legs_since_start` — это автоматически
обеспечивается gate `state_at_bar_start in {FREEZE, MONITORING}` на
шаге 5 FSM ordering. Поведение единообразно с WAIT -> FREEZE и
не требует отдельного флага.


Same-Bar ST Flip Suppression
----------------------------
На баре immediate entry `state_at_bar_start == OFF`. Если на этом
же баре есть ST flip, он НЕ переписывает `held_pos`, выставленный
по `candidate_dir`.

Это правило — естественное следствие gate на шаге 8 FSM ordering
(`state_at_bar_start in {FREEZE, MONITORING}`), без отдельного кода.

На последующих барах FREEZE/MONITORING ST flip обрабатывается обычной
логикой `_update_held_pos`. Например, immediate long entry на баре `t`,
короткий ST flip на баре `t+1` -> в trade_mode `both/revers` `held_pos`
становится `-1`.


Freeze Confirmed Legs == 0
--------------------------
Если `freeze_confirmed_legs == 0` и immediate entry произошёл на `t`,
разрешён за один decision step переход:

   OFF -> ST_ACTIVE_FREEZE -> ST_ACTIVE_MONITORING

(сначала шаг 2 даёт OFF -> FREEZE и `confirmed_legs_since_start = 0`,
затем шаг 6 проверяет `0 >= 0` и даёт FREEZE -> MONITORING).

Шаг 7 (MONITORING -> STOPPING) на этом баре НЕ срабатывает, потому
что он gated на `state_at_bar_start == MONITORING`, а
state_at_bar_start был OFF.

Поведение закрепляется acceptance #I8.


Disabled / Unavailable Representations
--------------------------------------
Единая таблица для всех новых полей:

| поле                                     | dataclass type   | disabled / N/A | filter_diagnostics dtype | filter_diagnostics value |
|------------------------------------------|------------------|----------------|--------------------------|--------------------------|
| candidate_duration_gate_enabled          | bool             | False          | int8                     | 0                        |
| candidate_duration_max_bars              | Optional[int]    | None           | int64                    | -1                       |
| candidate_duration_gate_passed           | n/a              | n/a            | int8                     | 1 (всегда pass когда gate disabled) |
| candidate_age_bars                       | n/a              | UNKNOWN        | int64                    | -1                       |
| candidate_leg_direction                  | n/a              | UNKNOWN        | int8                     | 0                        |
| candidate_entry_mode                     | str              | "wait_st_flip" (default) | object           | "wait_st_flip"           |
| immediate_candidate_entry_used           | n/a              | not used       | int8                     | 0                        |
| immediate_candidate_entry_block_reason   | n/a              | n/a            | object                   | см. priority ниже        |

Когда filter `enabled == false` (top-level disable):
- ни один из новых ключей в filter_diagnostics НЕ создаётся
  (consistent с текущим disabled-path bit-identity §10.6.6 #1);
- XLSX-колонки НЕ появляются.

Когда filter `enabled == true`, но новых блоков нет (legacy config):
- все ключи присутствуют со значениями по умолчанию из таблицы выше;
- XLSX-колонки присутствуют со значениями по умолчанию;
- bit-identity на уровне `filtered_positions` и старых diagnostic
  колонок гарантирована (acceptance #B1).


Diagnostics
-----------
Добавить в `filter_diagnostics` следующие per-bar ключи:

   candidate_age_bars                      int64
   candidate_leg_direction                 int8
   candidate_duration_max_bars             int64, -1 when disabled
   candidate_duration_gate_passed          int8
   candidate_entry_mode                    object/string
   immediate_candidate_entry_used          int8
   immediate_candidate_entry_block_reason  object/string

И top-level scalar (broadcast в per-bar для XLSX):

   candidate_duration_gate_enabled         int8

Whitelist для `immediate_candidate_entry_block_reason`:

   none                              immediate entry использован (used == 1)
   filter_off                        defensive only — нормальный disabled-
                                     path не создаёт ключи (см. ниже)
   daily_reset                       reset bar
   state_not_off                     FSM в WAIT/FREEZE/MONITORING/STOPPING
   entry_mode_wait_st_flip           mode == wait_st_flip
                                     (immediate entry концептуально
                                     не задумывается)
   height_gate_failed                mode == immediate, height_ok == false
   duration_gate_failed              mode == immediate, height_ok == true,
                                     duration_ok == false
   unknown_candidate_direction       mode == immediate, a_trig == true,
                                     candidate_dir == 0
   trade_mode_disallows_direction    mode == immediate, a_trig == true,
                                     candidate_dir != 0, но trade_mode
                                     запрещает это направление

Priority (highest first; ровно одно значение per bar):

   1. filter_off                       (defensive — обычно недостижим)
   2. daily_reset
   3. state_not_off
   4. entry_mode_wait_st_flip          ← поднят выше height/duration:
                                         в wait_st_flip mode immediate
                                         entry концептуально не пытается
                                         сработать, поэтому height/
                                         duration не имеют значения
   5. height_gate_failed
   6. duration_gate_failed
   7. unknown_candidate_direction
   8. trade_mode_disallows_direction
   9. none                             (используется ИСКЛЮЧИТЕЛЬНО при
                                         used == 1)

Алгоритм назначения причины (псевдокод):

   if filter_top_level_disabled:               # defensive
       reason = "filter_off"
   elif daily_reset_event[t]:
       reason = "daily_reset"
   elif state_at_bar_start != OFF:
       reason = "state_not_off"
   elif candidate_entry_mode == "wait_st_flip":
       reason = "entry_mode_wait_st_flip"
   # с этой точки candidate_entry_mode == "immediate_leg_direction"
   elif not height_ok:
       reason = "height_gate_failed"
   elif not duration_ok:
       reason = "duration_gate_failed"
   elif candidate_dir == 0:
       reason = "unknown_candidate_direction"
   elif not trade_mode_allows(candidate_dir):
       reason = "trade_mode_disallows_direction"
   else:
       reason = "none"
       used   = 1

Это гарантирует, что КАЖДЫЙ бар (где filter включён) получает ровно
одну причину; "none" зарезервирован исключительно для случая, когда
immediate entry действительно сработал.

Строгий инвариант:

   immediate_candidate_entry_used[t] == 1
   <=>
   immediate_candidate_entry_block_reason[t] == "none"

Заметки:

- `filter_off` фигурирует в whitelist для defensive-кодовых путей,
  где `immediate_candidate_entry_block_reason` строится несмотря на
  top-level disabled. В нормальном disabled-path
  (`trade_filter.enabled == false`) ни один из новых diagnostic-
  ключей вообще не создаётся (см. "Disabled / Unavailable
  Representations") — поэтому `filter_off` практически недостижим
  и присутствует только как whitelist-литерал.

- `entry_mode_wait_st_flip` НЕ означает ошибку. Это явное и
  единственное объяснение, почему immediate entry не использовался,
  на каждом баре в режиме `wait_st_flip`. Аналитик видит в XLSX
  одну и ту же причину на 100% таких баров — это норма.

- На большинстве реальных баров `height_gate_failed` будет основной
  причиной (height не дотянул до threshold). Это шум для счётчика
  блокировок — поэтому `Immediate Candidate Entries Blocked Count`
  в `filters_summary` исключает `height_gate_failed` (см. "XLSX").

Сосуществование с существующим `filter_block_reason`:

- `filter_block_reason` (существующий массив) описывает блокировки
  FSM-уровня (`local_median_unavailable`, `stopping_mode_no_new_entries`,
  `filter_off`, `trade_mode_disallowed_flip`, `daily_reset`);
- `immediate_candidate_entry_block_reason` (новый массив) описывает
  только причины, по которым immediate entry на этом баре не
  использовался;
- общие литералы (`filter_off`, `daily_reset`) намеренно совпадают
  по написанию (но создаются в разных условиях);
- разные литералы (`trade_mode_disallowed_flip` в filter_block_reason
  vs `trade_mode_disallows_direction` в immediate_*_block_reason)
  отражают разный контекст (flip vs candidate direction) и НЕ
  объединяются.


ZigZag_Trigger_Events Semantics
-------------------------------
Лист `ZigZag_Trigger_Events` остаётся листом реальных trigger
events, переведших FSM из OFF (через WAIT или через immediate entry).

Лист НЕ обязан включать blocked-by-duration / blocked-by-direction
бары — они доступны через `FilterDiagnostics_100` по diagnostic
arrays:

   height_ok == true
   duration_gate_passed == false
   immediate_candidate_entry_block_reason == "duration_gate_failed"

(или соответствующая пара для других reason).

Если позже понадобится отдельный audit-лист для всех blocked-by-gate
событий — это отдельный feature request, не часть v2.


XLSX
----
FilterDiagnostics_100 — вставить семь колонок ПОСЛЕ
`Candidate Trigger Threshold` и ПЕРЕД `Local Median N`:

   Candidate Age Bars
   Candidate Leg Direction
   Candidate Duration Max Bars
   Candidate Duration Gate Passed
   Candidate Entry Mode
   Immediate Candidate Entry Used
   Immediate Candidate Entry Block Reason

Финальный column order:

   ... Candidate Height %, Candidate Trigger Threshold,
   Candidate Age Bars, Candidate Leg Direction,
   Candidate Duration Max Bars, Candidate Duration Gate Passed,
   Candidate Entry Mode, Immediate Candidate Entry Used,
   Immediate Candidate Entry Block Reason,
   Local Median N, Local Median Available, ...

ZigZag_Trigger_Events — добавить шесть колонок после `Candidate Height %`,
**с теми же display-именами, что в FilterDiagnostics_100**:

   Candidate Age Bars
   Candidate Leg Direction
   Candidate Duration Max Bars
   Candidate Duration Gate Passed
   Candidate Entry Mode
   Immediate Candidate Entry Used

(колонка `Immediate Candidate Entry Block Reason` в Trigger_Events
не требуется — лист содержит только реальные events, где
reason == "none").

filters_summary, params section (добавить три строки):

   Candidate Duration Gate Enabled
   Candidate Duration Max Bars
   Candidate Entry Mode

filters_summary, period section (добавить две колонки):

   Immediate Candidate Entries Count
       Число баров на периоде, где `immediate_candidate_entry_used == 1`.

   Immediate Candidate Entries Blocked Count
       Число баров, где `immediate_candidate_entry_block_reason` ∈
       одной из СОДЕРЖАТЕЛЬНЫХ причин блокировки:

           {duration_gate_failed,
            unknown_candidate_direction,
            trade_mode_disallows_direction}

       Это все случаи, когда candidate-нога имела достаточный height
       (height_gate_failed уже не сработал), state был OFF, фильтр
       был включён, и режим был `immediate_leg_direction` —
       но конкретный gate (duration / direction / trade_mode)
       заблокировал вход.

       Намеренно ИСКЛЮЧЕНЫ как noise:

       - `none`                         — это успешные entries, считаются
                                          в Count, не в Blocked.
       - `filter_off`                   — disabled-path не создаёт ключи.
       - `daily_reset`                  — событие сессии, не "блок".
       - `state_not_off`                — повторные триггеры в активном
                                          lifecycle не дают полезной
                                          бизнес-информации (репортить
                                          как "blocked" 90% lifecycle
                                          бессмысленно).
       - `entry_mode_wait_st_flip`      — immediate entry в wait-режиме
                                          концептуально не пытается
                                          сработать.
       - `height_gate_failed`           — срабатывает на большинстве
                                          баров; включение раздуло бы
                                          Blocked Count до ~99% всех
                                          баров и сделало бы поле
                                          бесполезным для аналитики.

       Содержательная интерпретация поля:
       "сколько раз candidate-нога была реально готова к immediate
       entry (height OK, state OFF, mode immediate), но какой-либо
       gate (duration cap, неизвестное направление, trade_mode)
       помешал".


Backward Compatibility
----------------------
1. Если новых блоков (`candidate_duration_gate`, `candidate_entry`)
   нет в YAML, поведение фильтра идентично текущему:
   - duration_gate тождественно True;
   - mode == wait_st_flip;
   - filtered_positions и существующие diagnostic-колонки
     bit-identical с baseline (при том же random seed и тех же
     prices).

2. При `candidate_duration_gate.enabled: false` duration gate
   не влияет на trigger-A.

3. При `candidate_entry.mode: wait_st_flip` FSM идёт старым путём
   `OFF -> WAIT_FIRST_ST_FLIP -> ST_ACTIVE_FREEZE`.

4. Новый режим не меняет trigger-B, кроме явно описанного same-bar
   A+B priority при `immediate_allowed == true`.

5. XLSX baseline для disabled-path (`trade_filter.enabled == false`)
   не получает новые sheets или колонки. Existing sheets и колонки
   bit-identical с pre-v2 baseline.

6. XLSX baseline для enabled-path с legacy-config (без новых блоков)
   получает новые колонки со значениями по умолчанию из таблицы
   "Disabled / Unavailable Representations". Это не считается
   нарушением bit-identity — bit-identity для enabled-path
   гарантируется только на уровне `filtered_positions` и старых
   diagnostic-ключей.


Acceptance Criteria
-------------------
Группировка по областям. Все тесты — unit-уровня; реализация
в `wf_grid/tests/test_zigzag_volatility_breakout_v2.py` (новый
файл) и точечные расширения существующих WP3/WP4/WP5/WP9/WP10
тестов.

A. Config validation (Schema + Strict)
   A1. YAML с `candidate_duration_gate.{enabled, max_bars}` и
       `candidate_entry.{mode}` принимается.
   A2. Неизвестные ключи в этих блоках (например,
       `min_population_size`, `inclusive_bars`, `max_bars_source`,
       `quantile`) отклоняются strict-schema-валидатором как
       `unknown config key:`.
   A3. `candidate_entry.mode` принимает только литералы
       `wait_st_flip`, `immediate_leg_direction`. Любое другое
       значение — ConfigError.
   A4. `enabled: true` без `max_bars` — ConfigError
       `candidate_duration_gate.max_bars is required when enabled
       is true`.
   A5. `enabled: false` + явный `max_bars: 10` в YAML — ConfigError
       (через raw_user_keys, не через значение поля).
   A6. `enabled: true, max_bars: 0` — ConfigError.
   A7. `enabled: true, max_bars: -1` — ConfigError.
   A8. `enabled: true, max_bars: true` (bool) — ConfigError.
   A9. `enabled: true, max_bars: 1.5` (float) — ConfigError.
   A10. `enabled: "true"` (string) — ConfigError.
   A11. `enabled: false, max_bars: null` (key explicitly present
        with null value) — ConfigError. Проверка через raw_user_keys:
        даже `null`-значение считается "key present in YAML".
   A12. `enabled: false`, ключ `max_bars` ПОЛНОСТЬЮ отсутствует
        в YAML (даже не упомянут) — валидно, materialization
        пропускается, `candidate_duration_max_bars == None`.

B. Backward compatibility / disabled path
   B1. При полном отсутствии блоков `candidate_duration_gate` и
       `candidate_entry` поведение фильтра bit-identical с pre-v2
       baseline (filtered_positions + старые diagnostic-ключи).
   B2. При `trade_filter.enabled: false` ни один из новых
       diagnostic-ключей не создаётся.
   B3. При `candidate_duration_gate.enabled: false` (или блок
       отсутствует) `candidate_duration_gate_passed[t] == 1` для
       всех `t`, gate всегда pass.
   B4. При `candidate_entry.mode: wait_st_flip` FSM строго проходит
       OFF -> WAIT_FIRST_ST_FLIP -> ST_ACTIVE_FREEZE
       (никаких прямых OFF -> FREEZE).

C. Per-bar candidate state (WP4 расширение)
   C1. `candidate_age_bars[t] = -1` на (a) pre-bootstrap, (b) reset-баре,
       (c) pathological close, (d) UNKNOWN direction.
   C2. На confirm-баре `candidate_age_bars[t] >= 2`
       (новая нога стартует от end_bar предыдущей,
       end_bar < t).
   C3. `candidate_leg_direction[t]` на confirm-баре равно
       противоположному только что подтверждённой ноги.
   C4. `candidate_leg_direction[t]` НИКОГДА не равно `held_pos`
       в активных FSM-состояниях, если ZigZag candidate ушёл
       в противоположную сторону.

D. Global stats materialization
   D1. При `enabled: false` `candidate_duration_max_bars == None`
       (dataclass) и == -1 (filter_diagnostics broadcast).
   D2. При `enabled: true, max_bars: 10`
       `candidate_duration_max_bars == 10` и
       `candidate_duration_gate_enabled == True`.
   D3. `candidate_duration_max_bars` материализуется один раз
       на полном датасете и не зависит от per-WF train/OOS slicing
       (parity с `candidate_trigger_threshold` и `global_median`).
   D4. `metadata.config_snapshot.candidate_duration_gate` содержит
       raw user-supplied values (`{"enabled": true, "max_bars": 10}`).

E. Duration gate runtime
   E1. При `enabled: false` `duration_ok` тождественно `True` для
       всех баров (даже когда candidate_age_bars = -1).
   E2. При `enabled: true, max_bars: 10`:
        - `candidate_age_bars == 10` -> gate PASS;
        - `candidate_age_bars == 11` -> gate FAIL,
          reason = `duration_gate_failed`;
        - `candidate_age_bars == 0` или `-1` -> gate FAIL.
   E3. trigger-A с `candidate_age_bars > max_bars` НЕ переводит
       FSM из OFF (ни в WAIT, ни в FREEZE).
   E4. duration gate применяется ДО immediate entry check —
       blocked candidate не открывает позицию даже при
       `mode == immediate_leg_direction`.

F. FSM step ordering
   F1. На баре с `state==OFF, a_trig=true, immediate_allowed=true`,
       одновременным ST flip `flip_dir == -candidate_dir`:
        - `state` после step == ST_ACTIVE_FREEZE;
        - `held_pos == candidate_dir` (НЕ `flip_dir`);
        - `state_at_bar_start == OFF`;
        - `trigger_source == "candidate_threshold"`.
   F2. На баре с `state==OFF, a_trig=true, immediate_allowed=true,
       confirm_event[t]==1`:
        - `confirmed_legs_since_start` после step == 0 (НЕ 1).
   F3. На баре с `state==OFF, a_trig=true, b_trig=true,
       immediate_allowed=true`:
        - state -> ST_ACTIVE_FREEZE (immediate);
        - trigger_source == "both";
        - FSM НЕ уходит в WAIT_FIRST_ST_FLIP;
        - block_reason == "none", used == 1.
   F4. На баре с `state==OFF, a_trig=true, b_trig=true,
       mode=wait_st_flip`:
        - state -> WAIT_FIRST_ST_FLIP (legacy);
        - trigger_source == "both";
        - block_reason == "entry_mode_wait_st_flip", used == 0;
        - immediate entry НЕ происходит.
   F5. На баре с `state==OFF, a_trig=true, b_trig=false,
       mode=wait_st_flip`:
        - state -> WAIT_FIRST_ST_FLIP;
        - trigger_source == "candidate_threshold";
        - block_reason == "entry_mode_wait_st_flip", used == 0.
   F6. На баре с `state==OFF, a_trig=true, b_trig=false,
       mode=immediate_leg_direction`, заблокированным
       direction/trade_mode (i.e., immediate_allowed=false из-за
       candidate_dir==0 или trade_mode):
        - state остаётся OFF (НЕ переходит в WAIT);
        - filtered_positions[t+1] == 0;
        - trigger_source == "none";
        - block_reason == "unknown_candidate_direction"
          ИЛИ "trade_mode_disallows_direction" (в зависимости от
          конкретной причины), used == 0;
        - на ZigZag_Trigger_Events листе этот бар НЕ появляется.
   F7. На баре с `state==OFF, a_trig=true, b_trig=true,
       mode=immediate_leg_direction`, заблокированным
       direction/trade_mode:
        - state -> WAIT_FIRST_ST_FLIP (B fires legacy, A silenced);
        - trigger_source == "both" (raw conditions: оба истинны);
        - block_reason == "unknown_candidate_direction" или
          "trade_mode_disallows_direction", used == 0;
        - на ZigZag_Trigger_Events листе бар появляется
          (real FSM transition).
   F8. На баре с `state==OFF, a_trig=false, b_trig=false`
       (любой mode):
        - state остаётся OFF;
        - trigger_source == "none";
        - block_reason ∈ {entry_mode_wait_st_flip,
          height_gate_failed} в зависимости от mode;
        - used == 0.
   F9. На баре с `state ∈ {WAIT, FREEZE, MONITORING, STOPPING}`
       (любой a_trig/b_trig/mode):
        - trigger_source == "none" (suppressed §15.3);
        - block_reason == "state_not_off", used == 0;
        - повторный trigger НЕ стартует новый lifecycle.

G. trade_mode interaction
   G1. `mode=immediate_leg_direction, candidate_dir=+1, trade_mode=long`
       -> immediate long opens, used == 1.
   G2. `mode=immediate_leg_direction, candidate_dir=-1, trade_mode=short`
       -> immediate short opens, used == 1.
   G3. `mode=immediate_leg_direction, candidate_dir=+1, trade_mode=short,
       a_trig=true, b_trig=false`:
        - state остаётся OFF (НЕ переходит в WAIT);
        - reason = `trade_mode_disallows_direction`;
        - trigger_source == "none";
        - used == 0.
   G4. `mode=immediate_leg_direction, candidate_dir=0,
       a_trig=true, b_trig=false`:
        - state остаётся OFF;
        - reason = `unknown_candidate_direction`;
        - trigger_source == "none";
        - used == 0.
   G5. trade_mode in {"both", "revers"} и любой ненулевой
       candidate_dir -> immediate entry открывается.
   G6. `mode=immediate_leg_direction, candidate_dir=+1, trade_mode=short,
       a_trig=true, b_trig=true`:
        - state -> WAIT_FIRST_ST_FLIP (B fires legacy);
        - trigger_source == "both";
        - reason = `trade_mode_disallows_direction`;
        - used == 0;
        - бар появляется в ZigZag_Trigger_Events (real transition).

H. daily_reset interaction
   H1. На reset-баре immediate entry невозможен независимо
       от `candidate_height_pct[t]`. reason имеет приоритет
       `daily_reset`.
   H2. После reset bootstrap начинается заново; первый
       a_trig возможен только когда новый candidate набрал
       нужный height.
   H3. In-progress candidate-нога перед reset-баром стирается;
       её height в новой сессии не учитывается.

I. immediate entry lifecycle
   I1. Open-to-open timing: `filtered_positions[t]==0`,
       `filtered_positions[t+1]==candidate_dir`.
   I2. На последнем баре датасета (`t == n-1`) immediate entry
       не открывает позицию; state в diagnostics при этом
       ST_ACTIVE_FREEZE.
   I3. Same-bar opposite ST flip на immediate-entry-баре НЕ
       переписывает held_pos.
   I4. Immediate entry на confirm-баре открывает позицию по
       направлению **новой** candidate-ноги (противоположной
       только что подтверждённой).
   I5. Повторный trigger-A на t+1 (state == FREEZE) не стартует
       новый lifecycle, reason = `state_not_off`.
   I6. После immediate long entry на `t`, на `t+1` short ST flip
       в trade_mode `both/revers` -> `held_pos` становится `-1`.
   I7. Last bar reset (`daily_reset_event[n-1]==true`) +
       immediate-entry на `n-1` -> reset срабатывает на шаге 1
       FSM ordering, immediate НЕ происходит, state == OFF.
   I8. `freeze_confirmed_legs == 0` + immediate entry на `t`
       -> в один decision step state переходит OFF -> FREEZE
       -> MONITORING. Шаг 7 (MONITORING -> STOPPING) НЕ
       срабатывает (gate state_at_bar_start).

J. Diagnostics invariants
   J1. Strict invariant (двусторонний):
       `immediate_candidate_entry_used[t] == 1`
       <=>
       `immediate_candidate_entry_block_reason[t] == "none"`.
       Проверяется на каждом баре включённого фильтра — ни одного
       исключения.
   J2. Reason priority соблюдается:
        - На reset-баре с mode==wait_st_flip reason == `daily_reset`
          (не `entry_mode_wait_st_flip`).
        - На баре с state==WAIT и mode==immediate_leg_direction
          reason == `state_not_off` (не `height_gate_failed` и не
          `entry_mode_wait_st_flip`).
        - На баре с mode==wait_st_flip и низким height
          reason == `entry_mode_wait_st_flip` (не
          `height_gate_failed`) — wait-режим перекрывает gate-
          специфичные reasons.
   J3. На pathological-баре `candidate_age_bars[t] == -1`,
       `candidate_leg_direction[t] == 0`, a_trig == false,
       FSM-state не мутирует, reason == `height_gate_failed`
       (NaN candidate_height_pct → height_ok=false).
   J4. trigger_source[t] != "none" ВСЕГДА совпадает с фактическим
       FSM-переходом OFF → не-OFF на этом баре. Никогда не
       появляется на баре, где state остался OFF (включая F6 case).
       Проверяется через cross-check: для каждого бара с
       `trigger_source != "none"` верно `state_at_bar_start == OFF
       AND state_after_step != OFF`.

K. XLSX export
   K1. FilterDiagnostics_100: семь новых колонок присутствуют
       в указанном порядке после `Candidate Trigger Threshold`
       и перед `Local Median N`.
   K2. ZigZag_Trigger_Events: шесть новых колонок присутствуют
       с display-именами, идентичными соответствующим колонкам
       в FilterDiagnostics_100. Полный набор колонок в Trigger_Events
       — подмножество FilterDiagnostics_100 (без
       `Immediate Candidate Entry Block Reason`); все совпадающие
       колонки имеют одинаковое display-name (правило
       "same display names for shared columns").
   K3. filters_summary params section: три новых строки
       (Gate Enabled / Max Bars / Entry Mode).
   K4. filters_summary period section: две новых колонки
       (Immediate Entries Count / Blocked Count). Формулы:
        - `Count = sum(immediate_candidate_entry_used == 1)`;
        - `Blocked Count = sum(reason ∈ {duration_gate_failed,
          unknown_candidate_direction, trade_mode_disallows_direction})`.
       `height_gate_failed` явно НЕ включается в Blocked Count.
   K5. При `trade_filter.enabled: false` ни одна из новых
       колонок не появляется (disabled-path bit-identity).
   K6. При filter enabled + legacy config (без новых блоков)
       все новые колонки присутствуют со значениями по умолчанию
       из таблицы "Disabled / Unavailable Representations".
   K7. На датасете с большим количеством `height_gate_failed`-баров
       (~99% датасета) `Immediate Candidate Entries Blocked Count` в
       filters_summary остаётся пропорционально малым, считая
       только содержательные блокировки. Регрессионный тест: на
       синтетике из 1000 баров, где только 5 баров имели
       height_ok=true и из них 2 заблокированы duration, 1
       заблокирован trade_mode — Blocked Count == 3 (не 998).
   K8. Блокировки в `state_not_off` режиме НЕ попадают в Blocked
       Count: на датасете с 100 барами в FREEZE/MONITORING после
       одного immediate entry, Blocked Count для этого периода
       равен только числу содержательных блокировок ВНЕ активного
       lifecycle (state_not_off-баров там не включены).

L. Anti-drift
   L1. Существующий `filter_block_reason` whitelist не меняется
       (не появляются новые литералы из immediate-entry домена).
   L2. Существующая `_TRIGGER_SOURCE_*` константа не меняется
       (литералы `none / candidate_threshold / confirmed_median /
       both` сохраняются).
   L3. FSM state codes (ZigZagFSMState IntEnum) не меняются
       (immediate entry использует существующий ST_ACTIVE_FREEZE).


Out of Scope (явно)
-------------------
- median_above_threshold или любая другая population-based
  материализация duration cap;
- per-train или per-OOS пересчёт `candidate_duration_max_bars`;
- audit-лист всех blocked-by-duration баров (отдельный feature
  request, не часть v2);
- расширение whitelist-ов trade_mode (long / short / both / revers
  остаются единственными значениями);
- любое изменение OPEN_TO_OPEN execution model или trade extraction.
