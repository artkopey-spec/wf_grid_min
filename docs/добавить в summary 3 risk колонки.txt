Нужно добавить в **summary 3.0** три risk-метрики, связанные с убыточными сделками и сериями, чтобы они были доступны и на уровне сегментов `S1..SN`, и на агрегированном уровне по кандидату.

Метрики для добавления:

1. `max_losing_trade`
2. `worst_streak_pnl`
3. `max_losing_streak`

Цель:

* усилить summary 3.0 метриками хвостового риска и серийных убытков
* сохранить совместимость с текущей логикой summary, где уже есть агрегаты по окнам и wide-сегменты `S1..SN`
* добавить эти метрики без поломки текущих колонок и порядка блоков

Что нужно сделать:

### 1) Источник метрик

Реализовать расчёт трёх метрик на уровне **каждого OOS/WF-сегмента**:

* `max_losing_trade`

  * худшая отдельная сделка в сегменте
  * брать минимум по `net_pnl_pct` среди сделок сегмента
  * если убыточных сделок нет:

    * либо `0.0`, либо `NaN` — выбрать один контракт и применять везде одинаково
  * предпочтительно явно задокументировать контракт

* `max_losing_streak`

  * максимальная длина подряд идущей серии убыточных сделок в сегменте
  * считать по последовательности сделок сегмента, где сделка убыточна при `net_pnl_pct < 0`

* `worst_streak_pnl`

  * худший суммарный PnL среди всех подряд идущих убыточных серий в сегменте
  * пример:

    * сделки: `-1, -2, +1, -0.5, -3`
    * убыточные серии: `(-1, -2)` и `(-0.5, -3)`
    * суммы: `-3` и `-3.5`
    * `worst_streak_pnl = -3.5`

Важно:

* расчёт должен быть именно по **OOS trades** конкретного сегмента
* опираться на уже собранные сделки / trade rows, а не пытаться выводить эти метрики из bar-level returns
* если для сегмента нет сделок, поведение должно быть детерминированным и единообразным

### 2) Добавление в step-level / segment-level данные

Нужно, чтобы эти поля появились на уровне шага / сегмента и затем попали в wide summary:

Для каждого сегмента `Sx` добавить:

* `Sx_max_losing_trade`
* `Sx_max_losing_streak`
* `Sx_worst_streak_pnl`

Они должны собираться так же, как сейчас summary строит остальные segment-колонки `S1..SN` из step-level данных. Сейчас summary wide строится из `step_oos_long` и segment columns `S1_*..SN_*`, а агрегаты считаются по ok-окнам. 

### 3) Добавление в агрегаты кандидата

На candidate-level добавить агрегаты для этих трёх метрик по **ok OOS/WF окнам**, по той же схеме, как сейчас считаются агрегаты для `sum_pnl_pct`, `max_drawdown`, `profit_factor`, `avg_trade` и др.:

Для каждой из трёх новых метрик нужны:

* `_Mean`
* `_Std`
* `_Min`
* `_Max`
* `_Median`

То есть добавить:

Для `max_losing_trade`:

* `max_losing_trade_Mean`
* `max_losing_trade_Std`
* `max_losing_trade_Min`
* `max_losing_trade_Max`
* `max_losing_trade_Median`

Для `max_losing_streak`:

* `max_losing_streak_Mean`
* `max_losing_streak_Std`
* `max_losing_streak_Min`
* `max_losing_streak_Max`
* `max_losing_streak_Median`

Для `worst_streak_pnl`:

* `worst_streak_pnl_Mean`
* `worst_streak_pnl_Std`
* `worst_streak_pnl_Min`
* `worst_streak_pnl_Max`
* `worst_streak_pnl_Median`

Агрегация должна следовать текущему контракту:

* агрегировать только `step_status == "ok"`
* `Std` при `<2` валидных значениях → `NaN`
* all-NaN → все агрегаты `NaN`

Добавить новые поля в `summary 3.0` так, чтобы они были видны и удобны для анализа.
