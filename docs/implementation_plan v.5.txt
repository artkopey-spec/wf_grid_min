Purpose

implementation-ready plan from v.3 with the three counter-audit fixes
from v.4:

  C3-1  execution_mode must be set to "parallel" as soon as the

  C3-2  ZigZagGlobalStats immutability must be verified inside the

        index name and index frequency. It must preserve, not coerce.

Read this file instead of applying v.4 as a patch on top of v.3.


Goal
----
Speed up the most expensive stage of the pipeline: walk-forward
execution across grid points (OOS + Train), while preserving bit-exact
outputs versus sequential mode.

For the current config.yaml (~273 grid points x 7 WF steps = 3822
backtest calls), grid_point-level parallelism is the correct boundary.


Core Design
-----------
Parallelize by grid_point. Each worker task computes BOTH OOS and Train
for one grid_point.

One task:

    grid_point -> (grid_point_id, oos_step_results, train_step_results)

Worker initializer receives immutable shared context once per process:

- full OHLC data packed as numpy arrays + DatetimeIndex metadata;
- wf_slices;
- resolved GridConfig, including resolved_periods_per_year;
- prepend_bars;
- zigzag_global_stats, frozen inside the worker;
- project_root and donor_path for Windows spawn import safety.

Result collection uses concurrent.futures.as_completed. Completion
order does not affect final outputs because downstream collectors sort
canonically.

No production assert is allowed. Use explicit RuntimeError checks so
wf_grid/tests/test_no_production_asserts.py stays green.

2.2 pack_data contract:

    def pack_data(full_data: pd.DataFrame) -> dict:
        """Convert OHLC DataFrame to picklable parts.

        ROUND-TRIP CONTRACT:
        unpack_data(pack_data(df)) is observationally equal to df.copy()
        for any df that satisfies:
          - df.columns is a superset of {open, high, low, close};
          - df has a DatetimeIndex, tz-aware or tz-naive;
          - OHLC dtypes are numeric dtypes accepted by
            validate_ohlc_data.

        The contract is preserve, not normalise. Do not coerce OHLC
        columns to float64.

        Return:
          {
              "open": np.ndarray,
              "high": np.ndarray,
              "low": np.ndarray,
              "close": np.ndarray,
              "column_dtypes": dict[str, str],
              "index_int64_ns_utc": np.ndarray,  # dtype int64
              "index_tz": Optional[str],
              "index_name": Optional[str],
              "index_freq": Optional[str],
          }
        """

For tz-aware indexes, store UTC nanoseconds. For tz-naive indexes,
store the index asi8 values without localize/convert. This avoids
silent wall-clock offset drift.

3.4 Add parallel branch with enter-then-mark ordering:

    def _execute_wf_grid(
        grid_points,
        wf_slices,
        data,
        config,
        prepend_bars,
        zigzag_global_stats,
        *,
        _result: PipelineResult,
    ):
        n_workers = _resolve_max_workers(config, grid_points)

        if (
            not config.execution.parallel_enabled
            or n_workers == 1
            or len(grid_points) == 0
        ):
            _result.execution_mode = "sequential"
            return _execute_wf_grid_sequential(
                grid_points, wf_slices, data, config,
                prepend_bars, zigzag_global_stats,
            )

        # Enter-then-mark: once this line runs, a later failure must
        # still report that the parallel branch was attempted.
        _result.execution_mode = "parallel"

        try:
            return _execute_wf_grid_parallel(
                grid_points, wf_slices, data, config,
                prepend_bars, zigzag_global_stats, n_workers,
            )
        except _PARALLEL_INFRA_FAILURES as exc:
            if not config.execution.fallback_to_sequential:
                raise
            logger.warning(
                "Parallel execution failed (%s: %s); falling back "
                "to sequential mode for the entire wf_grid_execution "
                "stage.",
                type(exc).__name__,
                exc,
            )
            out = _execute_wf_grid_sequential(
                grid_points, wf_slices, data, config,
                prepend_bars, zigzag_global_stats,
            )
            _result.execution_mode = "parallel_then_fallback"
            return out

3.5 execution_mode contract:

    "sequential"
      - parallel_enabled=False, or
      - max_workers resolves to 1, or
      - len(grid_points) == 0,
      and the parallel branch was not entered.

    "parallel"
      - the parallel branch was entered;
      - applies whether the branch succeeded or failed;
      - if fallback was enabled but the fallback rerun also failed,
        mode remains "parallel".

    "parallel_then_fallback"
      - parallel branch raised an infrastructure failure;
      - fallback_to_sequential=True;
      - the sequential rerun completed successfully.

3.7 _resolve_max_workers:

    def _resolve_max_workers(config, grid_points) -> int:
        configured = config.execution.max_workers
        cpu = configured if configured is not None else (os.cpu_count() or 1)
        if len(grid_points) == 0:
            return 1
        return max(1, min(cpu, len(grid_points), 32))

3.8 _execute_wf_grid_parallel:

    def _execute_wf_grid_parallel(
        grid_points,
        wf_slices,
        data,
        config,
        prepend_bars,
        zigzag_global_stats,
        n_workers,
    ):
        from concurrent.futures import ProcessPoolExecutor, as_completed
        import multiprocessing
        from wf_grid.wf._mp_helpers import (
            pack_data, _init_worker, _run_grid_point_both,
        )

        ctx = multiprocessing.get_context("spawn")
        data_dict = pack_data(data)
        project_root = str(_PROJECT_ROOT)
        donor_path = str(_DONOR_ROOT)

        all_oos = {}
        all_train = {}

        with ProcessPoolExecutor(
            max_workers=n_workers,
            mp_context=ctx,
            initializer=_init_worker,
            initargs=(
                project_root,
                donor_path,
                data_dict,
                wf_slices,
                config,
                prepend_bars,
                zigzag_global_stats,
            ),
        ) as ex:
            futures = {
                ex.submit(_run_grid_point_both, gp): gp.grid_point_id
                for gp in grid_points
            }
            done = 0
            for fut in as_completed(futures):
                gp_id = futures[fut]
                rid, oos_results, train_results = fut.result()
                if rid != gp_id:
                    raise RuntimeError(
                        f"Worker returned grid_point_id {rid!r}, "
                        f"expected {gp_id!r}"
                    )
                all_oos[gp_id] = oos_results
                all_train[gp_id] = train_results
                done += 1
                if done % max(1, len(grid_points) // 10) == 0:
                    logger.info(
                        "  Parallel WF: %d/%d grid points done",
                        done,
                        len(grid_points),
                    )

        expected_ids = {gp.grid_point_id for gp in grid_points}
        missing_oos = expected_ids - set(all_oos)
        missing_train = expected_ids - set(all_train)
        if missing_oos or missing_train:
            raise RuntimeError(
                f"Parallel WF execution incomplete: "
                f"missing OOS {sorted(missing_oos)}, "
                f"missing Train {sorted(missing_train)}"
            )

        return all_oos, all_train

Phase 4. Error Policy
---------------------

Per-step behaviour stays unchanged:

- run_wf_for_grid_point catches per-step exceptions and returns a
  runtime_error StepResult.
- run_wf_train_for_grid_point does the same.
- Workers use those same runner functions, so StepResult.error_message
  and error_type round-trip through pickle.

Infrastructure classes:

1. Plain task exception
   Trigger: raise RuntimeError inside _run_grid_point_both or another
   uncaught task function.
   Manifestation: future.result() re-raises the original exception;
   pool stays alive.
   Policy: no fallback. Outer orchestrator catches into result.error.
   execution_mode remains "parallel".

2. Hard worker crash
   Trigger: os._exit(17), SIGKILL, segfault.
   Manifestation: future.result() raises BrokenProcessPool.
   Policy: fallback only if fallback_to_sequential=True. Discard
   partial all_oos/all_train and rerun the whole wf_grid_execution
   stage sequentially. Successful fallback sets
   execution_mode="parallel_then_fallback"; no fallback leaves
   execution_mode="parallel" with result.error set.

3. Initializer failure
   Trigger: raise inside _init_worker, for example ImportError under
   WF_GRID_FORCE_INIT_FAIL=1.
   Manifestation: BrokenProcessPool.
   Policy: same as hard worker crash.

4. Pickle / submission failure
   Trigger: unpicklable initargs or task argument.
   Manifestation: TypeError or pickle.PicklingError synchronously from
   submit or later from future.result().
   Policy: same fallback policy as other infrastructure failures.

When fallback runs:

- discard all partial parallel dictionaries;
- record failed-parallel-time + sequential-rerun-time together under
  wf_grid_execution;
- set execution_mode to "parallel_then_fallback" only after the
  sequential rerun succeeds;
- log one warning with failure type and message.

Dataclass / YAML Asymmetry
--------------------------

This is deliberate:

- ExecutionConfig() default has parallel_enabled=False so direct tests
  and programmatic callers stay sequential.
- config.yaml has parallel_enabled=True so production run.py gets
  parallelism by default.
- run_grid_pipeline(parallel_enabled=...) and CLI flags are the explicit
  override layer.

Precedence:

    explicit kwarg / CLI flag > config.yaml > dataclass default

This prevents Windows spawn from breaking monkeypatch isolation in unit
tests while still enabling parallel production runs by default after the
final config.yaml flip.
