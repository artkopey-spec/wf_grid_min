# TZ: Phase 2 plan for ZigZag ST trade_filter integration into SuperTrend Tester (legacy-only)

## 1. Goal

Prepare a detailed implementation plan for Phase 2: integration of
`trade_filter.type: zigzag_st_mode` into the SuperTrend Tester legacy code path.

Scope decision: in Phase 2, enabled ZigZag ST filtering is supported only for
`segmentation.mode: legacy` (100%, 75%, 50%, 33%, 25%). The existing
`segmentation.mode: equal_blocks` path remains supported only for disabled
filter runs (`trade_filter` absent or `enabled: false`). A config that combines
`segmentation.mode: equal_blocks` with `trade_filter.enabled: true` must fail
clearly before global stats materialization, backtest, or export.

The plan must be implementation-ready, auditable, and consistent with the
primary contract:

`docs/zigzag_st_trade_filter_spec_v1_1.txt`

Appendix A v1.1 is the source of truth. The Phase 2 plan must not change or
reinterpret this contract. This TZ intentionally narrows the tester delivery
scope to legacy mode only; equal_blocks enabled-filter support is future work
requiring a separate owner decision.

## 2. Existing context

Phase 1 plan:

`docs/zigzag_st_implementation_plan_hybrid_v2_final.txt`

Current Phase 1 scope is WF Grid only. Tester work is explicitly outside
Phase 1 and appears only as a backlog placeholder:

- tester config support for `trade_filter`;
- tester validators;
- wire filter into `run_all_periods`;
- reject `run_equal_blocks` / `segmentation.mode: equal_blocks` when
  `trade_filter.enabled: true`;
- `build_signal_events` carries diagnostics;
- tester Excel export;
- tester parity tests.

The Phase 2 plan must expand this placeholder into a concrete work plan.

## 4. Non-negotiable contract requirements

The Phase 2 plan must explicitly preserve the following Appendix A v1.1
requirements:

1. The filter is integrated inside the backtest execution path before
   returns/equity/trades/metrics are computed.
2. Post-filtering `trades_df` after the backtest is forbidden.
3. When enabled, the ZigZag ST FSM builds `filtered_positions` directly.
4. The disabled path (`trade_filter` absent or `enabled: false`) must remain
   baseline-compatible with current tester behavior.
5. All thresholds and ZigZag heights are numeric price fractions, not percent
   strings and not 0-100 values.
6. ZigZag pivots and height metrics use close-only price input.
7. `trade_mode` can only be narrowed by the filter; the filter must not expand
   allowed directions.
8. Same-bar ordering, `OPEN_TO_OPEN` indexing, `ST_STOPPING`,
   freeze counters, repeated trigger suppression, and fail-closed behavior
   must follow Appendix A.
9. Runtime diagnostics must expose the Appendix A required keyset when the
   filter is enabled.
10. Trade-level diagnostics must include required entry/exit fields and
    filter-specific exit reasons.

Any conflict between tester legacy behavior and Appendix A must be recorded in
the plan as a decision point requiring owner approval before implementation.

## 5. Required scope decisions in the plan

The plan must make explicit decisions for these areas.

### 5.1 Code reuse / synchronization model

Decide how tester receives the filter implementation:

- sync/copy the existing `donor/supertrend_optimizer/core/zigzag_st_filter.py`
  into the tester subtree;
- migrate tester backtest/result/run/trades code toward the main donor version;
- or introduce a shared module strategy.

The plan must identify the chosen mode, list files affected, and define a
regression gate proving the tester disabled path is unchanged.

### 5.2 Full-dataset global stats in tester

The plan must define exactly how `ZigZagGlobalStats` is materialized for tester.

Required baseline decision:

- for CLI / `run_batch_tester.py`, full dataset means the loaded full CSV
  DataFrame before legacy period slicing or equal-block segmentation;
- for direct function calls that receive a pre-sliced DataFrame, full dataset
  means the DataFrame passed into that public function, unless the plan adds an
  explicit `full_df` / `zigzag_global_stats` argument.

The plan must cover the enabled legacy tester mode and the equal_blocks
rejection/baseline policy:

- legacy mode: `run_all_periods` over 100%, 75%, 50%, 33%, 25%;
- equal_blocks mode: `run_equal_blocks` remains unchanged when the filter is
  absent/disabled; `trade_filter.enabled: true` must be rejected before stats or
  backtest.

### 5.3 Slice offsets and diagnostics alignment

The plan must define `global_offset` / index handling for legacy:

- legacy period slices using last `n_period` bars;
- `run_all_periods` over 100%, 75%, 50%, 33%, 25%;
- trade-level diagnostic indexing in period-slice coordinates.

The plan must explicitly state that equal_blocks extended-slice diagnostics,
target-segment trim, and trade row rebase rules are out of scope for enabled
filter runs in Phase 2. They may appear only as future work.

### 5.4 Tester config contract

The plan must define tester config support for:

```yaml
trade_filter:
  enabled: false
  type: zigzag_st_mode
  zigzag:
    global_stats_source: full_dataset
    leg_height_mode: pct
    reversal_threshold: ...
    candidate_trigger_threshold: ...
    candidate_trigger_quantile: auto
    global_median: auto
    local_window: ...
  triggers:
    candidate_threshold:
      enabled: true
    confirmed_median:
      enabled: true
  lifecycle:
    freeze_confirmed_legs: ...
    stop_check: confirm_bar_only
    stopping_exit: opposite_st_flip
  diagnostics:
    export_state_columns: true
    export_trigger_columns: true
```

The plan must decide whether tester uses plain dict validation or imports /
mirrors WF Grid dataclasses and validation rules. Unknown keys and invalid
types must fail clearly.

### 5.5 Signal events semantics

The plan must decide how `build_signal_events` changes when the filter is
enabled.

It must distinguish:

- raw SuperTrend flip events;
- filter-allowed entries/reversals;
- filter-blocked ST flips;
- filter stop exits, especially `filter_stopping_opposite_flip`.

The plan must prevent a misleading Signals sheet that looks like raw ST trades
when actual trades were filtered.

### 5.6 Excel export

The plan must define tester Excel changes for:

- legacy workbook sheets: Summary, Metrics_*, Trades_*, Signals, false start;
- equal_blocks workbook disabled/baseline parity only;
- conditional inclusion of filter diagnostic columns only when enabled;
- deterministic column order;
- empty-trade behavior with correct headers;
- no filter columns in disabled export parity tests.

The plan must state whether a dedicated bar-level diagnostics sheet is added,
for example `FilterDiagnostics_100`, and what size / feature flag controls it.
`FilterDiagnostics_S1` / equal_blocks diagnostic sheets are out of scope.

## 7. Required test matrix

The Phase 2 plan must define tests for at least these cases:

1. Import smoke for tester after adding filter code.
2. `trade_filter` absent equals current tester baseline.
3. `trade_filter.enabled=false` equals current tester baseline.
4. Invalid tester config keys/types fail before running backtest.
5. Enabled filter in `run_period` / `run_all_periods` changes positions before
   returns/equity/trades/metrics.
6. `segmentation.mode: equal_blocks` with `trade_filter.enabled: true` fails
   clearly before `build_zigzag_global_stats`, `run_equal_blocks`, or export.
7. No post-filtering of `trades_df`.
8. Close-only ZigZag behavior is invariant to altered high/low values.
9. Full-dataset global stats are computed once from the intended full tester
   dataset and reused consistently across legacy slices.
10. Legacy period diagnostics lengths align with each period result.
11. Legacy diagnostics summary counters are internally consistent.
12. Trade-level diagnostics attach correct entry state, trigger source, and
    exit reason.
13. `OPEN_TO_OPEN` entry and exit diagnostic indexing is pinned.
14. `trade_mode=long` blocks short entries and `trade_mode=short` blocks long
    entries.
15. `ST_STOPPING` closes only on opposite ST flip and does not open a reverse
    trade in that state.
16. Signals export distinguishes raw ST flips from filter decisions.
17. Legacy Excel disabled export has no filter columns and matches baseline.
18. Legacy Excel enabled export preserves filter columns in deterministic order.
19. Equal-blocks Excel disabled export has no filter columns and matches
    baseline.
20. Empty-trade enabled legacy run still exports valid headers.
21. The tester path remains compatible with current `run_batch_tester.py` and
    CLI usage.

## 9. Acceptance criteria for the plan itself

The plan is acceptable only if:

1. It references `docs/zigzag_st_trade_filter_spec_v1_1.txt` as the primary
   contract.
2. It expands the WP12 tester placeholder into concrete work packages.
3. It covers enabled legacy tester mode and explicitly rejects enabled
   equal_blocks mode while preserving disabled equal_blocks parity.
4. It explicitly forbids post-filtering trades.
5. It defines global stats materialization for tester.
6. It defines diagnostics and Excel behavior.
7. It contains disabled-path parity gates.
8. It contains enough file-level detail that implementation can begin without
   another discovery pass.
