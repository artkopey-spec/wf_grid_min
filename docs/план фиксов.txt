# 1. Executive Mini-Plan

1. **Закрыть metric contracts fail-closed.**
   Broken ok-step metrics не должны доходить до gates, aggregation, scoring как `ok`.
   Hard-required metrics: `sum_pnl_pct`, `max_drawdown`, `num_trades`.

2. **Исправить OOS boundary / carry-in mismatch через force-flat.**
   OOS начинается без carry-in позиции. Bar-level и trade-level OOS metrics должны описывать одну и ту же экономику.

3. **Сделать YAML strict.**
   Unknown top-level/nested keys должны падать с понятным путём к ключу. Добавляется `schema_version: 1`.

4. **Сделать ok-only семантику явной.**
   Агрегация остаётся только по `step_status == ok`. В summary/XLSX добавляется явный marker `aggregation_scope = "ok_steps_only"`.

5. **Убрать false green flags в отчёте.**
   Disabled gates не выглядят как passed, Tier 1 не выглядит как proven edge/trading signal, bucket matrix failure виден в artifacts.

# 4. Frozen Design Decisions

1. **Positive `max_drawdown` policy.**
   `max_drawdown > 0` на ok-шаге = metric contract violation.
   Result: `step_status="invalid"`, `error_type="metric_contract"`.
   Это не `gate_failed`. Hard pipeline error не используется в main flow.

2. **Metric validation location.**
   Validator = отдельная функция:
   `validate_metric_contracts(step_oos_long: pd.DataFrame) -> pd.DataFrame`
   в `wf_grid/collect/metric_contracts.py`.
   Вызов: после `collect_oos_steps(...)`, до `apply_step_gates(...)`.
   Collector остаётся thin. Gates работают только с `step_status == ok`. `status_model.py` не раздувается.

3. **Hard vs soft metric contract.**
   Hard-required metrics for ok steps:
   - `sum_pnl_pct`
   - `max_drawdown`
   - `num_trades`

   For hard metrics: missing / `NaN` / `inf` / `INVALID_METRIC_VALUE` = `invalid`.

   Soft metrics:
   - `win_rate`
   - `profit_factor`
   - `avg_trade`
   - `sharpe`
   - `sortino`
   - `cagr`

   For soft metrics: `INVALID_METRIC_VALUE` = `invalid`; legitimate `NaN` is allowed and aggregator drops it via existing `dropna()` semantics.

4. **Carry-in policy.**
   Chosen policy: **force-flat at OOS boundary**.
   OOS starts flat. Carry-in exposure is not counted in OOS trades/PnL.
   Train metrics are not recalculated.
   Force-flat применяется на bar-level (OOS starting state flat); trade-level filter `entry_index < boundary` остаётся в силе и согласован с bar-level. Double-filter и полное удаление filter одновременно запрещены.
   Expected effect: golden values and Tier 1 candidate set may change. This is an expected baseline update, not a regression.

5. **Disabled worst-segment gate representation.**
   `worst_segment_pnl_threshold: null` means:
   - `worst_segment_gate_enabled = False`
   - storage: `gate_ok_worst_segment = pd.NA` internally in the DataFrame
   - display: `"disabled"` / `"N/A"` in XLSX cells (rendered from the NA storage value; это И, не ИЛИ)
   - disabled gate is excluded from composite AND
   - disabled gate is excluded from `seed_gate_fail_reason`
   - `seed_gate_passed` остаётся `bool` dtype, не `object`/NA

6. **Strict YAML future fields.**
   Add allowed top-level `schema_version: 1`.
   Unknown keys fail.
   Future fields require schema update or explicit schema version bump.
   Existing repo configs must be updated in the same PR.

7. **Tier 1 meaning.**
   Tier 1 = strong candidate inside current Phase A ok-only logic.
   Tier 1 does not mean proven edge, paper-ready, real-money-ready, tradable, or trading signal.

8. **`INVALID_METRIC_VALUE` source.**
   `wf_grid/config/schema.py` imports/re-exports `INVALID_METRIC_VALUE` from `supertrend_optimizer.utils.constants`.
   Add drift test to fail if values diverge.

9. **`n_segments` definition.**
   `n_segments = step_oos_long["wf_step"].nunique()` as global unique WF step count.
   It is not `n_ok_steps` and not per-candidate `n_total_steps`.

# 5. Testing Strategy

**Unit Tests**

- Metric contracts:
  - positive `max_drawdown` on ok step fails closed;
  - `NaN max_drawdown` fails;
  - `inf` hard metric fails;
  - `INVALID_METRIC_VALUE` in hard metric fails;
  - `INVALID_METRIC_VALUE` in soft metric fails;
  - legitimate `NaN profit_factor` does not invalidate ok-step;
  - legitimate `NaN sortino` does not invalidate ok-step;
  - missing `sum_pnl_pct`, `num_trades`, `max_drawdown` fails instead of defaulting.
- Config:
  - unknown top-level YAML key fails;
  - unknown `gates.*` key fails;
  - unknown `scoring/ranking` key fails;
  - typo `worst_segment_pnl_treshold` fails;
  - `schema_version` mismatch fails clearly;
  - all repo configs outside `donor/` load after schema update.
- Disabled gate:
  - `worst_segment_pnl_threshold: null` gives disabled/N/A, not `True`;
  - `worst_segment_gate_enabled is False`;
  - `seed_gate_passed` remains bool, not NA;
  - `seed_gate_fail_reason` excludes disabled gate.

**Integration Tests**

- OOS boundary force-flat:
  - carry-in win;
  - carry-in loss;
  - carry-in flat;
  - reversal exactly at boundary;
  - only carry-in exposure in OOS;
  - train metrics are not recalculated by force-flat;
  - OOS bar/trade economics are reconciled under the chosen force-flat policy.
- Pipeline path:
  - invalid metric step excluded from ok aggregation;
  - tiering still runs on ok-only aggregates;
  - metric-contract invalid is not converted to `gate_failed`.

**Golden/Differential Tests**

- Aggregation remains `step_status == ok` only.
- `aggregation_scope == "ok_steps_only"` present in summary/XLSX.
- No all-segment/non-ok aggregate metrics introduced.
- No columns with `_all`, `_failed`, `_nonok` aggregate semantics appear.
- `INVALID_METRIC_VALUE` donor/wf_grid drift test.
- `test_golden_values.py` and `test_e2e_real_data.py` baselines may be updated in the same PR as M1/M2/M3/OOS1; document this as expected effect.

**XLSX/Summary Tests**

- Disclaimer substrings present:
  - `ok-only`
  - `not proof of edge`
  - `not a trading signal`
  - `limited debug`
- Forbidden wording absent (negative test `test_forbidden_wording_absent`):
  - forbidden substrings (case-insensitive): `proven`, `production-ready`, `production grade`, `tradable`, `paper-ready`, `trading signal`, `edge confirmed`
  - scope: user-facing strings in `ranking/tiering.py`, `export/xlsx_writer.py`, `export/summary_builder.py`, optional `export/disclaimers.py`
  - исключения: `not a trading signal` (разрешённый disclaimer) не учитывается как forbidden match
- Disabled worst-segment gate is rendered as `disabled`/`N/A`, not passed.
- Top-1 trades sheet is labelled limited export/debug, not full-grid audit.
- Bucket matrix failure visible via `bucket_matrix_status` and `bucket_matrix_error`.

**Regression Tests**

- Sparse `wf_step=[1,3,4]` gives `n_segments=3`.
- Empty `step_oos_long` gives `n_segments=0`.
- Existing ok-only ranking/tiering behavior remains stable except intended contract/boundary changes.
- Positive DD can no longer silently pass drawdown gates.
- `PipelineResult()` remains backward-compatible with new bucket matrix fields using defaults.

# 6. Definition of Done

DoD for current goal: **Research-safe ok-only Phase A artifact**.

Done when:

- broken ok-step hard metrics fail closed before gates/aggregation;
- required ok metrics cannot silently default to zero;
- soft metric NaN can remain legitimate missing data, while sentinel still fails;
- OOS boundary uses force-flat policy consistently;
- bar-level and trade-level OOS metrics no longer describe conflicting exposure;
- YAML typos fail loudly with exact key path;
- existing repo configs are updated to `schema_version: 1`;
- aggregation remains ok-only and is explicitly marked as `ok_steps_only`;
- Tier 1 remains, but means only Phase A ok-only tier;
- disabled gates are visibly disabled, not passed;
- bucket matrix failure is visible in summary/XLSX, not only logs;
- summary/XLSX cannot reasonably be read as production/trading report;
- golden baseline changes from metric contracts / force-flat are documented as expected;
- listed regression tests pass.

# 7. First PR Requirements

Обязательные условия для первого PR, без которых merge запрещён:

1. **Порядок реализации — из §3.**
   Первым шагом идёт YAML compat sweep + strict validator (§3 п.1). Реализация вне этого порядка сломает реальные configs при merge.

2. **Goldens обновляются одним commit'ом, отдельным от логики.**
   Все изменения `test_golden_values.py` и `test_e2e_real_data.py`, вызванные M1/M2/M3/OOS1, собираются в один отдельный commit в том же PR с сообщением вида:
   `"chore(goldens): baseline bump — expected effect of metric contracts + force-flat per plan §4.1/§4.3/§4.4"`.
   Смешивать обновление baseline с кодовыми изменениями в одном commit'е запрещено.

3. **`test_all_repo_configs_load` подключён в CI до merge strict validator.**
   Тест, проверяющий, что все `*.yaml` / `*.yml` вне `donor/` загружаются новым strict validator'ом без ошибок, должен попасть в CI раньше, чем в main окажется strict-YAML код. Порядок коммитов внутри PR:
   - commit A: добавить `schema_version: 1` в существующие configs + добавить `test_all_repo_configs_load` (пока зелёный на старом лояльном loader);
   - commit B: включить recursive strict validation в `loader.py`;
   - commit C: остальные фиксы по §3.
   Так падение теста в CI однозначно указывает на проблему именно в strict validator, а не в configs.

4. **PR-description явно перечисляет ожидаемые baseline-изменения.**
   В описании PR указывается: (a) какие goldens бампнуты и почему, (b) какие новые колонки появляются (`aggregation_scope`, `worst_segment_gate_enabled`, `bucket_matrix_status`, `bucket_matrix_error`), (c) что Tier 1 candidate set может измениться как expected consequence contracts + force-flat.
