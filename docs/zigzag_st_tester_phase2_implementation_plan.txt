        segment_trades  = filter_trades_to_segment(ext_result.trades_df, oos_boundary)
                          then rebase entry_index/exit_index by -oos_boundary
        recalc_trade_metrics(segment_metrics, segment_trades, min_trades_required)

## 3. Target architecture

### 3.1 Code reuse / synchronization model — Mode C is the only viable path

ТЗ §5.1 формально оставлял выбор между тремя режимами. Audit-fix v0.3 фиксирует, что Mode A и Mode B фактически отозваны, потому что Phase 1 engine integration уже целиком в active donor (donor/supertrend_optimizer/...) и активно тестируется (wf_grid/tests/test_wp*_zigzag*.py). Копировать тот же engine во вторую ветку (Mode A) или переносить его в третий subtree (Mode B) — это пытаться повторно реализовать уже работающий код и расходиться с ним.

Mode C — sync to active donor (единственный жизнеспособный):

- Активным runtime тестера становится `donor/supertrend_optimizer/...` (тот же subtree, что использует WF Grid Phase 1). `donor TESTER/` либо переписывается в тонкий wrapper над `donor/`, либо удаляется.
- Tester-only артефакты (`testing/runner.py`, `testing/signal_events.py`, `io/excel_tester.py`, `cli/tester.py`, `run_batch_tester.py`, `config_tester.yaml`) остаются и расширяются, но импортируют `engine/run.py`, `core/backtest.py`, `core/zigzag_st_filter.py`, `engine/result.py` из `donor/`.
- Преимущество: один `zigzag_st_filter.py`, один `RawBacktestArtifacts`, один `BacktestResult.filter_diagnostics` invariant; исключает drift между WF Grid и Tester.

Отозванные альтернативы (audit-fix v0.3):

- Mode A — sync copy donor/supertrend_optimizer/core/zigzag_st_filter.py и связанных файлов в donor TESTER/. ОТОЗВАН: дублирование уже-реализованного, постоянный риск drift, и dead-code в `donor TESTER/.../engine/run.py`. Если в `donor TESTER/` уже есть стейл-копии этих файлов, они УДАЛЯЮТСЯ или превращаются в `from supertrend_optimizer.engine.run import run_single_backtest` shim'ы.
- Mode B — shared module в новом subtree (shared/supertrend_filter/). ОТОЗВАН: большой rewrite imports в обеих ветках; не оправдан, когда Mode C уже даёт single source.

#### 3.3.2 filter_diagnostics_summary contract (Phase 2 tester aggregate)

`filter_diagnostics_summary` — отдельный dict, который ЖИВЁТ в `PeriodResult` (НЕ в `BacktestResult.filter_diagnostics`). Он содержит per-period агрегаты, нужные Excel-summary блокам и golden snapshots:

    summary = {
        "mode"                        : "zigzag_st_mode",          # echo
        "thresholds"                  : {                          # echo нормализованного config + materialized stats
            "reversal_threshold": float,
            "candidate_trigger_threshold": float,
            "candidate_trigger_quantile": Optional[float],
            "candidate_trigger_source"  : str,   # REQUIRED (owner decision v0.5.1):
                                                  # "explicit" | "quantile" — echo из
                                                  # ZigZagGlobalStats.candidate_trigger_source
                                                  # (Phase 1 plan §3.2). Не Optional.
            "global_median": float,
            "local_window": int,
            "freeze_confirmed_legs": int,
        },
        "global_offset"               : int,                       # slice metadata
        "counters"                    : {
            "raw_st_flips"            : int,  # все ST color changes (decision bars)
            "passed_entry_signals"    : int,  # raw flips, разрешённые фильтром
            "blocked_entry_signals"   : int,  # raw flips, заблокированные
            "blocked_filter_off"          : int,
            "blocked_waiting_first"       : int,
            "blocked_trade_mode"          : int,
            "blocked_local_median"        : int,
            "blocked_invalid_stats"       : int,
            "blocked_stopping"            : int,
            "lifecycle_starts"            : int,
            "median_stop_triggered"       : int,
            "exits_opposite_flip"         : int,
        },
        "bars_in_state"               : {
            "OFF": int, "WAIT_FIRST_ST_FLIP": int,
            "ST_ACTIVE_FREEZE": int, "ST_ACTIVE_MONITORING": int,
            "ST_STOPPING": int,

Sanity-инварианты (audit-fix v0.5; все проверяются тестом #11):

1. `passed_entry_signals + blocked_entry_signals == raw_st_flips` (full reconciliation raw → decision).
2. `sum(blocked_*) == blocked_entry_signals` (block-reason histogram complete).
3. `lifecycle_starts == count(state_arr transitions WAIT_FIRST_ST_FLIP -> ST_ACTIVE_FREEZE)` (cross-source check со `state_arr`).
4. `median_stop_triggered == sum(filter_diagnostics["median_stop_triggered"])` (cross-source check с bar-level int8 mask).
5. `exits_opposite_flip == count(trade rows where exit_reason == "filter_stopping_opposite_flip")` (cross-source check с trade-level diagnostics, spec §13 lines 554-558).
6. `sum(bars_in_state.values()) == len(filter_diagnostics["trade_filter_state"]) == len(positions)`.

ВАЖНО (audit-fix v0.5): bars_in_state-инвариант записан через `len(positions)`, не через `n_period`. В legacy тестере Phase 2 `early_exit_enabled=False` hardcoded (§6.2), поэтому фактически `len(positions) == n_period`, но опираться на это в инварианте небезопасно: active donor `BacktestResult.__post_init__` синхронно режет `filter_diagnostics` по `len(positions)` при early-exit truncation (Phase 1 plan §8.2 rule 7). Если в будущем default'ы изменятся, инвариант через `len(positions)` останется корректным.

Для legacy attribution ведётся на полном period slice; `oos_boundary` не используется.

`counters` ключи являются стабильными summary-алиасами поверх `filter_block_reason` enum из spec §13 (например, `blocked_waiting_first` агрегирует `waiting_for_allowed_st_flip`, `blocked_local_median` агрегирует `local_median_unavailable`, `blocked_stopping` агрегирует `stopping_mode_no_new_entries`). Это упрощает Excel-колонки, не меняя bar-level enum. Не-blocked counters (`lifecycle_starts`, `median_stop_triggered`, `exits_opposite_flip`) — derived из `state_arr` / bar-level int8 mask / trade rows соответственно (см. cross-source инварианты выше).

## 4. Slice offsets, indexing, diagnostics alignment

### 4.1 Legacy path (run_all_periods)

Каждый период берёт last n_period = max(1, floor(n_total * fraction)) bars:

    period_slice = df.iloc[-n_period:]
    global_offset = n_total - n_period   # абсолютный индекс start в полном df

FSM работает на длине n_period. filter_diagnostics имеют длину len(positions) (с учётом возможной early-exit truncation, в тестере early_exit hardcoded False — см. §6).

local_median_N накапливается causally внутри slice и не заимствует confirmed legs из соседних периодов; global_offset используется только для metadata/diagnostics.

### 4.2 Equal-blocks mode policy

`run_equal_blocks` не получает enabled filter wiring в Phase 2 v0.4.

Config gate:

    if segmentation.mode == "equal_blocks" and trade_filter.enabled:
        raise ConfigError(
            "trade_filter.type=zigzag_st_mode is supported only with "
            "segmentation.mode=legacy in Phase 2"
        )

Disabled-path (`trade_filter` absent или `enabled=false`) сохраняет текущую equal_blocks логику без новых полей, колонок, листов и без пересчёта ZigZagGlobalStats.

### 4.3 Trade diagnostics indexing (legacy)

Trade-level diagnostics (`entry_filter_state`, `entry_trigger_source`, `exit_reason`) приклеиваются `attach_trade_filter_diagnostics(...)` внутри `run_single_backtest` после `extract_trades`. В legacy `run_period` rebase не нужен: индексы изначально в координатах period slice.

### 4.4 Indexing rule (наследуется из Phase 1 §8.4)

OPEN_TO_OPEN (canonical для тестера тоже):

    entry_signal_idx = max(entry_index - 1, 0)
    exit_signal_idx  = max(exit_index  - 1, 0)

exit_reason определяется по FSM state на exit_signal_idx, а не на exit_index (см. Phase 1 plan §8.4.1).

Edge-case pending_open_trade_at_end (audit-fix v0.5; spec §13 lines 554-558):

    pending_exit_idx = len(filter_diagnostics["trade_filter_state"]) - 1
                     # == len(positions) - 1 by length-invariant
    if exit_index >= pending_exit_idx:
        exit_signal_idx = SKIP        # exit lookup НЕ выполняется
        exit_reason     = "pending_open_trade_at_end"
    else:
        exit_signal_idx = max(exit_index - 1, 0)
        # exit_reason из state на exit_signal_idx

Контракт фиксирует defensive `>=` (а не строго `==`), потому что это уже реализованный invariant active donor: `attach_trade_filter_diagnostics` делает `if exit_index >= pending_exit_idx` (`zigzag_st_filter.py:1597`). Spec docstring (`zigzag_st_filter.py:1526`) формулирует правило как `==`; `>=` — out-of-range guard на случай race-condition / bound errors. Тестер наследует эту защиту 1-в-1; никаких параллельных интерпретаций.

Equal-blocks edge `exit_index >= n_bars_segment` out of scope v0.4, потому что enabled filter diagnostics для equal_blocks не материализуются.

### 4.5 Координаты артефактов

| Артефакт | Где хранится | Координаты |
|---|---|---|
| BacktestResult.filter_diagnostics (от run_single_backtest) | runtime, до tester-side обработки | period slice |
| PeriodResult.filter_diagnostics (legacy) | tester result | period slice (= extended, prepend=0 в legacy) |
| trades_df.{entry_filter_state,entry_trigger_source,exit_reason} | tester result | trade-level columns, attached in period-slice coordinates |
| PeriodResult.filter_diagnostics_summary | tester result | агрегаты по period slice |

Критический инвариант: len(PeriodResult.filter_diagnostics[k]) == len(result.positions) для каждого legacy period.

**Финальная сигнатура shared validator (audit-fix v0.5.1 — fixing H-4).**
Текущая wf_grid функция: `_validate_trade_filter(tf: TradeFilterConfig, errors: list[str], raw_user_keys: frozenset[tuple[str, ...]])` (loader.py:559). Tester вызывает shared напрямую: `validate_trade_filter(tf, errors, raw_user_keys, caller_pipeline="tester")`. Нет `parsed_dict`, нет `raw_presence_set` — только стандартные `TradeFilterConfig` + `frozenset[tuple[str, ...]]`.

#### 5.3.1 Strict schema (reject unknown keys)

Эталон: docs/zigzag_filter.md §6.3 + tests/cli/test_zz_filters_config.py в C:\2.0 — TESTER. Phase 2 принимает аналогичное правило на уровне tester:

- ошибка форматируется по образцу Phase 1: ConfigError(f"trade_filter: unknown key {dotted_path!r}; allowed keys: {sorted(allowed)}").

Это закрывает class-of-bugs «опечатка в YAML работает молча».

#### 5.3.2 Raw-YAML presence tracking (явное расширение Phase 1 §6.4.1)

- explicit (ключ присутствует в YAML, даже с тем же значением, что был бы default) — это user-intent.

Loader должен пробрасывать raw-presence-set из yaml.safe_load до validator'а отдельным аргументом (не реконструировать сравнением с default). Тест #4 §10 покрывает three-case: numeric+absent / auto+explicit / numeric+explicit.

        zigzag_global_stats = build_zigzag_global_stats(
            close=df["close"].values,
            trade_filter_config=cfg.trade_filter,
        )
        # Сигнатура active donor: build_zigzag_global_stats(close, trade_filter_config)
        # (zigzag_st_filter.py:611). Close-only ZigZag — no high/low (Appendix A §3.4).
        # Init failure (Appendix A §12.3 spec):
        #   no confirmed legs / global_median NaN/Inf
        #   auto threshold + n_legs_total < min_legs_for_quantile
        #   materialized candidate_trigger_threshold NaN/Inf
        # -> ConfigError ДО любой сегментации.

zigzag_global_stats передаётся только в legacy run (`run_all_periods`). Для `equal_blocks + enabled` конфигурация отвергается до построения stats; для disabled equal_blocks stats не нужны.

### 7.2 Direct function calls (tests, ad-hoc)

run_period(df_slice, ..., zigzag_global_stats=...) принимает уже материализованные stats. Для тестов и debug, если zigzag_global_stats=None при trade_filter_config.enabled=true, поведение fail-closed:

- run_period рейзит ConfigError("zigzag_global_stats required when filter enabled");
- эта проверка выполняется до run_single_backtest чтобы не запускать ZigZag pivot pass на partial-dataset.

run_period не вычисляет stats сам по df_slice: per-spec §12, global_median — full-dataset statistic, не slice-local. План тестера явно фиксирует, что df_slice, переданный в run_period, считается «full dataset для этого вызова» только тогда, когда caller сам это явно гарантирует (например, тест на синтетических данных).

### 7.3 run_all_periods legacy-mode policy

run_all_periods(df, ..., trade_filter_config, zigzag_global_stats) использует переданный df как full dataset для расчёта zigzag_global_stats, если caller не передал готовые stats:

    if trade_filter_config is not None and trade_filter_config.enabled:
        if zigzag_global_stats is None:
            zigzag_global_stats = build_zigzag_global_stats(
                close=df["close"].values,
                trade_filter_config=trade_filter_config,
            )
            # Same init-failure rules.

Это соответствует §5.2 ТЗ: для CLI/batch full dataset = валидированный df до сегментации; для direct calls — df, переданный в публичную функцию.

run_equal_blocks(...) не материализует zigzag_global_stats в Phase 2 v0.4. Если caller передал enabled trade_filter_config, функция завершает работу ConfigError до backtest.

## 8. Signal events semantics

### 8.1 Ограничение текущей build_signal_events

testing/signal_events.py:build_signal_events сейчас:

- работает только на 100% slice (legacy path, run_all_periods вызывает с trend=results[0].result.trend);
- регистрирует все ST color changes (prev != curr, prev != 0);
- не различает «ST flip разрешён фильтром» vs «ST flip заблокирован фильтром» vs «фильтр в ST_STOPPING и закрывает позицию».

Когда trade_filter.enabled=true, лист Signals может вводить в заблуждение: показывает все raw ST flips, тогда как реальные сделки совсем другие.

### 8.2 Phase 2 контракт build_signal_events

build_signal_events(...) расширяется новым optional parameter:

    def build_signal_events(
        df, trend, atr_period, trade_mode, execution_model,
        filter_diagnostics: Optional[Dict[str, np.ndarray]] = None,  # NEW
    ) -> pd.DataFrame

Поведение:

1. filter_diagnostics is None (disabled или фильтр не активен) -> текущее поведение bit-identical, никаких новых столбцов. Никаких новых строк (audit-fix v0.3).
2. filter_diagnostics is not None:
   - набор строк остаётся ПРЕЖНИМ — `prev != curr and prev != 0` (см. signal_events.py:112). Initialization-флипы 0 -> ±1 в Signals НЕ инжектируются (они skipped в active donor; добавление таких строк сломало бы disabled-baseline-bit-identity и golden snapshots, см. §10 #17, #19);
   - добавляются ТОЛЬКО новые столбцы:
     - `filter_state_at_signal` (OFF / WAIT_FIRST_ST_FLIP / ST_ACTIVE_FREEZE / ST_ACTIVE_MONITORING / ST_STOPPING) — берётся из `filter_diagnostics["trade_filter_state"][signal_bar_index]` (string, не int8);
     - `filter_decision` ∈ { entry_allowed, entry_blocked_filter_off, entry_blocked_waiting_first, entry_blocked_trade_mode, entry_blocked_local_median, entry_blocked_invalid_stats, entry_blocked_stopping } — производное от `filter_block_reason` + `filter_allowed_entry` на `signal_bar_index`. Сценарий `raw_initialization` (для 0 -> ±1) В этом списке НЕТ, так как такие строки не попадают в Signals (см. п.выше);
     - `filter_block_reason` — копия `filter_diagnostics["filter_block_reason"][signal_bar_index]` (str из spec §13 enum: `none` / `filter_off` / `waiting_for_allowed_st_flip` / `trade_mode_disallowed_flip` / `local_median_unavailable` / `invalid_stats` / `stopping_mode_no_new_entries` / `insufficient_global_stats`);
     - `filter_trigger_source` — копия `filter_diagnostics["trade_filter_trigger_source"][signal_bar_index]` (str: `candidate_threshold` / `confirmed_median` / `both` / `none`).
3. Lookup на signal_bar_index (decision bar t), не на execution bar t+1. Это согласовано с правилом Appendix A v1.1 §10 — decision на close, execution на next open.
4. Empty result (нет ST flips): DataFrame с правильными headers (включая filter columns если enabled) и нулём строк.

ВАЖНО (audit-fix v0.3): Phase 2 НЕ ТРОГАЕТ цикл итерации `for t in range(1, n)` и условие `if curr == prev or prev == 0: continue` (signal_events.py:107–113). Disabled-bit-identity baseline зависит от того, что цикл генерирует РОВНО те же строки. Любая попытка инжектировать 0 -> ±1 как «raw_initialization»-строку отозвана. Для покрытия 0 -> ±1 семантики предусмотрен FSM unit test (см. §10 #15: «0 -> ±1 не открывает в WAIT и не закрывает в STOPPING»), не отдельная Signals-строка.

### 8.3 Защита от misleading Signals sheet

ТЗ §5.5 явно требует: «prevent a misleading Signals sheet that looks like raw ST trades when actual trades were filtered».

OWNER DECISION (v0.5.1): default — **same sheet name `Signals`, NO meta-row, filter columns only**. Black-listed варианты:

- ~~Переименование в `Signals_Filtered`~~ — отозвано: ломает downstream tooling, у которого имя листа hardcoded.
- ~~Meta-row в первом ряду (`# trade_filter: zigzag_st_mode (enabled)`)~~ — отозвано: ломает обычный `pd.read_excel(header=0)` / openpyxl readers, у которых первая строка трактуется как header.

Финальное решение в плане:

- лист Signals СОХРАНЯЕТ имя `Signals` И при enabled=true, И при disabled. Header строго на первой строке (row 1); никаких meta-строк, prefix-комментариев или дополнительных aux-rows;
- enabled-run отличается от disabled-run ИСКЛЮЧИТЕЛЬНО наличием четырёх filter-колонок (Filter State at Signal, Filter Decision, Filter Block Reason, Filter Trigger Source) — см. §9.5 точную позицию;
- downstream tool, который хочет detect'ить filtered run, делает это по факту присутствия колонки `Filter Decision` в header — а не по sheet name или meta-row;
- false-start sheet (false start) тоже учитывает filter_block_reason для совпадения trade «->» signal: blocked rows фильтруются из false-start логики (false-start про trades, а не про filter blocks);
- header в signals_df расширяется четырьмя filter-колонками; ТОЧНАЯ позиция в layout — single source of truth: §9.5 (Excel-layer): «после `Is Reversal`, до `Exec Price`». Эта же позиция используется при формировании in-memory `signals_df` — никаких альтернативных слотов (в т.ч. «после st_color_after», как было в черновике v0.2). Audit-fix v0.5: предыдущая формулировка снята.

Совместимость baseline: disabled-run Signals layout ровно такой же, как до Phase 2 (никаких четырёх filter-колонок, никаких meta-rows). Это покрывается golden-test #17 (legacy disabled). Enabled-run с четырьмя filter-колонками — golden-test #18.

Spec reference: Appendix A v1.1 §3.3, §10, §13, §15.4, §17.13

### 9.2.1 ZigZag_Trigger_Events reconstruction formula (audit-fix v0.5)

Источник истины: active donor `zigzag_st_filter.py:1234, 1299–1307`. `trade_filter_trigger_source[t]` инициализирован `"none"` для всех баров (line 1234) и устанавливается в `candidate_threshold` / `confirmed_median` / `both` ТОЛЬКО когда `state_at_bar_start == ZigZagFSMState.OFF` (line 1301). Repeated triggers в WAIT_FIRST_ST_FLIP / ST_ACTIVE_FREEZE / ST_ACTIVE_MONITORING / ST_STOPPING подавлены (spec §15.3) — значение остаётся `"none"`.

Следовательно reconstruction trigger events — без heuristic'и:

    trigger_event_rows = [
        t for t in range(n_bars)
        if filter_diagnostics["trade_filter_trigger_source"][t] != "none"
    ]

Контрактный invariant (тестируется тестом #30): для каждого `t in trigger_event_rows` верно `state_arr[t-1] in {OFF}` ИЛИ (`t == 0` AND `state_arr[t] != OFF`). То есть переход из OFF — необходимое условие trigger event, и `trigger_source != "none"` — достаточное.

Что НЕ используется (отозвано из v0.4 черновика): heuristic через `confirmed_legs_since_start` delta. `confirmed_legs_since_start` — динамический счётчик confirmed legs внутри активного lifecycle (`zigzag_st_filter.py:1328`); он шагает на confirm-bar в FREEZE / MONITORING. Это НЕ событие запуска контура B (которое уже отражено в `trigger_source == "confirmed_median"` при OFF→WAIT). Использовать delta этого счётчика для reconstruction trigger events некорректно — он вообще не возрастает в OFF.

Trigger ID нумерация — последовательная по возрастанию `t` для каждого period slice (1, 2, 3, …). Linked Trade ID заполняется на этапе `_link_trades_to_legs` (§9.5.1).

#### 9.5.1 Trade <-> trigger linker (двухступенчатый)

Эталон: io/excel_tester.py:_link_trades_to_legs в C:\2.0 — TESTER (~80 lines, two-step matching).

Линковка делается ОДИН РАЗ при экспорте, после того как trades_df и trigger_events_df оба сформированы. Алгоритм:

1. Шаг 1 — exact match: для каждого trade ищем trigger_events row, у которого trigger_bar == entry_signal_idx (decision bar). Это покрывает 100% случаев same-bar trigger + flip.
2. Шаг 2 — backward search: если шаг 1 не нашёл, ищем последний trigger_events row, у которого trigger_bar < entry_signal_idx и trigger_source != NONE и lifecycle_event=ENTER_ACTIVE этого же FSM-цикла. Это покрывает случай, когда trigger сработал на t1, но ST flip случился позже на t2 > t1 (freeze отсчитывается от t1, FSM пропускает первый flip после freeze).
3. Если ни один шаг не нашёл — Triggering * колонки получают "N/A" (sentinel из §9.1.1). В trades_df enabled-run такого случая быть не должно (любой trade происходит из allowed entry, у которого был trigger в FSM-цикле); отсутствие линка для конкретного trade row = test failure (см. §10 #18). Возможный «легитимный» N/A — pending_open_trade_at_end на последнем баре, если он попал в trades_df без явного trigger (то есть открыт ещё до lifecycle start; в текущей семантике Appendix A это не должно случаться, но колонка устойчива к этому случаю).

Линковка не модифицирует trades_df на «горячем» пути — только при export. Это важно: filter_diagnostics в BacktestResult не зависит от того, делаем ли мы export, и golden tests in-memory snapshots (#10/#11/#12) проходят даже когда экспорт-пакет не запущен.

## 11. Disabled-path rollback / parity strategy

1. Default value trade_filter.enabled = false (или блок отсутствует) во всех shipped configs (config_tester.yaml).
2. Phase 2 hard contract: disabled run производит byte-identical XLSX с baseline (snapshot pre-Phase-2). Это checking baseline workbook capture перед началом любых правок Phase 2 (analog WP1 в Phase 1):
   - donor TESTER/tests/baselines/result_legacy.baseline.xlsx
   - donor TESTER/tests/baselines/result_equal_blocks.baseline.xlsx
3. Golden tests (#17, #19) сравнивают полный XLSX (sheet names, column orders, cell values, formulas/formatting) с baseline: #17 legacy disabled, #19 equal_blocks disabled regression guard.
4. Любая регрессия в disabled-path = блокирующий incident; rollback-policy: revert последнего PR до зелёного baseline.
5. Rollback в production: переключить trade_filter.enabled на false (или удалить блок). Никаких side-effects не остаётся (config-only включение).

Spec reference: Appendix A v1.1 §11, §11.1, §17.1, §18
