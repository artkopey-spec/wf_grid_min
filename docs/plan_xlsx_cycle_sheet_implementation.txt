Финальный план реализации листа cycle в XLSX tester (v3 clean)
==============================================================

Статус документа
----------------
Версия 3.0 clean - самостоятельный финальный план, готовый к реализации.
Заменяет все предыдущие версии целиком.

В документ уже внесены исправления по итогам инженерного аудита:
- устранена неоднозначность materialize для `local_window`;
- явно определено поведение при non-finite `close/high/low` внутри цикла;
- wrapper `compute_confirmed_legs_reset_aware(...)` синхронизирован с
  validation-контрактом `compute_zigzag_per_bar(...)`;
- тестовый план разделен на обязательный минимальный набор перед
  реализацией и расширенную acceptance-карту.

Целевая область: legacy tester XLSX export.
Основной модуль экспорта: `donor/supertrend_optimizer/io/excel_tester.py`.
Исходное ТЗ: `docs/tz_xlsx_cycle_sheet.txt`.


Согласованные отклонения от ТЗ
------------------------------
Следующие пункты сознательно расходятся с буквальным текстом ТЗ и считаются
подписанными до начала реализации.

1. Количество колонок: 22 вместо 20.
   Колонки 21 `Сделок в цикле` и 22
   `% сделок с положительным фин результатом в цикле` добавлены по
   согласованию с владельцем ТЗ. Acceptance criteria фиксируют 22 колонки.

2. Символ отрицательного направления: ASCII `-` (0x2D), а не Unicode `−`
   (U+2212). Причина: стабильность строковых тестов и downstream-обработки.

3. Границы цикла берутся в close-decision шкале diagnostics
   (`FilterDiagnostics_100`), без сдвига на `open(t+1)`. Бар перехода в
   `OFF` не входит в цикл.


1. Цель
-------
При экспорте legacy tester результатов в Excel добавить лист `cycle`.

Лист содержит одну строку на каждый завершенный lifecycle ZigZag-ST filter
FSM для периода `100%`.

Lifecycle определяется строго по diagnostics FSM:
- `active[t] = trade_filter_state[t] != "OFF"`;
- цикл начинается на первом active-баре после `OFF`, либо на `0`, если
  `active[0] == True`;
- цикл заканчивается на последнем active-баре перед возвратом в `OFF`;
- trailing active segment без возврата в `OFF` не попадает в лист.

Торговую логику SuperTrend, ZigZag-ST FSM, execution model, позиции,
сделки, backtest, config loader/schema и YAML/config контракт менять нельзя.

Новый YAML/config ключ добавлять нельзя.


2. Non-goals и ограничения
--------------------------
1. Лист `cycle` не влияет на торговые решения, позиции, сделки, metrics,
   существующие diagnostic arrays или backtest results.
2. Лист `cycle` не привязан к `diagnostics.export_state_columns` или
   `diagnostics.export_trigger_columns`. Это отдельный export artifact.
3. При enabled filter и наличии `pr_100.filter_diagnostics` лист `cycle`
   пишется даже если `FilterDiagnostics_100` и `ZigZag_Trigger_Events`
   отключены diagnostic flags.
4. При disabled filter лист `cycle` отсутствует, а builder не вызывается.
5. `_build_cycle_sheet_df(...)` запрещено вызывать вне ветки:

   `filter_enabled and pr_100 is not None and pr_100.filter_diagnostics is not None`

6. ZigZag pivot logic нельзя реализовывать вручную в exporter. Confirmed legs
   и per-bar confirmed events должны приходить только из разрешенных helper'ов
   в `core/zigzag_st_filter.py`.


3. Когда писать лист
--------------------
Лист `cycle` пишется только если одновременно выполняются условия:

1. `trade_filter_config is not None`;
2. `trade_filter_config.enabled is True`;
3. найден `PeriodResult` с `period_label == "100%"`;
4. `pr_100.filter_diagnostics is not None`.

Если условия 1-4 выполнены, но входы деградированы, лист все равно пишется,
но может содержать только заголовки:
- `df` отсутствует;
- `df` не содержит `close`, `high`, `low`;
- в diagnostics отсутствует обязательный ключ;
- aligned prefix имеет `n == 0`;
- `zigzag_reversal_threshold` невалиден;
- `local_window` невалиден.

Если filter disabled или `pr_100.filter_diagnostics is None`, лист `cycle` не
пишется вообще, а `_build_cycle_sheet_df(...)` не вызывается.


4. Контракт листа
-----------------
Имя листа: `cycle`.

Колонки обязательны, порядок стабильный:

1. `Начало цикла`
   Дата/время `start_bar` из `df.iloc[:n].index`, нормализованное через
   `_excel_safe_datetime_value(...)`.

2. `Конец цикла`
   Дата/время `end_bar` из `df.iloc[:n].index`. Это последний non-OFF бар
   цикла. `off_bar = end_bar + 1` в цикл не входит.

3. `Направление цикла`
   Один символ: `+` или ASCII `-`. Если направление нельзя надежно вывести,
   писать пустое значение, строку цикла не пропускать.

   Правила по `trade_filter_trigger_source[start_bar]`:
   - `candidate_threshold`: брать `candidate_leg_direction[start_bar]`.
   - `both`: брать `candidate_leg_direction[start_bar]`.
     Это post-pivot A-direction нового candidate на confirm-bar, а не
     direction только что закрытой confirmed leg.
   - `confirmed_median`: брать direction confirmed ZigZag leg с
     `confirm_bar == start_bar` из `compute_confirmed_legs_reset_aware(...)`.
     Если такой leg не найден, писать пустое направление.
   - `none` и любые unknown values: писать пустое направление.

   `st_flip_dir` не использовать как fallback.

4. `Баров в цикле`
   `end_bar - start_bar + 1`.

5. `Ног ZigZag в цикле`
   Количество confirmed legs, у которых `confirm_bar` попадает в
   `[start_bar, end_bar]`.

6. `Медиана ног`
   `np.median(...)` по heights confirmed legs цикла. Scale: fraction.
   При нуле ног: `NaN`.

7. `Ног выше порога триггера кандидата`
   Количество ног цикла, где `leg_height_pct > threshold`.
   Сравнение строгое: `height == threshold` не считается.
   При `legs_count == 0` и валидном threshold: `0`.
   Если threshold не materialize: `NaN`.

8. `Размер цикла, %`
   `(close_end - close_start) / close_start * 100`.
   Если `close_start` не finite или `close_start <= 0`: `NaN`.

9. `ID цикла`
   Порядковый номер завершенного цикла на листе, начиная с 1.
   Назначается после отбрасывания незавершенного хвоста.

10. `Start bar index`
    Локальный индекс `start_bar` внутри aligned prefix `0..n-1`.

11. `End bar index`
    Локальный индекс `end_bar` внутри aligned prefix `0..n-1`.

12. `Цена начала`
    `close[start_bar]`, если OHLC interval valid; иначе `NaN`.

13. `Цена конца`
    `close[end_bar]`, если OHLC interval valid; иначе `NaN`.

14. `High цикла`
    `max(high[start_bar : end_bar + 1])`, если OHLC interval valid; иначе
    `NaN`.

15. `Low цикла`
    `min(low[start_bar : end_bar + 1])`, если OHLC interval valid; иначе
    `NaN`.

16. `Макс. движение по циклу, %`
    Для direction `+`:
    `(high_cycle - close_start) / close_start * 100`.

    Для direction `-`:
    `(close_start - low_cycle) / close_start * 100`.

    Если direction пустой, OHLC interval invalid или `close_start <= 0`:
    `NaN`.

17. `Макс. просадка внутри цикла, %`
    Для direction `+`:
    `(close_start - low_cycle) / close_start * 100`.

    Для direction `-`:
    `(high_cycle - close_start) / close_start * 100`.

    Если direction пустой, OHLC interval invalid или `close_start <= 0`:
    `NaN`.

18. `Причина завершения`
    Определяется по `off_bar = end_bar + 1`:
    - если `off_bar < n` и `daily_reset_event[off_bar] == 1`, писать
      `daily_reset`;
    - иначе писать `FSM_OFF`.

    `filter_block_reason[off_bar]` не использовать: это per-decision-bar
    причина блока, а не причина завершения lifecycle.

19. `Макс. высота ноги`
    Максимальная height confirmed leg в цикле. Scale: fraction.
    При нуле ног: `NaN`.

20. `Доля ног выше порога, %`
    `legs_above_threshold / legs_count * 100`.
    При нуле ног: `NaN`.
    Если threshold не materialize: `NaN`.

21. `Сделок в цикле`
    Количество сделок 100% периода, вход которых относится к циклу по
    close-decision шкале diagnostics.

    Mapping:
    `entry_signal_idx = max(int(entry_index) - 1, 0)`.

    Сделка входит в цикл, если:
    `start_bar <= entry_signal_idx <= end_bar`.

    Использовать raw `pr_100.trades_df`.
    Если `trades_df` отсутствует, пустой, не содержит `entry_index` или
    конкретный `entry_index` is NaN, такие сделки пропускать.

22. `% сделок с положительным фин результатом в цикле`
    `positive_trades / trades_in_cycle * 100`.

    `positive_trades` - сделки цикла с `net_pnl_pct > 0`.
    `net_pnl_pct` уже в percent scale 0..100, повторно умножать на 100 нельзя.

    Если `trades_in_cycle == 0`: `NaN`.
    Если `net_pnl_pct` отсутствует или хотя бы у одной сделки цикла
    non-finite: `NaN` для всей метрики этого цикла.


5. Единицы измерения
--------------------
1. ZigZag leg heights (`Медиана ног`, `Макс. высота ноги`) - fraction scale.
   Пример: `0.0123` означает 1.23%.
2. Все колонки, в названии которых есть `%`, пишутся в percent scale 0..100.
3. `candidate_trigger_threshold` из diagnostics - fraction scalar.
   Не умножать на 100 перед сравнением с leg heights.
4. `net_pnl_pct` из `trades_df` уже percent scale 0..100.
   Не умножать повторно.


6. Источники данных
-------------------
Основные данные приходят в `export_tester_results(...)`.

6.1. `PeriodResult` 100%

Использовать только:

`pr_100 = next((pr for pr in period_results if pr.period_label == "100%"), None)`

Из него нужны:
- `pr_100.filter_diagnostics`;
- `pr_100.filter_diagnostics_summary` только если понадобится существующим
  optional sheets;
- raw `pr_100.trades_df`.

6.2. `filter_diagnostics`

Обязательные ключи:
- `trade_filter_state`;
- `trade_filter_trigger_source`;
- `candidate_trigger_threshold`;
- `zigzag_reversal_threshold`;
- `local_window`;
- `daily_reset_event`;
- `candidate_leg_direction`.

Если хотя бы один обязательный ключ отсутствует, `_build_cycle_sheet_df(...)`
возвращает пустой DataFrame с `CYCLE_SHEET_COLUMNS`.

6.3. Scalar materialize

Все scalar values берутся только из diagnostics arrays. YAML/config не читать.

Общий helper для float scalar (`candidate_trigger_threshold`,
`zigzag_reversal_threshold`):
1. Взять aligned slice `arr[:n]`.
2. Найти первое finite значение.
3. Проверить, что все finite значения в aligned slice строго равны первому
   finite значению через `==`, без tolerance.
4. Вернуть scalar.
5. Если finite значений нет или finite значения неравны, scalar считается
   not materialized.

`candidate_trigger_threshold`:
- если не materialize, строку цикла сохранять;
- threshold-dependent поля писать `NaN`.

`zigzag_reversal_threshold`:
- если не materialize, non-finite, `<= 0` или `>= 1`, вернуть пустой DataFrame
  с заголовками;
- `compute_zigzag_per_bar(...)` и `compute_confirmed_legs_reset_aware(...)`
  в этом случае не вызывать.

`local_window`:
1. Взять aligned slice `local_window[:n]`.
2. Найти первое finite значение.
3. Проверить, что все finite значения строго равны первому finite значению
   через `==`, без tolerance.
4. Проверить, что scalar является integer-like значением:
   `float(value).is_integer()` и значение не является bool.
5. Привести к `int`.
6. Проверить `>= 1`.

Если любая проверка `local_window` не прошла, вернуть пустой DataFrame с
заголовками. Нули не игнорировать и не искать "первое ненулевое" значение.
Массивы вроде `[0, 20, 20]` считаются drift/invalid, а не валидным scalar.

6.4. `df`

`df` должен содержать:
- datetime-like индекс;
- `close`;
- `high`;
- `low`.

Если `df` отсутствует или не содержит нужные OHLC колонки, builder возвращает
пустой DataFrame с заголовками.

Поведение при несортированном или дублирующемся индексе вне контракта. В
штатном tester path индекс строго возрастающий.

Datetime values нормализовать через `_excel_safe_datetime_value(...)`.

6.5. Confirmed ZigZag legs

Ручная реализация ZigZag запрещена.

Добавить публичный helper в `donor/supertrend_optimizer/core/zigzag_st_filter.py`:

```
compute_confirmed_legs_reset_aware(
    close: np.ndarray,
    reversal_threshold: float,
    *,
    daily_reset_event: np.ndarray,
) -> List[ConfirmedLeg]
```

Контракт helper'а:
- экспортируется через `__all__`;
- использует тот же validation path, что и `compute_zigzag_per_bar(...)`,
  через `_validate_close_and_threshold(..., fn_name=...)`;
- при `close is None`, empty input или коротком/непригодном input возвращает
  `[]` так же, как существующие close-only helpers;
- invalid `reversal_threshold` является `ConfigError`, но exporter обязан
  отфильтровать такие случаи заранее и не вызывать helper;
- реализация только делегирует в `_run_close_only_zigzag_pass(...)` с
  `daily_reset_event=daily_reset_event` и возвращает `pass_result.legs`;
- pivot logic не дублируется.

Для колонок 5, 6, 7, 19, 20 использовать:

```
compute_zigzag_per_bar(
    close[:n],
    reversal_threshold,
    local_window,
    daily_reset_event=daily_reset_event[:n],
)
```

Membership confirmed legs:
- `per_bar.confirm_event[t] == 1` означает confirmed leg на баре `t`;
- height этой ноги брать из `per_bar.last_confirmed_leg_height_pct[t]`;
- учитывать только `t in [start_bar, end_bar]`.

Для direction при `confirmed_median` использовать список `ConfirmedLeg` из
`compute_confirmed_legs_reset_aware(...)` и искать
`leg.confirm_bar == start_bar`.

6.6. Trades 100%

Использовать только raw `pr_100.trades_df`.

Нельзя читать данные с уже записанного Excel-листа `Trades_100`.

Для count нужна колонка `entry_index`.
Для positive percent нужны `entry_index` и `net_pnl_pct`.


7. Alignment и границы массивов
-------------------------------
Перед построением циклов все входы приводятся к одному aligned prefix:

```
n = min(
    len(df),
    len(trade_filter_state),
    len(close),
    len(high),
    len(low),
    len(trade_filter_trigger_source),
    len(candidate_leg_direction),
    len(candidate_trigger_threshold),
    len(zigzag_reversal_threshold),
    len(local_window),
    len(daily_reset_event),
)
```

Все массивы и `df` срезаются до `[:n]`.

Если `n == 0`, вернуть пустой DataFrame с заголовками.

`Start bar index` и `End bar index` всегда локальные индексы внутри aligned
prefix. Они совпадают с `Bar Index` на `FilterDiagnostics_100` только если
`len(df) == n`.


8. Семантика цикла и OPEN_TO_OPEN
---------------------------------
Лист `cycle` агрегирует по close-decision шкале diagnostics:

```
active[t] = trade_filter_state[t] != "OFF"
```

Правила:
- start: `active` меняется `False -> True`;
- если `active[0] == True`, start первого segment равен `0`;
- end: последний `True` перед следующим `False`;
- `off_bar = end_bar + 1`;
- trailing active segment без возврата в `OFF` пропускается;
- `start_bar` и `end_bar` не сдвигать.

`off_bar` не входит в:
- `Баров в цикле`;
- OHLC interval;
- percent metrics;
- confirmed legs membership interval;
- trade membership interval.


9. Non-finite OHLC внутри цикла
-------------------------------
Exporter не должен падать при non-finite `close/high/low` внутри detected
cycle interval.

Для каждого цикла отдельно:
- если любой `close/high/low` в `[start_bar, end_bar]` non-finite, считать
  OHLC interval invalid;
- `Цена начала`, `Цена конца`, `High цикла`, `Low цикла` писать `NaN`;
- `Размер цикла, %`, `Макс. движение по циклу, %`,
  `Макс. просадка внутри цикла, %` писать `NaN`;
- строку цикла сохранять;
- leg metrics и trade metrics считать по своим источникам, если они доступны.

Это правило не отменяет scalar guards. Если `zigzag_reversal_threshold` или
`local_window` invalid, builder возвращает пустой DataFrame до вызова ZigZag
helpers.


10. Алгоритм `_build_cycle_sheet_df(...)`
-----------------------------------------
Сигнатура:

```
_build_cycle_sheet_df(
    filter_diagnostics: Dict[str, np.ndarray],
    df: Optional[pd.DataFrame],
    trades_df: Optional[pd.DataFrame] = None,
) -> pd.DataFrame
```

Алгоритм:
1. Создать пустой DataFrame с `CYCLE_SHEET_COLUMNS`.
2. Проверить `filter_diagnostics`, `df`, OHLC колонки и обязательные keys.
   При проблеме вернуть пустой DataFrame.
3. Посчитать `n` по aligned prefix. Если `n == 0`, вернуть пустой DataFrame.
4. Подготовить aligned arrays длины `n`.
5. Materialize `candidate_trigger_threshold`.
   Если не materialize, threshold-dependent fields будут `NaN`, но строки
   циклов сохраняются.
6. Materialize и pre-validate `zigzag_reversal_threshold`.
   Если invalid, вернуть пустой DataFrame.
7. Materialize и pre-validate `local_window`.
   Если invalid, вернуть пустой DataFrame.
8. Вызвать `compute_zigzag_per_bar(...)`.
9. Вызвать `compute_confirmed_legs_reset_aware(...)`.
10. Подготовить trade membership helper из raw `trades_df`.
11. Найти completed active segments по `trade_filter_state != "OFF"`.
12. Пропустить trailing active segment.
13. Для каждого completed segment рассчитать все 22 колонки.
14. Строки отсортировать по `Start bar index`.
15. Назначить `ID цикла` после фильтрации хвоста, начиная с 1.
16. Вернуть DataFrame с ровно 22 колонками.


11. Excel writer и порядок листов
----------------------------------
Добавить в `excel_tester.py`:
- `CYCLE_SHEET_NAME = "cycle"`;
- `CYCLE_SHEET_COLUMNS = (...)`;
- `_build_cycle_sheet_df(...)`;
- `_write_cycle_sheet(writer, cycle_df)`.

Форматирование:
- писать через `cycle_df.to_excel(..., sheet_name="cycle", index=False)`;
- включать autofilter всегда, включая пустой лист с одними заголовками;
- для `Начало цикла` и `Конец цикла` применять формат
  `YYYY-MM-DD HH:MM:SS`;
- форматирование остальных листов не менять.

Подключение в `export_tester_results(...)`:
1. Вычислить `filter_enabled`.
2. Найти `pr_100` и `trades_100_raw`.
3. Записать существующие optional filter sheets:
   - `FilterDiagnostics_100`, если `export_state_columns=True`;
   - `ZigZag_Trigger_Events`, если `export_trigger_columns=True`;
   - `filters_summary`, если `export_state_columns=True`.
4. После них, строго внутри ветки
   `filter_enabled and pr_100 is not None and pr_100.filter_diagnostics is not None`,
   построить `cycle_df` и записать `cycle`.
5. `cycle` пишется последним среди filter-diagnostics-dependent листов.
6. `cycle` не зависит от `export_state_columns` и `export_trigger_columns`.

13. Тестовый план
-----------------
Тесты делятся на две группы:
- обязательный минимальный набор перед реализацией;
- расширенная acceptance-карта, которую нужно закрыть до финальной приемки.

13.1. Минимальный обязательный набор

Этот набор пишется первым как failing tests и должен пройти после
реализации:

1. Headers:
   `_build_cycle_sheet_df(...)` возвращает ровно `CYCLE_SHEET_COLUMNS`
   в стабильном порядке, 22 колонки.

2. Completed cycles и unfinished tail:
   synthetic `trade_filter_state`: `OFF`, active, `OFF`, trailing active.
   Ожидание: одна строка, trailing active пропущен, `ID цикла == 1`.

3. Post-close FSM boundary:
   segment `[start=2, end=6]`, `off_bar=7`.
   Ожидание: OHLC/legs/trades используют только 2..6, reason читается по 7,
   start/end не сдвигаются.

4. Scalar guards:
   invalid `zigzag_reversal_threshold`, invalid `local_window`, `n == 0`,
   missing `daily_reset_event` возвращают пустой DataFrame с заголовками.
   Builder не вызывает ZigZag helpers при invalid reversal/window.

5. `local_window` materialize:
   uniform integer-like values валидны; `[0, 20, 20]`, non-integer,
   non-uniform finite values invalid.

6. Direction matrix:
   `candidate_threshold`, `both`, `confirmed_median`, `none`, unknown.
   `both` берет `candidate_leg_direction[start_bar]`.
   `st_flip_dir` не используется.

7. Trades mapping:
   `entry_signal_idx = max(entry_index - 1, 0)`;
   `off_bar` не входит;
   NaN `entry_index` пропускается.

8. Positive trades percent:
   `net_pnl_pct > 0` считается positive;
   `0` не positive;
   `net_pnl_pct` не умножается на 100;
   non-finite `net_pnl_pct` внутри цикла дает `NaN` для win percent.

9. Enabled export:
   workbook содержит `cycle`, headers корректны, autofilter включен.

10. Enabled export with diagnostic flags off:
    `cycle` присутствует при `pr_100.filter_diagnostics` even если
    optional diagnostic sheets отключены.

11. Disabled no-builder:
    при disabled filter `cycle` отсутствует и `_build_cycle_sheet_df(...)`
    не вызывается через monkeypatch/spy.

12. Helper consistency rail:
    `compute_zigzag_per_bar(...)` и
    `compute_confirmed_legs_reset_aware(...)` дают одинаковые confirm bars
    и heights на одинаковых входах с `daily_reset_event`.

13.2. Расширенная acceptance-карта

1. Active starts at zero:
   если `active[0] == True` и далее есть OFF, `start_bar == 0`.

2. Confirmed legs membership:
   учитываются only confirm bars в `[start_bar, end_bar]`;
   `off_bar` не учитывается.

3. Threshold strictness:
   `height == threshold` не считается;
   `height > threshold` считается;
   non-uniform threshold => колонки 7 и 20 `NaN`, строка сохраняется;
   `legs_count == 0` => column 7 `0`, column 20 `NaN`.

4. Percent metrics:
   вручную проверить `Размер цикла, %`,
   `Макс. движение по циклу, %`, `Макс. просадка внутри цикла, %`
   для direction `+`, direction `-`, empty direction, `close_start <= 0`.

5. Empty legs:
   count `0`, median `NaN`, max `NaN`, above-threshold `0` при валидном
   threshold, share `NaN`.

6. Median with even leg count:
   heights `[0.01, 0.03]` => median `0.02`.

7. Alignment mismatch:
   mismatched lengths не падают, используется aligned prefix,
   datetime берется из `df.iloc[:n]`.

8. Daily reset reason:
   `daily_reset_event[off_bar] == 1` => `daily_reset`;
   иначе `FSM_OFF`.

9. Adjacent cycles:
   два цикла с одним OFF между ними не сливаются, IDs монотонны.

10. B-rescue / `both`:
    direction cycle == `candidate_leg_direction[start_bar]` и отличается
    от direction confirmed leg с `confirm_bar == start_bar`.

11. Helper reuse spy:
    `_build_cycle_sheet_df(...)` вызывает `compute_zigzag_per_bar(...)` и
    `compute_confirmed_legs_reset_aware(...)`.

12. Enabled export without `daily_reset_event`:
    workbook содержит `cycle`, лист пустой с заголовками, export не падает.

13. Existing WP-T7 sheets:
    не сломаны `Signals`, `Trades_100`, `FilterDiagnostics_100`,
    `ZigZag_Trigger_Events`, `filters_summary`, `false start` и порядок
    существующих листов.

14. Reset-aware confirmed legs:
    на короткой серии с `daily_reset_event` confirm bars и heights совпадают
    между per-bar API и public wrapper.

15. Non-finite OHLC inside cycle:
    OHLC и percent metrics `NaN`, строка сохраняется, exporter не падает.

16. Daily reset event inside cycle:
    reset внутри active segment не завершает cycle сам по себе; цикл следует
    diagnostics FSM, а ZigZag helpers обрабатывают reset-aware legs.

17. Autofilter on empty cycle sheet:
    `wb["cycle"].auto_filter.ref` не пустой.

18. Enabled + `filter_diagnostics is None`:
    `cycle` не пишется, builder не вызывается, exporter не падает.

15. Риски и митигации
----------------------
1. Смешивание FSM boundary и trade execution boundary.
   Митигация: close-decision scale зафиксирована в §8 и тестируется boundary
   cases.

2. Неверное direction для `both`.
   Митигация: direction берется из `candidate_leg_direction[start_bar]`;
   отдельный тест проверяет отличие от только что закрытой confirmed leg.

3. Дублирование ZigZag logic.
   Митигация: public wrapper над `_run_close_only_zigzag_pass(...)`;
   exporter не вызывает private helper напрямую.

4. Disabled baseline regression.
   Митигация: `cycle` пишется только enabled + diagnostics present;
   disabled no-builder test обязателен.

5. Percent/fraction confusion.
   Митигация: §5 фиксирует scale каждого поля; tests проверяют cell values.

6. Mismatched lengths.
   Митигация: aligned prefix rule в §7.

7. Export flag matrix drift.
   Митигация: `cycle` независим от diagnostic flags; flags-off export test
   обязателен.

8. Helper drift между per-bar API и public leg-list wrapper.
   Митигация: consistency rail test; оба API используют один close-only pass.

9. Invalid scalar вызывает `ConfigError` в ZigZag helper.
   Митигация: exporter pre-validates `zigzag_reversal_threshold` и
   `local_window` до вызова helper'ов.


16. Критерии приемки
--------------------
Реализация считается готовой, если:

1. При enabled ZigZag-ST filter и доступных diagnostics в XLSX есть лист
   `cycle`.
2. При disabled filter лист `cycle` отсутствует, builder не вызывается,
   disabled workbook baseline не меняется.
3. Лист содержит ровно 22 обязательные колонки в стабильном порядке.
4. Одна строка соответствует одному завершенному close-decision FSM
   lifecycle.
5. Незавершенный lifecycle в конце ряда не попадает в лист.
6. OFF-transition bar не входит в OHLC/percent/leg/trade membership interval.
7. Direction определяется по trigger source; unknown/none не приводят к
   падению; `st_flip_dir` не используется.
8. Leg metrics согласованы с confirmed legs по confirm-bar rule.
9. Trade metrics согласованы с raw `pr_100.trades_df` по
   `entry_signal_idx = max(entry_index - 1, 0)`.
10. Threshold берется из materialized diagnostics scalar.
11. Invalid threshold-dependent scalar сохраняет rows, но пишет `NaN` в
    threshold-dependent fields.
12. Invalid `zigzag_reversal_threshold`, invalid `local_window`, `n == 0`,
    missing required diagnostics или missing OHLC дают пустой лист с
    заголовками и не роняют exporter.
13. Все `%` колонки в percent scale 0..100; leg-height колонки в fraction.
14. Existing tester XLSX sheets и WP-T7 behavior не ломаются.
15. `cycle` пишется последним среди filter-diagnostics-dependent листов.
16. Autofilter присутствует даже на пустом листе `cycle`.
17. `both` берет post-pivot A-direction из
    `candidate_leg_direction[start_bar]`.
