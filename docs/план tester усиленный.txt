Аудит и усиленный план реализации ветки tester в проекте 3.1_wf_grid
=====================================================================

Дата актуализации: 2026-04-24

Статус: GO WITH CHANGES.

Главное правило
---------------

Реализация tester должна быть additive:

- копировать только новые директории cli/, testing/, io/;
- добавить только две недостающие функции в существующие файлы;
- не перезаписывать общий движок supertrend_optimizer из donor TESTER;
- не копировать data.csv из donor TESTER;
- не копировать donor TESTER/supertrend_optimizer/__init__.py, если его не было в текущем donor.

Запрещено выполнять команды вида:

    Copy-Item -Recurse "donor TESTER\supertrend_optimizer\*" "donor\supertrend_optimizer\"

Такой wildcard-copy перезапишет core/, data/, engine/, utils/ и может вернуть look-ahead
семантику close_to_close в WF-grid.


Почему нельзя массово перезаписывать donor
------------------------------------------

Одноименные файлы в donor TESTER и текущем donor/supertrend_optimizer отличаются.
Текущая версия donor местами строже и безопаснее:

- utils/enums.py содержит только ExecutionModel.OPEN_TO_OPEN и комментарий о том,
  что CLOSE_TO_CLOSE удален из-за look-ahead bias.
- engine/run.py явно бросает ValueError, если execution_model != OPEN_TO_OPEN.
- utils/warmup.py содержит валидацию atr_period_range через ConfigError.
- data/timeframe.py содержит более строгую обработку annualization_factor /
  periods_per_year, unsorted/tz/NaT индексов и suppress_warnings.

Итог: TESTER-файлы core/backtest.py, core/trades.py, engine/run.py, utils/enums.py,
data/timeframe.py и utils/warmup.py нельзя заменять целиком.


Блокеры перед реализацией
-------------------------

B1. io/__init__.py

В donor TESTER/supertrend_optimizer/io/ нет __init__.py. Импорты могут работать как
namespace-package, но для согласованности нужно создать пустой файл:

    donor/supertrend_optimizer/io/__init__.py

B2. close_to_close в cli/tester.py

После копирования cli/tester.py нужно:

- убрать "close_to_close" из choices у аргумента --execution-model;
- после чтения execution_model из YAML явно отвергать любое значение кроме None
  и "open_to_open" через ConfigError.

Цель: пользователь должен получить понятную ошибку, а не traceback на
ExecutionModel("close_to_close").

B3. close_to_close в run_batch_tester.py

Перед созданием ExecutionModel в run_batch_tester.py нужно явно проверить:

- None -> использовать ExecutionModel.OPEN_TO_OPEN;
- "open_to_open" -> разрешить;
- любое другое значение -> ConfigError с понятным текстом.

B4. coerce_annualization_config_value

Рекомендуемый контракт: минимальный, только для tester CLI.

Функция нужна физически, потому что cli/tester.py импортирует ее напрямую.
Но она не должна становиться общей точкой annualization для WF-grid.

Реализация:

- bool запрещен;
- int разрешен;
- float разрешен только если это целое конечное число;
- str "auto" разрешен;
- str с целым числом разрешен;
- прочее -> ValueError.

В docstring указать: "for tester CLI only". Для общей логики WF-grid продолжает
использоваться существующая resolve_periods_per_year_from_config().

B5. Manifest и backup

В проекте нет .git, поэтому до правок обязательно:

- создать backup_pre_tester_<ts>/;
- сохранить zip donor/supertrend_optimizer/;
- сохранить копии run.py, config.yaml, data.csv;
- создать pre_tester_manifest.txt с SHA256 для:
  - donor/supertrend_optimizer/core/backtest.py
  - donor/supertrend_optimizer/core/trades.py
  - donor/supertrend_optimizer/engine/run.py
  - donor/supertrend_optimizer/utils/enums.py
  - donor/supertrend_optimizer/data/timeframe.py
  - donor/supertrend_optimizer/utils/warmup.py
  - run.py
  - config.yaml
  - data.csv

После правок создать post_tester_manifest.txt и проверить, что запрещенные к
перезаписи файлы core/backtest.py, core/trades.py, engine/run.py, utils/enums.py
остались побитово одинаковыми.

Важно: data/timeframe.py и utils/warmup.py НЕ должны совпадать по hash после
реализации, потому что в них сознательно добавляются новые функции. Для этих двух
файлов контроль другой: diff должен содержать только ожидаемый hunk с новой функцией
и, при необходимости, минимальным импортом. Существующие функции и их сигнатуры не
должны измениться.

S6. Negative CLI close_to_close:

PowerShell:

    $env:PYTHONPATH='donor'
    python -m supertrend_optimizer.cli.tester --csv data.csv --atr 14 --mult 3.0 --mode long --out x.xlsx --execution-model close_to_close

cmd.exe:

    set PYTHONPATH=donor
    python -m supertrend_optimizer.cli.tester --csv data.csv --atr 14 --mult 3.0 --mode long --out x.xlsx --execution-model close_to_close

Ожидается:

- exit code != 0;
- argparse отклоняет аргумент;
- нет traceback.

Без PYTHONPATH=donor этот тест из корня проекта может завершиться ModuleNotFoundError,
и это не проверит argparse. Поэтому PYTHONPATH здесь обязателен.

Важно: на Windows не использовать /tmp/x.xlsx; использовать локальный путь x.xlsx
или .\tmp\x.xlsx.
