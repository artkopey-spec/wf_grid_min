# Implementation Plan: Volume Filter v5

This document is the final standalone implementation plan for the `volume`
trade filter based on `docs/TZ_volume_filter_v3.txt`.

It replaces the implementation strategy from
`docs/volume_filter_implementation_plan_tz_v4.txt` but does not modify or
delete that historical file.

v5 is not a patch-list over v4. It is a complete implementation plan intended
to be executable as a sequence of small, reviewable PRs.

Core corrections versus v4:

- `trade_filter.enabled` is only the root switch, not a ZigZag switch.
- `zigzag.enabled` and `volume.enabled` are independent subfilter switches.
- Runtime helpers use strict post-validation semantics, not `bool(...)`
  coercion of raw config values.
- Resolver logic and runtime helper logic are explicitly separated.
- `trade_filter.type` is treated as a legacy ZigZag marker, not a required
  field for standalone volume.
- The W2.C legality change is an explicit breaking config migration.
- `filter_config_snapshot` is an independent end-to-end transport field.
- Trade-level diagnostics become subfilter-aware and are not owned by
  `zigzag_st_filter.py`.
- Golden and performance tests are placed in pytest-discovered paths.
- Export schemas are explicitly split by active subfilter groups.
- The per-bar diagnostics retention flag is named to avoid conflict with
  `trade_filter.diagnostics`.

## 0. Architecture and Runtime Contract

### 0.1 Runtime Modes

The implementation supports exactly four runtime modes.

1. **baseline**

   Conditions:

   ```text
   trade_filter absent
   OR trade_filter.enabled=false
   ```

   Runtime:

   ```text
   run_backtest_fast -> generate_positions(...)
   ```

   Invariant:

   ```text
   positions / returns / equity / trades / export schemas are bit-identical
   to the pre-volume baseline.
   ```

2. **zigzag-only**

   Conditions:

   ```text
   trade_filter.enabled=true
   zigzag.enabled=true
   volume.enabled=false
   ```

   Runtime:

   ```text
   build_zigzag_global_stats(...)
   run_backtest_fast -> zigzag_st_filter.apply(...)
   ```

   Invariant:

   ```text
   Bit-identical to the current ZigZag runtime before volume work.
   This is not no-filter baseline parity.
   ```

3. **standalone-volume**

   Conditions:

   ```text
   trade_filter.enabled=true
   zigzag.enabled=false
   volume.enabled=true
   ```

   Runtime:

   ```text
   build_volume_global_metrics(...)
   run_backtest_fast -> volume_only_filter.apply(...)
   ```

   Invariant:

   ```text
   ZigZag stats are not built.
   ZigZag apply is not called.
   Volume-only FSM owns positions and diagnostics.
   ```

4. **zigzag+volume**

   Conditions:

   ```text
   trade_filter.enabled=true
   zigzag.enabled=true
   volume.enabled=true
   ```

   Runtime:

   ```text
   build_zigzag_global_stats(...)
   build_volume_global_metrics(...)
   run_backtest_fast -> zigzag_st_filter.apply(..., volume_runtime=...)
   ```

   Invariant:

   ```text
   ZigZag owns lifecycle.
   Volume gates lifecycle starts from OFF only.
   Volume does not close, reset or otherwise mutate an active ZigZag lifecycle.
   Volume does not gate WAIT_FIRST_ST_FLIP -> ACTIVE_*.
   ```

### 0.2 Grid Search Invariants

- Volume parameters do not participate in grid enumeration.
- Volume parameters do not enter `grid_point_id`.
- Volume parameters are identical for all grid candidates and WF steps in one
  run.
- The `grid_point_id` builder is not changed.
- Cross-volume comparison is performed by running the pipeline multiple times
  with different config files.

### 0.3 `trade_filter.type` Contract

`trade_filter.type` is retained only as a legacy ZigZag marker.

Rules:

- Existing ZigZag YAML may continue to use:

  ```text
  trade_filter.type: zigzag_st_mode
  ```

- `type=zigzag_st_mode` implies a ZigZag legacy marker when
  `zigzag.enabled` is absent.
- Standalone volume does not require `trade_filter.type`.
- `trade_filter.enabled=true` no longer means `trade_filter.type` is required.
- If `type=zigzag_st_mode` and `zigzag.enabled=false`, the config is rejected
  unless an explicit migration exception is documented and tested.
- No new `trade_filter.type` literal is introduced in v5.

Reason:

```text
The runtime mode is determined by subfilter flags, not by a root type enum.
```

### 0.4 Strict Runtime Helpers

In `donor/supertrend_optimizer/core/trade_filter_config.py` add:

```text
is_trade_filter_enabled(tf) -> bool
is_zigzag_enabled(tf) -> bool
is_volume_enabled(tf) -> bool
```

Runtime helpers are post-validation helpers. They must not coerce malformed raw
YAML values with `bool(...)`.

Final semantics:

```text
is_trade_filter_enabled(tf):
  tf is not None
  AND tf.enabled is True

is_zigzag_enabled(tf):
  is_trade_filter_enabled(tf)
  AND getattr(tf, "zigzag", None) is not None
  AND getattr(tf.zigzag, "enabled", None) is True

is_volume_enabled(tf):
  is_trade_filter_enabled(tf)
  AND getattr(tf, "volume", None) is not None
  AND getattr(tf.volume, "enabled", None) is True
```

Compatibility window rule:

- Before W2.A materializes `zigzag.enabled`, `is_zigzag_enabled` may use
  `getattr(tf.zigzag, "enabled", True)` only for legacy objects that have
  already passed validation or are hand-built test fixtures.
- Resolvers must not use these runtime helpers to classify malformed raw
  values.

Resolver functions must preserve malformed raw values so Phase 4 validation can
emit canonical `ConfigError`.

### 0.5 Diagnostics Contract

`filter_diagnostics` is a per-bar export-facing diagnostics container.

Mandatory keys for every enabled filter:

```text
trade_filter_state
filter_block_reason
```

Optional ZigZag keys are emitted only when `is_zigzag_enabled(tf)=true`.
Optional volume keys are emitted only when `is_volume_enabled(tf)=true`.

Standalone volume must not emit ZigZag-only optional keys.
ZigZag-only must not emit volume optional keys.
ZigZag+volume may emit both groups.

Consumers must be subfilter-aware:

- Generic consumers may read only mandatory keys.
- ZigZag consumers must guard ZigZag optional keys with `is_zigzag_enabled`.
- Volume consumers must guard volume optional keys with `is_volume_enabled`.
- Summary and export layers must not infer active mode from
  `trade_filter.enabled`.

### 0.6 Runtime Codes vs Diagnostic Strings

`VolumeRuntime` is the compact transport/runtime layer. It stores int-coded
categorical arrays under TZ-aligned names without `_code` suffixes:

```text
volume_regime                    np.ndarray dtype=int8
volume_condition_block_reason    np.ndarray dtype=int8
volume_initial_direction         np.ndarray dtype=int8
```

`filter_diagnostics` is the user-facing layer. It stores strings under the same
logical keys:

```text
filter_diagnostics["volume_regime"]                    dtype=object
filter_diagnostics["volume_condition_block_reason"]    dtype=object
filter_diagnostics["volume_initial_direction"]         dtype=object
```

The duplicate key names are intentional. The contract is defined by
`(container, key)`:

```text
VolumeRuntime.volume_regime          -> int8 code
filter_diagnostics["volume_regime"]  -> string/object
```

Consumers such as signal events, Excel writers, summary builders and collectors
read strings from `filter_diagnostics`. They never decode int8 codes.

### 0.7 Snapshot Transport Contract

`filter_config_snapshot` is independent from `filter_diagnostics`.

It is transported end-to-end:

```text
VolumeRuntime.filter_config_snapshot
  -> volume_only_filter.apply / zigzag_st_filter.apply result
  -> RawBacktestArtifacts
  -> BacktestResult
  -> PeriodResult / StepResult
  -> exporters / fingerprint
```

Rules:

- Never store snapshot inside `filter_diagnostics`.
- Strip functions may remove per-bar diagnostics but must preserve snapshot.
- If volume is disabled, no volume snapshot fields appear.
- If trade_filter is disabled or absent, `filter_config_snapshot is None`.

### 0.8 Per-Bar Diagnostics Retention Flag

Do not introduce a root `diagnostics` block because it conflicts conceptually
with `trade_filter.diagnostics`.

Add this config field instead:

```text
export.retain_per_bar_filter_diagnostics: bool = False
```

Meaning:

- `False`: per-bar filter diagnostics are stripped before `StepResult` storage.
- `True`: per-bar filter diagnostics are retained and may be exported to
  `WF_FilterDiagnostics`.

The flag affects only `wf_grid`; tester keeps its existing per-run diagnostic
export behavior.

### 0.9 Test Discovery Contract

All mandatory tests must live in pytest-discovered paths.

Current default discovery covers:

```text
wf_grid/tests
donor TESTER/tests
```

Therefore v5 places new default tests under these directories, or updates
`pyproject.toml` explicitly before adding tests elsewhere.

Rules:

- Golden characterization tests must run in the default suite.
- W3 100k performance smoke must run in the default suite.
- Slow performance tests are marked `slow_perf` and skipped by normal commands
  only if the project config explicitly supports that marker.

Goal:


- `donor/supertrend_optimizer/io/excel_tester.py`

- `donor/supertrend_optimizer/cli/tester.py`

  - helper replacement;
  - no new business logic beyond build/forward runtimes.
- `wf_grid/export/xlsx_writer.py`

## W1. Lowercase Columns and Collision Observability

Goal:

```text
Detect lowercase collisions before Volume and volume collapse into one column.
```

This is a data-contract behavior change for `wf_grid`: mixed-case OHLCV becomes
accepted after normalization. It must be documented and tested as such.

### W1.1 Shared Collision Helper

In `donor/supertrend_optimizer/data/validator.py` add:

```text
_check_no_duplicate_lowercase_columns(columns, source, *, mode)
```

`mode`:

- `raise`: raise `DataValidationError`;
- `info`: emit INFO log and return.

Message contains:

```text
duplicate lowercase columns
```

and lists the colliding original names.

### W1.2 wf_grid Fail-Fast

In `wf_grid/pipeline/orchestrator.py`, after CSV load and before lowercasing:

```text
_check_no_duplicate_lowercase_columns(list(df.columns), source, mode="raise")
df.columns = df.columns.str.lower()
validate_ohlc_data(...)
```

### W1.3 Tester Loader Observability

In `donor/supertrend_optimizer/data/loader.py::load_ohlc_csv`, before existing
lowercasing:

```text
_check_no_duplicate_lowercase_columns(list(df.columns), file_path, mode="info")
```

No parser/date/shape semantics change.

### W1.4 Tests

- wf_grid mixed-case OHLCV becomes valid after lowercase normalization.
- wf_grid `Volume` + `volume` raises before overwrite.
- tester loader emits INFO for the same collision and does not raise.
- OHLC-only with disabled volume remains valid.
- ZigZag-only parity survives lowercase normalization.

## W2. Config Loading, Schema and Migration

W2 is split into four PRs:

```text
W2.0   phase pipeline
W2.A   soft zigzag.enabled
W2.B   volume schema
W2.C1  behavior-preserving validator split
W2.C2  explicit breaking legality migration
```

### W2.0 Loader Phase Pipeline

Update `wf_grid/config/loader.py::load_grid_config` and active tester config
path `donor/supertrend_optimizer/cli/tester.py` to follow this order:

1. `_read_yaml(path)`;
2. `_validate_strict_schema(raw)`;
3. collect trade_filter unknown keys if the block exists;
4. `raw_user_keys = collect_raw_user_keys(raw)`;
5. `cfg = _build_config(raw)`;
6. Phase 3 resolvers:
   - `resolve_zigzag_enabled_in_place(cfg.trade_filter, raw_user_keys)`;
   - `resolve_volume_enabled_in_place(cfg.trade_filter, raw_user_keys)`;
7. Phase 4 validation:
   - `_validate_config(cfg, raw_user_keys=raw_user_keys)`;
8. Phase 5 defaults:
   - `resolve_trade_filter_mode_in_place`;
   - `resolve_exit_off_mode_in_place`;
   - `resolve_exit_b_immediate_off_in_place`;
   - `resolve_time_filter_in_place`;
   - `resolve_volume_defaults_in_place`;
9. optional `resolve_periods_per_year`.

Resolver rules:

- They do not raise on malformed bools.
- They preserve malformed raw values for Phase 4 validation.
- They do not call strict runtime helpers on malformed values.

Data validation boundary:

- Config loader never calls `validate_volume_filter_data`.
- Volume data validation runs only after data load, OHLC validation and
  lowercase normalization.

### W2.0 Resolver Truth Table

Add tests and PR documentation for this truth table:

```text
trade_filter absent
  -> root=false, zigzag=false, volume=false, mode=baseline, valid

trade_filter.enabled=false
  -> root=false, zigzag=false, volume=false, mode=baseline, valid

enabled=true + type=zigzag_st_mode + no explicit zigzag.enabled
  -> root=true, zigzag=true, volume=false, mode=zigzag-only, valid

enabled=true + non-enabled zigzag fields + no explicit zigzag.enabled
  -> root=true, zigzag=true, volume=false, mode=zigzag-only, valid

enabled=true + zigzag.enabled=true + volume absent
  -> root=true, zigzag=true, volume=false, mode=zigzag-only, valid

enabled=true + zigzag.enabled=false + volume.enabled=true
  -> root=true, zigzag=false, volume=true, mode=standalone-volume, valid

enabled=true + zigzag.enabled=true + volume.enabled=true
  -> root=true, zigzag=true, volume=true, mode=zigzag+volume, valid

enabled=true + zigzag.enabled=false + volume absent/disabled
  -> invalid: at least one subfilter must be enabled

enabled=true + volume.enabled malformed
  -> invalid in Phase 4, no runtime mode selected

enabled=true + zigzag.enabled malformed
  -> invalid in Phase 4, no runtime mode selected
```

### W2.A Soft `zigzag.enabled`

Add to `TradeFilterZigZagConfig`:

```text
enabled: Optional[bool] = None
```

Add `enabled` to allowed keys under `trade_filter.zigzag`.

Compatibility resolver before W2.C2:

```text
if explicit zigzag.enabled:
    preserve raw value for Phase 4 validation
elif legacy ZigZag marker present:
    set True
elif tf.enabled is true:
    set True
else:
    set False
```

Legacy ZigZag marker:

```text

```

Tests:

- legacy YAML without `zigzag.enabled` resolves true;
- explicit true resolves true;
- explicit false without volume is rejected with migration hint;
- empty `zigzag: {}` + enabled true remains compatible during W2.A;
- ZigZag-only behavior is bit-identical.

### W2.B Volume Schema

Add:

```text
TradeFilterVolumeConfig
TradeFilterConfig.volume: Optional[TradeFilterVolumeConfig] = None
TRADE_FILTER_ALLOWED_KEYS["trade_filter.volume"]
```

Fields:

```text
enabled
mode
short_window
baseline_window
threshold_ratio
regime_low_ratio
regime_high_ratio
direction_lookback_bars
```

Type policy:

- raw values are stored as `object` / `Optional`;
- strict typing happens in validator.

Validation when `volume.enabled=true`:

```text
volume.enabled                 bool
volume.mode                    volume_A | volume_B
volume.short_window            int >= 1
volume.baseline_window         int >= 1 and >= short_window
volume.threshold_ratio         finite number > 0
volume.regime_low_ratio        finite number > 0
volume.regime_high_ratio       finite number > 0 and > regime_low_ratio
volume.direction_lookback_bars int >= 1
```

Required when enabled:

```text
mode
short_window
baseline_window
threshold_ratio
```

Defaults materialized after validation:

```text
regime_low_ratio        -> 0.8
regime_high_ratio       -> 1.2
direction_lookback_bars -> 3
```

Tests:

- valid `volume_A`;
- valid `volume_B`;
- absent volume block;
- `volume.enabled=false`;
- malformed `volume.enabled`;
- invalid mode/window/ratios/lookback;
- unknown key under `trade_filter.volume`;
- defaults materialize only after validation.

### W2.C1 Behavior-Preserving Validator Split

Refactor `validate_trade_filter` without changing behavior.

Extract:

```text
_validate_root_enabled_and_type(...)
_validate_zigzag_block(...)
_validate_lifecycle_block(...)
_validate_triggers_block(...)
_validate_time_filter_block(...)
_validate_volume_block(...)
_validate_caller_pipeline_gate(...)
```

Acceptance:

- all existing tests pass;
- no standalone-volume behavior is promised yet;
- ZigZag-only remains bit-identical.

### W2.C2 Explicit Breaking Legality Migration

This PR intentionally changes config semantics.

New final resolver:

```text
if explicit zigzag.enabled:
    preserve raw value for Phase 4 validation
elif legacy ZigZag marker present:
    set True
elif volume.enabled is true and no legacy marker:
    set False
elif tf.enabled is true:
    set False
else:
    set False
```

Final legality dispatch:

```text
if not is_trade_filter_enabled(tf):
    skip subfilter validation
elif zigzag.enabled is True and volume.enabled is not True:
    validate ZigZag path
elif zigzag.enabled is True and volume.enabled is True:
    validate ZigZag + volume path
elif zigzag.enabled is not True and volume.enabled is True:
    validate standalone volume path
else:
    error: at least one of zigzag.enabled, volume.enabled must be true
```

Standalone-volume rules:

- volume block required and fully validated;
- ZigZag fields other than `enabled` are rejected if present;
- `trade_filter.type=zigzag_st_mode` is rejected when `zigzag.enabled=false`;
- `trade_filter.triggers` is forbidden;
- lifecycle block is optional, scalar-validated if present, and inert at
  runtime;
- time_filter is optional and active as standalone reset.

ZigZag+volume rules:

- ZigZag block required and fully validated;
- volume block required and fully validated;
- lifecycle required and fully validated;
- triggers optional for legacy ZigZag path.

Migration notes:

- `empty zigzag: {}` + `trade_filter.enabled=true` with no volume becomes
  invalid.
- Existing users should add `zigzag.enabled: true` or remove the empty block
  and use a canonical legacy ZigZag marker.
- PR description must include before/after YAML examples.

Tests:

- standalone volume without ZigZag block is valid;
- standalone volume with `type=zigzag_st_mode` is invalid;
- standalone volume with triggers block is invalid;
- both subfilters disabled is invalid;
- migration error messages include actionable hints;
- ZigZag-only behavior remains bit-identical.

## W3. Data Validation, Volume Metrics and VolumeRuntime

### W3.1 Volume Data Validation

In `donor/supertrend_optimizer/data/validator.py` add:

```text
validate_volume_filter_data(df, trade_filter_config) -> pd.DataFrame
```

No-op when volume is not enabled.

When volume is enabled:

- `volume` column required;
- numeric dtype required;
- no NaN;
- no Inf;
- `volume >= 0`;
- input DataFrame is not mutated unexpectedly.

Every error contains:

```text
trade_filter.volume
```

Call-sites:

- `wf_grid/pipeline/orchestrator.py` after CSV load, collision check,
  lowercase normalization and OHLC validation;
- active tester runtime after `load_ohlc_csv` and OHLC validation.

### W3.2 `volume_metrics.py`

Create:

```text
donor/supertrend_optimizer/core/volume_metrics.py
```

Public API:

```text
build_volume_global_metrics(volume, close, volume_cfg) -> VolumeRuntime
materialize_volume_regime(codes: np.ndarray) -> np.ndarray
materialize_volume_block_reason(codes: np.ndarray) -> np.ndarray
materialize_volume_initial_direction(codes: np.ndarray) -> np.ndarray
```

Rules:

- validate `len(volume) == len(close)`;
- rolling medians use `pandas.Series(...).rolling(window=W, min_periods=W).median()`;
- calculations are causal and include decision bar `t`;
- no numpy-loop, numba or bottleneck implementation in v5.

### W3.3 Metrics and Edge Cases

Metrics:

```text
short_median_volume
baseline_median_volume
median_relative_volume
volume_regime
volume_condition_allowed
volume_condition_block_reason
volume_initial_direction
```

Edge-case rules must be tested:

- `baseline_median_volume == 0` -> regime/block reason `volume_baseline_zero`;
- all-zero volume;
- `len(data) < baseline_window`;
- integer volume dtype;
- very large finite volume values;
- NaN/Inf rejected before metrics build;
- equal close momentum produces `volume_initial_direction=unknown`;
- close ties do not create fake direction;
- timezone index works with reset helpers downstream.

`VolumeRuntime.slice(start, end)`:

- validates `0 <= start <= end <= reference_length`;
- returns a frozen `VolumeRuntime`;
- slices per-bar arrays with NumPy views;
- preserves read-only flags;
- sets:

  ```text
  absolute_offset = parent.absolute_offset + start
  reference_length = end - start
  filter_config_snapshot is same object reference
  ```

Tests:

- `np.shares_memory` for every per-bar array;
- chained slices share memory with original parent;
- empty slice is valid;
- bounds violations raise `ValueError`;
- snapshot identity is preserved.
