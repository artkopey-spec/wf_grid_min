## 0. Решения, фиксируемые до старта реализации

| # | Решение | Обоснование |
|---|---|---|
| 0.1 | Default `trade_filter.zigzag.daily_reset = False` | Acceptance-инвариант «без поля → baseline bit-identical». |
| 0.2 | Dataclass-изменения в `donor/supertrend_optimizer/core/trade_filter_config.py` | Это shared-источник для Phase 1 (WF Grid) и Phase 2 (TESTER). `wf_grid/config/schema.py` — re-export shim, не трогаем. |

## 4. Helper `_infer_daily_reset_event`

Размещение: `donor/supertrend_optimizer/core/zigzag_st_filter.py`, рядом с `_validate_close_and_threshold`.

### 4.1 Сигнатура и порядок проверок

```python
def _infer_daily_reset_event(
    index: Optional[pd.Index],
    n: int,
    *,
    enabled: bool,
) -> np.ndarray:
    """Calendar-day reset event mask (bool[n]).

    Source: алгоритм нормализации заимствован 1:1 из
    ``donor zigzag/engine/run.py:_infer_session_ids`` (plan v2.0 §10.1).
    Отличия от донора:
    - возвращает bool[]-event, а не int64[]-session_ids;
    - short-circuit при ``enabled=False`` (baseline bit-identity);
    - non-monotonic / non-datetime → ``ConfigError`` (fail-closed),
      вместо warn+fallback к нулям.
    """
    # 0.5: short-circuit ДО любых проверок индекса
    if not enabled:
        return np.zeros(n, dtype=bool)

    # 0.4: type-gate
    if not isinstance(index, pd.DatetimeIndex):
        raise ConfigError(
            "trade_filter.zigzag.daily_reset=true requires DatetimeIndex; "
            f"got {type(index).__name__}"
        )
    if len(index) != n:
        raise ConfigError(
            f"daily_reset: index length {len(index)} != n={n}"
        )

    # 0.3: monotonic-gate
    if not index.is_monotonic_increasing:
        raise ConfigError(
            "trade_filter.zigzag.daily_reset=true requires monotonic-increasing "
            "DatetimeIndex; got non-monotonic"
        )

    # Нормализация — буквальный перенос из donor zigzag/engine/run.py:98-102.
    # CRITICAL: tz_localize(None), а не tz_convert(None) — иначе MSK 23:55
    # сместится в UTC 20:55 и граница полуночи сломается.
    if index.tz is not None:
        normalized = index.tz_localize(None).normalize()
    else:
        normalized = index.normalize()

    days = normalized.astype("int64").to_numpy()
    event = np.zeros(n, dtype=bool)
    if n >= 2:
        event[1:] = days[1:] != days[:-1]
    # event[0] всегда False — нет «предыдущего бара»
    return event
```

### 4.2 Edge-cases (закрепляются тестами 9.2)

- `n == 0` → `np.zeros(0, dtype=bool)` без падений.
- `n == 1` → `np.array([False])`.
- DST-переход внутри одного дня → `event[t] = False` (даты совпадают после `normalize()`).
- Weekend gap (пятница→понедельник) → один transition, не три.

### 5.2 Reset-блок в основном цикле `_run_close_only_zigzag_pass`

Вставить **первой операцией** в `for t in range(1, n):` — ДО проверки pathological-bar (строки 281–291):

```python
for t in range(1, n):
    c = float(arr[t])

    # --- DAILY RESET (must run BEFORE pathological/main logic) -------------
    # Comment-обоснование заимствовано из donor zigzag/zigzag_filter.py:401-407:
    # "Calendar-day boundary is OHLC-agnostic. Even on a NaN/inf bar the
    #  session boundary is real and operational state must be cleared;
    #  otherwise the first valid bar of the new day will not re-seed."
    # NB: legs[], last_confirmed_idx, last_confirmed_height — НЕ сбрасываются
    # (накопительная история нужна для local_median_N; подтверждено донором).
    if daily_reset_event is not None and daily_reset_event[t]:
        if math.isfinite(c) and c > 0.0:
            run_min_bar = t; run_min_price = c
            run_max_bar = t; run_max_price = c
            last_pivot_bar = t; last_pivot_price = c
            run_ext_bar = t; run_ext_price = c
        else:
            # Pathological bar на reset: убиваем in-flight candidate, но не
            # пере-seed'им — следующий валидный бар сам поднимет seed (как
            # обычная UNKNOWN-ветка).
            run_min_price = float("nan"); run_max_price = float("nan")
            last_pivot_price = float("nan"); run_ext_price = float("nan")
        direction = _LEG_DIR_UNKNOWN

    # --- PATHOLOGICAL BAR (existing logic, unchanged) ----------------------
    if not math.isfinite(c) or c <= 0.0:
        confirmed_leg_idx_at_t[t] = last_confirmed_idx
        last_confirmed_leg_height_pct[t] = last_confirmed_height
        continue
    ...
```

### 5.3 Семантические следствия

- `direction = UNKNOWN` после reset → `candidate_height_pct[t]` остаётся `NaN` (см. строки 415–421) → contour-A trigger на reset-баре не сработает.
- `confirm_event[t] == 0` (UNKNOWN-ветка не делает confirm) → contour-B trigger на reset-баре не сработает. **Это ключевой инвариант**: «вчерашний candidate не подтверждается на первом баре нового дня».
- `confirmed_leg_idx_at_t[t]` остаётся cumulative (`= last_confirmed_idx`) → `local_median_N` дня 2 продолжает учитывать леги дня 1.

### 5.4 Совместимость

При `daily_reset_event=None` или `np.zeros(n, dtype=bool)` reset-блок не выполняется → bit-identity с baseline.

### 6.3 Reset FSM в основном цикле

В `for t in range(n):` (после строки 1271) **первой операцией**, ДО snapshot `state_at_bar_start`:

```python
for t in range(n):
    # --- DAILY RESET (must be first; resets state seen by all rules below) -
    if daily_reset_event[t]:
        state = ZigZagFSMState.OFF
        confirmed_legs_since_start = -1
        held_pos = 0
        _stopping_start = -1

    state_at_bar_start = state
    ...
```

### 6.4 Политика «flat from next bar» (автоматически)

- На бар t reset → `state = OFF`.
- Дальше в цикле: `if state in (OFF, WAIT): next_pos = 0` → `filtered_positions[t+1] = 0`.
- `filtered_positions[t]` уже записан на предыдущей итерации — отражает позицию активную во время бара t. Не трогаем.

### 6.5 Edge-cases

- **`t == 0`**: `daily_reset_event[0]` всегда `False` (см. 4.2) → reset на первом баре slice не происходит.
- **Одновременный reset + valid trigger + allowed flip на одном баре**: после reset `state=OFF, held_pos=0`, дальше FSM-логика бара выполняется как обычно: `OFF + (a_trig or b_trig) → WAIT`, `WAIT + flip → FREEZE`. Новый lifecycle может стартовать прямо на reset-баре — это допустимо и желательно (не теряем сигналы на стыке дней).

## 8. OOS prepend / trim

### 8.1 Без правок `step_executor.py`

`step_executor.py:307-309` уже режет любые ключи синхронно:
```python
filter_diagnostics_oos = {k: v[oos_boundary:] for k, v in _ext_filter_diag.items()}
```

`daily_reset_event[oos_boundary:]` подхватится автоматически, длина совпадёт с `oos_returns/positions/equity`.

### 8.2 Семантика на boundary

- `daily_reset_event` считается на extended-slice `ext_index = full_index[ext_start:test_end]`.
- День меняется внутри prepend → reset срабатывает в prepend → FSM приходит в OOS в `OFF + flat`.
- День меняется ровно на границе prepend↔OOS → `daily_reset_event[oos_boundary] == 1` → reset на первом OOS-баре.
- `prepend_applied == 0` → `daily_reset_event[0] == False` (нет «предыдущего дня») → reset не срабатывает на первом OOS-баре. Это корректно.

### 8.3 Force-flat (`_apply_oos_force_flat`)

Не трогает diagnostics, только `oos_returns/positions/equity`. Совместим без правок: carry-in трейд, закрытый по reset в prepend, в `ext_trades_df` имеет `exit_index < oos_boundary` → не попадает в carry-mask → force-flat не срабатывает (корректно).

## 10. Acceptance criteria

| AC | Критерий |
|---|---|
| AC-1 | `daily_reset: false` (default) → `positions`, `returns`, `equity_curve`, `trades_df` bit-identical baseline'у |
| AC-2 | `daily_reset: true` + tz-aware DatetimeIndex → ZigZag и FSM сбрасываются на первом баре новой календарной даты в локальном времени индекса |
| AC-3 | tz-aware index использует `tz_localize(None).normalize()` (Moscow 23:55 / 00:05 дают reset, а не сдвинутая UTC-граница) |
| AC-4 | `daily_reset` виден в `filter_diagnostics["daily_reset_event"]`, `daily_reset_enabled`, в trade-level `exit_reason="filter_daily_reset"`, в `filter_block_reason="daily_reset"`, в summary `daily_reset_count` |
| AC-5 | Длины `positions`, `returns`, `equity_curve`, всех ключей `filter_diagnostics` равны между собой; OOS-trim корректен |
| AC-6 | `daily_reset: true` + non-DatetimeIndex или non-monotonic → `ConfigError` до запуска WF (fail-closed) |
| AC-7 | Confirmed-leg history (`legs[]`, `last_confirmed_idx`, `local_median_N`) сохраняется через reset — `local_median_N` дня 2 учитывает леги дня 1 |
| AC-8 | Оба whitelist'а (`_ALLOWED_KEYS` и `TRADE_FILTER_ALLOWED_KEYS`) синхронно содержат `daily_reset` |
| AC-9 | 13+ существующих вызовов `run_backtest_fast` без `index` продолжают работать |
| AC-10 | Phase 2 TESTER (через shared dataclass) собирается и проходит существующие тесты `test_phase2_*` |
| AC-11 | Test-override `daily_reset_event` в `apply()` работает (паттерн из донора) |
| AC-12 | Унаследованный из донора tz-test (`test_tz_aware_index_moscow_midnight`) проходит — guard против регрессии `tz_localize` → `tz_convert` |
