# Appendix A — ZigZag ST Trade Filter Spec v1.1

## 0. Назначение документа

Этот документ является якорным specification для внедрения ZigZag-фильтра сделок в основной SuperTrend/WF pipeline приложения.

Любой план внедрения, декомпозиция задач, PRD, TODO-list или patch-plan по этому фильтру должен прикладывать этот spec как Appendix A и проверяться на соответствие ему. Если в ходе разработки поведение фильтра меняется, spec может быть обновлён на новую версию только после явного разрешения владельца/заказчика. Без такого разрешения план обязан оставаться в рамках текущей версии spec и не имеет права неявно переопределять его правила.

v1.1 уточняет implementation-blocking поведение:

- точную формулу ZigZag-ног и единицы измерения;
- область расчёта `local_median_N`;
- поведение `WAIT_FIRST_ST_FLIP` с учётом `trade_mode`;
- порядок `signal на закрытии -> сделка на следующем открытии`;
- расчёт auto `candidate_trigger_threshold`;
- insufficient-data / fail-closed поведение;
- diagnostics keyset и trade-level exit reasons.

## 1. Цель фильтра

Фильтр должен разрешать торговлю по SuperTrend только в периоды, когда ZigZag-структура рынка показывает достаточно высокий режим движения/волатильности.

Фильтр не заменяет SuperTrend. Он управляет допуском к ST-trading mode:

- когда разрешено ждать первый ST flip;
- когда разрешены новые ST-входы/реверсы;
- когда режим должен перейти в graceful stop;
- когда режим полностью выключается.

## 2. Ключевая идея

В приложении существует один общий `ST-trading mode`.

Он может быть запущен двумя независимыми контурами:

- **Контур A — Early / Candidate trigger**: текущая candidate ZigZag-нога достигла настраиваемого порога из config, например quantile-порога вроде p80.
- **Контур B — Confirmed / Median trigger**: `local_median_N` по последним `N` подтверждённым ZigZag-ногам достигла `global_median`, где `N = zigzag.local_window` из config.

Контуры A и B можно включать/выключать через config независимо.

После запуска оба контура используют один и тот же жизненный цикл `ST-trading mode`.

## 3. Термины

### 3.1 ZigZag-нога

**Confirmed ZigZag leg** — подтверждённая ZigZag-нога. Используется для расчёта rolling window и `local_median_N`.

**Candidate ZigZag leg** — текущая формирующаяся ZigZag-нога. Она может быть ещё не подтверждена. Используется только контуром A для раннего запуска режима.

**ZigZag reversal** — разворот, достаточный для подтверждения текущей ZigZag-ноги и начала новой ноги. Порог разворота задаётся параметром `zigzag.reversal_threshold` из config.

### 3.2 Метрики и параметры фильтра

`reversal_threshold` — обязательный параметр ZigZag из config. Задаётся как доля от цены, не как процент. Пример: `0.005` означает 0.5% движения цены. Этот параметр используется ZigZag-логикой для определения подтверждения разворота/confirmed-ноги и формирования candidate-ноги. Он не должен быть захардкожен в коде.

`candidate_height_pct` — текущая высота candidate-ноги. Формула должна быть единой в реализации ZigZag и тестах. В v1.1 значение хранится как доля от цены, а не как умноженный процент.

`candidate_trigger_threshold` — настраиваемый порог из config для запуска контура A по текущей candidate-ноге. Значение может быть задано явно как numeric fraction или рассчитано как выбранный статистический уровень, например quantile p80. `p80` — только пример, не обязательная константа поведения.

`global_median` — глобальная медиана высот подтверждённых ZigZag-ног, рассчитанная по всему входному массиву данных. Это один глобальный baseline для данного dataset/run, а не rolling/window-local и не per-WF-step median. Используется контуром B и stop-check.

`local_median_N` — медиана высот последних `N` confirmed ZigZag-ног, где `N = zigzag.local_window` из config. В примерах может использоваться `N=5`, но 5 не является жёстко зашитой константой поведения. Считается только по confirmed-ногам.

`local_median_N` считается по последним `N` confirmed ZigZag-ногам, доступным в causal execution-slice history. Она не ограничивается ногами после текущего старта lifecycle.

`freeze_confirmed_legs` и `local_window` независимы:

- `freeze_confirmed_legs` считает confirmed ZigZag-ноги после старта lifecycle;
- `local_window` задаёт rolling window для `local_median_N` по slice-local confirmed-leg history.

Следовательно, `freeze_confirmed_legs < local_window` — валидная конфигурация. Это не ошибка и не warning само по себе.

Все ZigZag filter thresholds и height metrics в config и runtime представлены как числовые доли цены, не как percent strings:

- `reversal_threshold: 0.005` означает 0.5% цены;
- `candidate_trigger_threshold: 0.012` означает 1.2% цены;
- `candidate_height_pct = 0.012` означает, что высота текущей candidate-ноги равна 1.2% её базовой цены;
- `global_median = 0.008` означает, что медиана высот confirmed-ног на full dataset равна 0.8%;
- `local_median_N = 0.010` означает, что локальная медиана высот confirmed-ног равна 1.0%.

Config values должны быть numeric fractions. Percent strings вроде `"0.5%"` недопустимы.

Несмотря на суффикс `_pct`, v1.1 хранит `candidate_height_pct`, confirmed-leg heights, `global_median`, `local_median_N` и materialized `candidate_trigger_threshold` как доли, а не как значения в шкале 0–100.

### 3.3 ST flip

`ST flip` — смена направления/цвета SuperTrend между двумя ненулевыми противоположными состояниями:

- `+1 -> -1` — short flip;
- `-1 -> +1` — long flip.

Переход из warmup/unknown состояния не является tradable ST flip:

- `0 -> +1` — initialization, не long flip;
- `0 -> -1` — initialization, не short flip.

Такие initialization transitions не могут быть использованы в `WAIT_FIRST_ST_FLIP` для входа и не могут закрывать позицию в `ST_STOPPING`.

Фильтр не меняет формулу SuperTrend. Он только решает, можно ли использовать ST flip как вход/реверс/выход.

### 3.4 ZigZag formula

ZigZag v1.1 использует **close-only price stream**.

В v1.1 не используется high/low wick-based pivot detection и не используется derived stream вроде `hlc3` / `ohlc4`.

#### 3.4.1 Pivot detection

ZigZag engine отслеживает текущий candidate extremum по close stream.

ZigZag reversal подтверждается, когда close price разворачивается от текущего candidate extremum минимум на `zigzag.reversal_threshold`, где `reversal_threshold` — numeric fraction of price из config.

#### 3.4.2 Confirmed-leg height

Для каждой confirmed ZigZag-ноги:

```text
height_pct = abs(end_pivot_price - start_pivot_price) / start_pivot_price
```

Значение хранится как доля цены.

#### 3.4.3 Candidate-leg height

Для текущей unconfirmed candidate-ноги:

```text
candidate_height_pct[t] = abs(current_candidate_extremum_price - last_confirmed_pivot_price) / last_confirmed_pivot_price
```

Значение хранится как доля цены и использует ту же шкалу, что explicit `candidate_trigger_threshold`.

#### 3.4.4 Edge cases

- `last_confirmed_pivot_price <= 0` invalid и должен быть unreachable после OHLC validation.
- Identical consecutive close values дают zero candidate movement.
- `reversal_threshold` никогда не auto-calibrated; он берётся только из config.

## 4. Состояния фильтра

Фильтр должен быть реализован как конечный автомат.

Обязательные состояния:

1. `OFF`
2. `WAIT_FIRST_ST_FLIP`
3. `ST_ACTIVE_FREEZE`
4. `ST_ACTIVE_MONITORING`
5. `ST_STOPPING`

### 4.1 OFF

Базовое состояние. ST-trading mode выключен.

Новые ST-входы запрещены.

Переход из `OFF` в `WAIT_FIRST_ST_FLIP` происходит, если выполнено хотя бы одно включённое условие запуска:

- Контур A включён и `candidate_height_pct >= candidate_trigger_threshold`.
- Контур B включён и на confirm-bar новой ZigZag-ноги `local_median_N >= global_median`.

### 4.2 WAIT_FIRST_ST_FLIP

Фильтр ждёт следующий ST flip, разрешённый текущим `trade_mode`.

Только ST flip, разрешённый текущим `trade_mode`, может открыть первую позицию и запустить lifecycle:

- разрешённый `long flip` открывает LONG;
- разрешённый `short flip` открывает SHORT;
- состояние переходит в `ST_ACTIVE_FREEZE`;
- начинается отсчёт первых `freeze_confirmed_legs` confirmed ZigZag-ног после старта режима, где `freeze_confirmed_legs = lifecycle.freeze_confirmed_legs` из config.

Если ST flip запрещён текущим `trade_mode`:

- позиция не открывается;
- lifecycle не стартует;
- состояние остаётся `WAIT_FIRST_ST_FLIP`;
- фильтр продолжает ждать следующий разрешённый ST flip.

Примеры:

- `trade_mode=long`: short flip пропускается. Если фильтр всё ещё находится в `WAIT_FIRST_ST_FLIP`, следующий long flip открывает LONG и запускает `ST_ACTIVE_FREEZE`.
- `trade_mode=short`: long flip пропускается. Если фильтр всё ещё находится в `WAIT_FIRST_ST_FLIP`, следующий short flip открывает SHORT и запускает `ST_ACTIVE_FREEZE`.
- `trade_mode=both` / `revers`: long и short flips разрешены.

До первого разрешённого ST flip позиция не открывается.

### 4.3 ST_ACTIVE_FREEZE

ST-trading mode активен. Сделки идут по обычной ST-логике с учётом `trade_mode`.

Freeze-период длится первые `freeze_confirmed_legs` confirmed ZigZag-ног после старта режима.

Во время freeze:

- `local_median_N` может считаться для диагностики;
- unavailable / invalid `local_median_N` является только диагностикой;
- median-stop не выключает режим;
- новые ST-входы/реверсы разрешены по обычной ST-логике, если они разрешены текущим `trade_mode`.

После накопления `freeze_confirmed_legs` confirmed ZigZag-ног после старта режима состояние переходит в `ST_ACTIVE_MONITORING`.

### 4.4 ST_ACTIVE_MONITORING

ST-trading mode активен. Сделки идут по обычной ST-логике с учётом `trade_mode`.

На каждом confirm-bar новой ZigZag-ноги:

1. Обновляется rolling window последних `N` confirmed-ног, где `N = zigzag.local_window`.
2. Пересчитывается `local_median_N` по causal execution-slice history.
3. Выполняется stop-check:
   - если `local_median_N` доступна и `local_median_N >= global_median`, режим остаётся активным;
   - если `local_median_N` доступна и `local_median_N < global_median`, режим переходит в `ST_STOPPING`;
   - если `local_median_N` unavailable, NaN или Inf на confirm-bar, режим переходит в `ST_STOPPING` как fail-closed behavior.

Stop-check выполняется только на confirm-bar новой ZigZag-ноги, а не на каждой свече.

### 4.5 ST_STOPPING

Graceful stop режима.

В этом состоянии:

- новые ST-входы запрещены;
- реверсы запрещены;
- если открытой позиции нет, режим сразу переходит в `OFF`;
- если позиция есть, она не закрывается немедленно;
- текущая позиция закрывается на ближайшем обратном ST flip;
- после закрытия позиции режим переходит в `OFF`.

В `ST_STOPPING` нельзя открывать новую позицию по ST flip. Разрешён только выход из текущей позиции по обратному ST flip.

## 5. Контур A — Early / Candidate trigger

Контур A запускает ожидание ST-trading mode раньше, по текущей candidate ZigZag-ноге.

Правила:

1. На закрытии каждого бара читается текущая candidate-нога.
2. Candidate-нога не обязана быть confirmed.
3. Если контур A включён и `candidate_height_pct >= candidate_trigger_threshold`, фильтр переходит из `OFF` в `WAIT_FIRST_ST_FLIP`.
4. В `WAIT_FIRST_ST_FLIP` фильтр ждёт следующий ST flip, разрешённый текущим `trade_mode`.
5. Первый разрешённый ST flip открывает позицию по направлению SuperTrend и запускает общий жизненный цикл режима.

Контур A не использует `local_median_N` для запуска. Его порог берётся из config как `candidate_trigger_threshold`; quantile p80 может быть одним из вариантов такого порога, но не является обязательным значением.

## 6. Контур B — Confirmed / Median trigger

Контур B запускает ожидание ST-trading mode позже, только после confirm новой ZigZag-ноги.

Правила:

1. На закрытии бара проверяется, подтвердилась ли новая ZigZag-нога.
2. Если новой confirmed-ноги нет, контур B ничего не решает на этом баре.
3. Если новая confirmed-нога есть, её высота добавляется в rolling window последних confirmed-ног.
4. Считается `local_median_N` по последним `N` confirmed-ногам в causal execution-slice history, включая только что подтверждённую, где `N = zigzag.local_window` из config.
5. Если контур B включён и `local_median_N >= global_median`, фильтр переходит из `OFF` в `WAIT_FIRST_ST_FLIP`.
6. Первый разрешённый ST flip открывает позицию по направлению SuperTrend и запускает общий жизненный цикл режима.

Контур B не использует candidate-ногу для запуска.

## 7. Общий жизненный цикл A и B

Независимо от того, какой контур запустил режим, дальнейшее поведение одинаковое:

```text
OFF
  ↓ enabled A && candidate_height_pct >= candidate_trigger_threshold
    OR enabled B && local_median_N >= global_median на confirm-bar
WAIT_FIRST_ST_FLIP
  ↓ allowed ST flip according to trade_mode
ST_ACTIVE_FREEZE
  ↓ прошло freeze_confirmed_legs confirmed ZigZag legs после старта режима
ST_ACTIVE_MONITORING
  ↓ local_median_N < global_median на confirm-bar новой ноги
    OR local_median_N unavailable/invalid на confirm-bar
ST_STOPPING
  ↓ текущая позиция закрылась по обратному ST flip / либо позиции уже нет
OFF
```

## 8. Правила допуска сделок

### 8.1 OFF

Запрещены все новые ST-входы.

### 8.2 WAIT_FIRST_ST_FLIP

Разрешён только первый ST flip, который разрешён текущим `trade_mode`.

ST flip, запрещённый текущим `trade_mode`, игнорируется для входа и lifecycle-start.

### 8.3 ST_ACTIVE_FREEZE

Разрешены ST-входы, выходы и реверсы по обычной ST-логике, если они разрешены текущим `trade_mode`.

Median-stop не проверяется на выключение режима.

### 8.4 ST_ACTIVE_MONITORING

Разрешены ST-входы, выходы и реверсы по обычной ST-логике, если они разрешены текущим `trade_mode` и пока `local_median_N >= global_median` на confirm-bar.

### 8.5 ST_STOPPING

Новые ST-входы запрещены.

Если позиция есть, разрешён только выход по ближайшему обратному ST flip.

## 9. Trade mode совместимость

Фильтр должен учитывать существующий `trade_mode` приложения:

- `long`: разрешены только long-входы; short flip в active state может использоваться как выход из LONG, если это соответствует текущей ST-логике движка.
- `short`: разрешены только short-входы; long flip в active state может использоваться как выход из SHORT.
- `revers` / `both`: разрешены long и short входы/реверсы по текущей логике движка.

Фильтр не должен расширять возможности `trade_mode`. Он может только сузить допуск к сделкам.

В `WAIT_FIRST_ST_FLIP` disallowed flip не запускает lifecycle:

- при `trade_mode=long` short flip пропускается;
- при `trade_mode=short` long flip пропускается;
- при `trade_mode=both` / `revers` оба направления разрешены.

## 10. Точка интеграции в приложении

Фильтр должен работать внутри основного backtest/strategy execution path до расчёта итоговых returns/equity/trades/metrics.

Нельзя реализовывать этот фильтр как пост-фильтрацию `trades_df` после бэктеста, потому что тогда:

- bar-level returns/equity уже будут рассчитаны по нефильтрованной торговле;
- trade-level rows будут описывать другой набор сделок;
- метрики, агрегаты, gates, scoring и ranking могут стать несогласованными.

Правильная интеграция: фильтр должен участвовать в генерации сигналов/позиций, то есть до формирования финальных сделок и метрик.

Canonical execution rule for v1.1:

- ZigZag filter conditions и ST flip events оцениваются на закрытии бара;
- signal decision происходит на close бара `t`;
- execution происходит на open бара `t+1`;
- position array меняется начиная с `t+1`;
- сделка не открывается на том же close бара `t`.

Если trigger A или B и разрешённый ST flip возникли на одном decision bar `t`:

- trigger обрабатывается первым;
- same-bar allowed ST flip может быть использован как первый разрешённый flip;
- позиция открывается на следующем open, `t+1`.

Если same-bar ST flip запрещён текущим `trade_mode`:

- он пропускается;
- позиция не открывается;
- lifecycle не стартует;
- состояние остаётся `WAIT_FIRST_ST_FLIP`.

## 11. Config contract

В config должен появиться отдельный блок фильтра. Рекомендуемый namespace:

```yaml
trade_filter:
  enabled: true
  type: zigzag_st_mode

  zigzag:
    global_stats_source: full_dataset  # global_median is calculated on the full input data array
    leg_height_mode: pct               # v1.1 stores relative heights as fractions despite _pct suffix
    reversal_threshold: 0.005          # доля от цены, не %; обязательный параметр ZigZag
    candidate_trigger_threshold: auto  # numeric fraction or auto; p80 is only an example
    candidate_trigger_quantile: 0.80   # required only when threshold is auto
    global_median: auto
    local_window: 5                    # configurable N for local_median_N; 5 is default/example

  triggers:
    candidate_threshold:
      enabled: true
    confirmed_median:
      enabled: true

  lifecycle:
    freeze_confirmed_legs: 5           # configurable; 5 is default/example
    stop_check: confirm_bar_only
    stopping_exit: opposite_st_flip

  diagnostics:
    export_state_columns: true
    export_trigger_columns: true
```

### 11.1 Disabled config behavior

Если блок `trade_filter` отсутствует:

- фильтр выключен;
- baseline behavior сохраняется.

Если `trade_filter.enabled = false`:

- `type` и nested blocks опциональны;
- если `type` указан, он должен быть `zigzag_st_mode`;
- unknown keys всё равно отклоняются strict schema;
- поведение должно совпадать с baseline без фильтра.

Если `trade_filter.enabled = true`:

- `type = zigzag_st_mode` обязателен;
- `zigzag` block обязателен;
- `triggers` block обязателен;
- `lifecycle` block обязателен;
- `diagnostics` block опционален;
- минимум один trigger должен быть enabled.

### 11.2 Validation requirements

Config validation должна отклонять:

- `trade_filter.enabled` отсутствует, когда блок `trade_filter` присутствует;
- `trade_filter.enabled` не bool;
- `trade_filter.enabled=true` без `type`;
- `trade_filter.enabled=true` с `type != zigzag_st_mode`;
- `trade_filter.enabled=false` с указанным `type != zigzag_st_mode`;
- enabled filter без `zigzag` block;
- enabled filter без `triggers` block;
- enabled filter без `lifecycle` block;
- `triggers.candidate_threshold.enabled` не bool;
- `triggers.confirmed_median.enabled` не bool;
- оба triggers disabled при enabled filter;
- `zigzag.reversal_threshold` отсутствует, равен `None`, `NaN`, `<= 0`, `>= 1` или является percent string вроде `"0.5%"`;
- `zigzag.local_window` не integer >= 1;
- `lifecycle.freeze_confirmed_legs` не integer >= 0;
- unsupported `lifecycle.stop_check`;
- unsupported `lifecycle.stopping_exit`;
- invalid `candidate_trigger_threshold` по rules ниже.

### 11.3 candidate_trigger_threshold validation

Если `candidate_trigger_threshold` numeric:

- он должен быть finite, `> 0` и `< 1`;
- он использует ту же fraction scale, что `candidate_height_pct`;
- `candidate_trigger_quantile` не должен быть явно указан пользователем вместе с numeric threshold.

Если `candidate_trigger_threshold = auto`:

- `candidate_trigger_quantile` обязателен;
- `candidate_trigger_quantile` должен быть в `(0, 1)`.

## 12. Calibration / global stats contract

`global_median` рассчитывается по всему входному массиву данных и является глобальным baseline для данного dataset/run. Она не пересчитывается отдельно на каждом WF step и не является train-window-only статистикой.

Для WF pipeline canonical rule:

- `global_median` строится один раз по всем confirmed ZigZag-ногам полного входного массива данных с использованием `zigzag.reversal_threshold` из config.
- Это intentionally global dataset statistic. Поэтому план внедрения должен явно маркировать её как `full_dataset` statistic, чтобы её не перепутали с walk-forward-pure calibration threshold.
- `candidate_trigger_threshold` берётся из config. Если он задан как auto/quantile-based, источник расчёта фиксируется этим spec.
- `reversal_threshold` не является калибруемой метрикой; это обязательный параметр ZigZag из config и должен применяться одинаково в train/OOS/tester path.

### 12.1 Auto candidate_trigger_threshold

Если `candidate_trigger_threshold = auto`, он материализуется из full-dataset confirmed ZigZag leg heights.

Source population:

- все confirmed ZigZag leg heights из полного входного dataset;
- высоты считаются по той же close-only ZigZag formula из §3.4;
- используется `zigzag.reversal_threshold` из config.

Quantile method:

```text
np.quantile(confirmed_heights_pct, q=candidate_trigger_quantile, method="linear")
```

`candidate_trigger_quantile` задаёт `q`.

Пример: `candidate_trigger_quantile: 0.80` означает p80. `0.80` — config value/example, а не hardcoded behavior.

### 12.2 Minimum sample for auto threshold

Для `candidate_trigger_threshold = auto`:

```text
min_legs_for_quantile = max(zigzag.local_window, 10)
```

Если в full dataset меньше confirmed ZigZag-ног, чем `min_legs_for_quantile`, global stats initialization завершается ошибкой до WF execution.

Если `candidate_trigger_threshold` задан explicit numeric value, `min_legs_for_quantile` не применяется к materialization порога контура A.

### 12.3 Global stats initialization failure

Если `trade_filter.enabled = true` и full-dataset `ZigZagGlobalStats` невозможно материализовать, pipeline initialization завершается ошибкой до WF execution.

Примеры:

- нет confirmed ZigZag-ног для `global_median`;
- `global_median` is NaN or Inf;
- `candidate_trigger_threshold = auto` и confirmed legs меньше `min_legs_for_quantile`;
- materialized `candidate_trigger_threshold` is NaN or Inf.

Runtime fail-closed применяется к slice-local unavailable runtime stats, а не к invalid global initialization.

## 13. Diagnostics / export contract

Для проверки фильтра в результатах должны быть доступны диагностические поля. Минимальный bar-level набор:

- `trade_filter_enabled`
- `trade_filter_state`
- `trade_filter_trigger_source`
- `zigzag_reversal_threshold`
- `candidate_height_pct`
- `candidate_trigger_threshold`
- `local_median_N`
- `local_median_available`
- `local_window`
- `global_median`
- `global_stats_available`
- `confirmed_legs_since_start`
- `freeze_confirmed_legs`
- `median_stop_triggered`
- `stopping_started_at_index`
- `filter_allowed_entry`
- `filter_block_reason`

`trade_filter_trigger_source` values:

- `candidate_threshold`
- `confirmed_median`
- `both`
- `none`

`filter_block_reason` values include:

- `none`
- `filter_off`
- `waiting_for_allowed_st_flip`
- `trade_mode_disallowed_flip`
- `local_median_unavailable`
- `invalid_stats`
- `stopping_mode_no_new_entries`
- `insufficient_global_stats`

На уровне сделок желательно добавить:

- `entry_filter_state`
- `entry_trigger_source`
- `exit_reason`

Trade-level `exit_reason` values:

- `st_flip`: обычный ST exit/reversal вне filter stopping semantics;
- `filter_stopping_opposite_flip`: позиция закрыта, пока фильтр находится в `ST_STOPPING`, по ближайшему обратному ST flip;
- `pending_open_trade_at_end`: открытая сделка закрыта/помечена на последнем баре, если текущее donor behavior так делает.

Existing donor reasons могут сохраняться, но filter-specific reasons выше обязательны.

## 14. Инварианты

1. `reversal_threshold` берётся только из config и применяется как доля от цены, не как процент.
2. `candidate_height_pct`, confirmed-leg heights, `global_median`, `local_median_N` и materialized `candidate_trigger_threshold` используют одну fraction scale с config thresholds.
3. `local_median_N` считается только по confirmed ZigZag-ногам, где `N = zigzag.local_window` из config.
4. `local_median_N` считается по causal slice-local confirmed-leg history и не ограничивается ногами после lifecycle start.
5. `freeze_confirmed_legs` считает confirmed legs после lifecycle start и независим от `local_window`.
6. Stop-check выполняется только на confirm-bar новой ZigZag-ноги.
7. Candidate-нога используется только для запуска контура A.
8. Контур B не запускается без новой confirmed-ноги.
9. Первые `freeze_confirmed_legs` confirmed ZigZag-ног после старта режима не могут выключить режим через median-stop.
10. В `WAIT_FIRST_ST_FLIP` только ST flips, разрешённые `trade_mode`, могут открыть первую позицию и запустить lifecycle.
11. ST flip `0 -> ±1` является initialization, а не tradable flip.
12. Signal decisions происходят на close бара; position changes исполняются с next open.
13. Same-bar trigger и allowed ST flip могут быть использованы вместе under close-decision / next-open execution.
14. В `ST_STOPPING` новые ST-входы запрещены.
15. Открытая позиция в `ST_STOPPING` закрывается только на ближайшем обратном ST flip.
16. После закрытия позиции в `ST_STOPPING` состояние становится `OFF`.
17. Если позиции нет при входе в `ST_STOPPING`, состояние сразу становится `OFF`.
18. При `trade_filter.enabled: false` результаты должны совпадать с baseline без фильтра.

## 15. Edge cases

### 15.1 Недостаточно confirmed-ног

`local_median_N` unavailable, когда в causal execution-slice history на текущем баре доступно меньше `zigzag.local_window` confirmed ZigZag-ног.

Для контура B:

- если `local_median_N` unavailable на confirm-bar, контур B не может запустить activation.

Во время `ST_ACTIVE_FREEZE`:

- unavailable `local_median_N` — только diagnostics;
- median-stop намеренно выключен во freeze.

Во время `ST_ACTIVE_MONITORING`:

- если `local_median_N` unavailable, NaN или Inf на confirm-bar, фильтр переходит в `ST_STOPPING` как fail-closed behavior.

`freeze_confirmed_legs < local_window` валиден и не означает, что `local_median_N` unavailable, потому что `local_median_N` использует всю causal slice-local confirmed-leg history, а не только legs после lifecycle start.

### 15.2 Оба контура сработали одновременно

Если A и B сработали на одном баре, состояние всё равно одно: `WAIT_FIRST_ST_FLIP`.

Для диагностики можно фиксировать `trigger_source = both`, но торговое поведение не должно раздваиваться.

### 15.3 Повторный trigger во время active mode

Если фильтр уже находится в `WAIT_FIRST_ST_FLIP`, `ST_ACTIVE_FREEZE`, `ST_ACTIVE_MONITORING` или `ST_STOPPING`, новые trigger-события A/B не должны перезапускать lifecycle и не должны сбрасывать freeze-счётчик.

### 15.4 ST flip на том же баре, что trigger

Canonical execution rule for v1.1:

Все ZigZag filter conditions и ST flip events оцениваются на close бара.

Если trigger A или B и разрешённый ST flip возникли на одном decision bar `t`:

- filter trigger обрабатывается первым;
- same-bar allowed ST flip может быть использован как первый разрешённый flip;
- resulting position change исполняется на next bar open, `t+1`;
- сделка не открывается на same bar close.

Если same-bar ST flip не разрешён текущим `trade_mode`:

- он пропускается;
- позиция не открывается;
- lifecycle не стартует;
- состояние остаётся `WAIT_FIRST_ST_FLIP`.

Under close-decision / next-open execution:

- signal decision happens on close of bar `t`;
- execution happens on open of bar `t+1`;
- position array changes from `t+1`.

### 15.5 Gap / missing data

При невозможности корректно рассчитать runtime ZigZag, candidate leg, local median или ST flip фильтр должен fail-closed according to state-specific rules in §15.7.

Invalid global stats initialization обрабатывается до WF execution согласно §12.3.

### 15.6 Некорректный reversal_threshold

Если `reversal_threshold` отсутствует, равен `None`, `NaN`, `<= 0`, `>= 1` или передан как строка с процентом вроде `0.5%`, config validation должна завершиться ошибкой до запуска pipeline.

### 15.7 Runtime invalid / unavailable stats policy

#### OFF

Invalid or unavailable `candidate_height_pct` or `local_median_N` cannot trigger activation.

State remains `OFF`.

#### WAIT_FIRST_ST_FLIP

Invalid or unavailable stats:

- do not create additional triggers;
- do not reset lifecycle;
- do not open a position;
- do not cancel a previously valid trigger unless a future spec version explicitly adds cancellation behavior.

State remains `WAIT_FIRST_ST_FLIP` until an allowed ST flip arrives.

#### ST_ACTIVE_FREEZE

During freeze:

- `local_median_N` may be computed for diagnostics;
- invalid or unavailable `local_median_N` is diagnostics-only;
- median-stop is intentionally disabled.

#### ST_ACTIVE_MONITORING

Stop-check is evaluated only on confirm-bar.

If `local_median_N` is valid:

- `local_median_N >= global_median` -> remain `ST_ACTIVE_MONITORING`;
- `local_median_N < global_median` -> enter `ST_STOPPING`.

If `local_median_N` is unavailable, NaN, or Inf on confirm-bar:

- enter `ST_STOPPING` as fail-closed behavior.

#### ST_STOPPING

Runtime stats are ignored.

No new entries or reversals are allowed.
Existing position closes only on nearest opposite ST flip.
If there is no open position, state transitions to `OFF`.

## 16. Non-goals

В рамках этого spec фильтр не должен:

- менять формулу SuperTrend;
- менять формулу ATR/multiplier;
- менять WF slicing;
- менять scoring/ranking напрямую;
- пост-фактум удалять сделки из `trades_df` без пересчёта bar-level экономики;
- неявно считать `global_median` per-WF-step или train-window-only статистикой; по этому spec она является global full-dataset median;
- неявно использовать будущие OOS-данные для `candidate_trigger_threshold`; auto threshold по этому spec считается по full input dataset как intentionally global statistic;
- открывать позиции без ST flip;
- использовать ST flip, запрещённый текущим `trade_mode`, для открытия первой позиции в `WAIT_FIRST_ST_FLIP`;
- считать `0 -> ±1` initialization tradable ST flip;
- подменять `reversal_threshold` автоматически рассчитанным значением, если он задан явно в config.

## 17. Acceptance criteria

### 17.1 Функциональные критерии

1. При выключенном фильтре pipeline воспроизводит baseline.
2. ZigZag использует `reversal_threshold` из config как долю от цены, не как процент.
3. ZigZag confirmed-leg и candidate-leg heights используют exact close-only formula из §3.4.
4. Explicit numeric `candidate_trigger_threshold` использует ту же fraction scale, что `candidate_height_pct`.
5. Auto `candidate_trigger_threshold` материализуется из full-dataset confirmed-leg heights с pinned quantile method.
6. Если auto threshold имеет insufficient confirmed-leg sample, pipeline initialization падает до WF execution.
7. Контур A запускает `WAIT_FIRST_ST_FLIP` по candidate-ноге `>= candidate_trigger_threshold` без ожидания confirmed-ноги; p80 является только примером возможного значения/метода расчёта этого порога.
8. Контур B запускает `WAIT_FIRST_ST_FLIP` только на confirm-bar и только при `local_median_N >= global_median`.
9. Первый разрешённый ST flip после `WAIT_FIRST_ST_FLIP` открывает позицию по направлению ST.
10. В `WAIT_FIRST_ST_FLIP` disallowed ST flip пропускается и не запускает lifecycle.
11. Для `trade_mode=long`, short flip в `WAIT_FIRST_ST_FLIP` игнорируется; следующий разрешённый long flip может открыть LONG, если состояние всё ещё `WAIT_FIRST_ST_FLIP`.
12. Для `trade_mode=short`, long flip в `WAIT_FIRST_ST_FLIP` игнорируется; следующий разрешённый short flip может открыть SHORT, если состояние всё ещё `WAIT_FIRST_ST_FLIP`.
13. Same-bar filter trigger и allowed ST flip на bar close могут привести к entry, исполняемому на next bar open.
14. Переход `0 -> ±1` в ST trend не является tradable ST flip.
15. Freeze длится ровно `freeze_confirmed_legs` confirmed ZigZag-ног после старта режима.
16. В freeze median-stop не выключает режим.
17. После freeze stop-check выполняется только на confirm-bar.
18. При `local_median_N < global_median` режим переходит в `ST_STOPPING`.
19. Если `local_median_N` unavailable/invalid в `ST_ACTIVE_MONITORING` на confirm-bar, фильтр переходит в `ST_STOPPING`.
20. `freeze_confirmed_legs < local_window` валиден и сам по себе не вызывает warning, config reject или unavailable median.
21. В `ST_STOPPING` новая позиция не открывается.
22. Открытая позиция закрывается на ближайшем обратном ST flip.
23. После выхода из позиции режим возвращается в `OFF`.
24. Метрики, trades, returns, equity и позиции соответствуют одному и тому же отфильтрованному набору торговых решений.

### 17.2 Технические критерии

1. Config validation ловит неизвестные/ошибочные ключи фильтра.
2. Config validation требует корректный `zigzag.reversal_threshold`.
3. Config validation покрывает disabled block, missing nested blocks, numeric/auto threshold rules и wrong lifecycle enum values.
4. Unit tests покрывают exact ZigZag formula fixtures.
5. Unit tests покрывают каждый переход state machine.
6. Unit tests покрывают `trade_mode`-disallowed first flip behavior.
7. Unit tests покрывают same-bar trigger + allowed ST flip under close-signal / next-open execution.
8. Unit tests покрывают `0 -> ±1` non-flip initialization.
9. Unit tests покрывают state-specific invalid/unavailable stats behavior.
10. Integration tests покрывают OOS execution path.
11. Regression test сравнивает `trade_filter.enabled: false` с baseline.
12. Diagnostics позволяют восстановить, почему каждая сделка была разрешена или заблокирована.

Implementation plan обязан ссылаться на разделы этого spec при каждой задаче.

Формат ссылки в плане:

```text
Spec reference: Appendix A §<section number>
```

Любой пункт плана без spec reference считается неподтверждённым и не должен попадать в реализацию.

Изменение самого spec допускается только после явного разрешения владельца/заказчика. Нельзя обновлять версию spec, менять lifecycle, thresholds, state machine, config contract или acceptance criteria по инициативе implementation plan без отдельного подтверждения.
