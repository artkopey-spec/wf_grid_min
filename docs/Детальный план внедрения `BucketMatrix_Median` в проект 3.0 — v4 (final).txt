# Детальный план внедрения `BucketMatrix_Median` в проект 3.0 — v4 (final)

---

## 1. Executive Summary

**Цель**: добавить в Excel-экспорт 3.0 новый лист **`BucketMatrix_Median`**, содержащий полную median-based матрицу бакетов параметров (ATR × Multiplier) с производными метриками и stability heatmap.

**Контракт**:

`BucketMatrix_Median` в 3.0 строится по **pre-gate** `step_oos_long` (до `apply_step_gates`), агрегирует **только `ok`-строки** (статусы на pre-gate стадии: `ok`, `no_trades`, `insufficient_bars`, `runtime_error` — статус `gate_failed` ещё не существует); после ok-фильтра обязательно очищает `INVALID_METRIC_VALUE → NaN`; работает только при одном `trade_mode` (fail-fast при >1); использует 3.0-native шаги `Step1…StepN`; считает `total_steps` по полному pre-gate `step_oos_long` через `wf_step.nunique()`; получает bucket-параметры из `BucketConfig`; вычисляет `bucket_size` через перечисление физических grid points по `optimization.multiplier_step`; поддерживает в MVP только `metric_column="sum_pnl_pct"`; пишет heatmap на тот же лист через `to_excel(startrow=...)` (как donor); возвращает `20 + N` колонок; сохраняет donor-compatible ограничения по `wins_count` tie-bias и `above_median_count >= median`; не удаляет пустые бакеты; допускает NaN-ячейки в heatmap; не включает `stability_zone` и не вводит `heatmap_value_column`.

**Принципиальное отличие от доноров**: в donor медианная матрица строится по `step.top_entries` — подвыборке top-K кандидатов на каждом WF-шаге. В 3.0 источник данных — **pre-gate** `step_oos_long`, полная таблица OOS-результатов для **всех** точек сетки на **всех** шагах WF, отфильтрованная до `step_status == "ok"` (до наложения step gates) и очищенная от `INVALID_METRIC_VALUE`. Pre-gate семантика делает статистическую базу builder'а ближе к donor'у: donor'овские `top_entries` тоже не проходили через gates.

**Выход**: новый лист в XLSX-книге, идущий после `WF_Train_Trades`, содержащий:
- Основную таблицу: 1 строка на бакет, отсортированную по `bucket_stability_score` DESC, затем `(atr_bucket ASC, mult_bucket_ticks ASC)` для deterministic tie-break.
- Heatmap: pivot по `atr_bucket` × `mult_bucket_ticks`, value = `bucket_stability_score`.
- Корректное поведение на пустых данных (placeholder vs normal sheet — truth-table §3.7).

## 3. Специфика 3.0, которую нельзя потерять

### 3.1 Pre-gate `step_oos_long` как источник, `ok`-фильтр + sentinel-cleaning как обязательный агрегационный pipeline

Входной DataFrame — **pre-gate** `step_oos_long`:
- Это DataFrame **до** вызова `apply_step_gates()`.
- Каждая строка = одна оценённая комбинация `(grid_point_id, wf_step)`.
- Содержит колонки: `grid_point_id`, `atr_period`, `multiplier`, `trade_mode`, `wf_step`, `step_status`, `sum_pnl_pct`, и все остальные метрики.
- Покрытие: **все** точки сетки × **все** WF-шаги.
- Статусы на этом этапе: `ok`, `no_trades`, `insufficient_bars`, `runtime_error`. Статус `gate_failed` **ещё не существует** (gates не применены).

**Почему pre-gate, а не post-gate**: в donor'е `top_entries` собираются без каких-либо gates. Если builder в 3.0 получит post-gate данные, строки с `gate_failed` (валидные метрики, но < min_trades или > max_drawdown) будут исключены ok-фильтром. Это сужает выборку относительно donor'а, смещает медианы и scores. Pre-gate семантика = максимальное приближение к donor.

**Naming convention в pipeline**: pre-gate DataFrame обозначается `step_oos_long_pre_gate` в `orchestrator.py` для явного отличия от `result.step_oos_long` (post-gate). Builder **всегда** получает `step_oos_long_pre_gate`.

**Обязательный агрегационный pipeline** (выполняется в указанном порядке):

1. **Валидация обязательных колонок** (fail-fast): проверить наличие `wf_step`, `step_status`, `atr_period`, `multiplier`, `sum_pnl_pct`, `trade_mode` в DataFrame. Отсутствие любой → `ValueError` с перечислением отсутствующих колонок.
2. **Валидация уникальности** (fail-fast): `step_oos_long.duplicated(subset=["grid_point_id", "wf_step"]).any()` → `True` → `ValueError`. Дубликаты тихо искажают медианы, std и pct_params_positive_pnl через двойной учёт.
3. **trade_mode guard**: `step_oos_long["trade_mode"].nunique() > 1` → `ValueError`. Проверка выполняется на полном `step_oos_long`, до ok-фильтрации.
4. **total_steps capture**: `total_steps = step_oos_long["wf_step"].nunique()` — определяется из полного `step_oos_long`, до ok-фильтрации. Используется как знаменатель во всех ratio/score формулах.
5. **wf_step continuity guard**: `set(step_oos_long["wf_step"].unique()) == set(range(1, total_steps + 1))` → если не совпадает → `ValueError` с перечислением отсутствующих/лишних шагов. Защита от повреждённого входа и от 0-based indexing.
6. **ok-filter**: `ok_df = step_oos_long[step_oos_long["step_status"] == "ok"].copy()` — **обязательный `.copy()`** для safe in-place mutation.
7. **sentinel-cleaning**: `ok_df[metric_column] = ok_df[metric_column].replace(INVALID_METRIC_VALUE, np.nan)` — **обязательное присвоение обратно** в колонку (без assign-back = no-op в pandas). Применяется к:
   - step-level median computation (`Step1`…`StepN`);
   - raw value accumulation для `std_bucket`;
   - raw value accumulation для `pct_params_positive_pnl`.

Для MVP (`metric_column="sum_pnl_pct"`) достаточно `INVALID_METRIC_VALUE → NaN`. Обработка `MAX_VALID_METRIC` capping не требуется, т.к. `sum_pnl_pct` не входит в `_CAPPED_METRICS`.

### 3.2 Bucket assignment

Правила бакетизации:
- `atr_bucket = round(atr_period / atr_bucket_step) * atr_bucket_step`
- `mult_bucket_ticks = round(multiplier / mult_bucket_step)`

Параметры берутся из `BucketConfig`:
- `atr_bucket_step: int = 2`
- `mult_bucket_step: float = 0.2`
- `min_buckets_for_median: int = 5`

Bucket assignment покрывает **весь search space** из `config.optimization.atr_period_range` × `config.optimization.multiplier_range`, а не только наблюдённые бакеты. Полный набор бакетов генерируется детерминировано из конфига.

**Контракт**: builder работает только при одном `trade_mode` в `step_oos_long`. Bucket key = `(atr_bucket, mult_bucket_ticks)`, без `trade_mode` компонента.

**`bucket_size`** вычисляется перечислением **физических grid points** по `optimization.atr_period_range`, `optimization.multiplier_range`, `optimization.multiplier_step`, затем каждая точка группируется в бакет через `bucket.atr_bucket_step`, `bucket.mult_bucket_step`. Инвариант: `sum(bucket_sizes.values()) == len(enumerate_grid(config))` при single `trade_mode`.

### 3.3 Step-ячейки

Для каждого бакета `(ab, mt)` на шаге `wf_step = s` (1-based):
1. Отфильтровать из `ok_df` строки: `wf_step == s` AND `(atr_bucket, mult_bucket_ticks) == (ab, mt)`.
2. Sentinel-cleaning уже применён к `ok_df` на этапе §3.1 п.7.
3. Значение `Step{s}` = `median(metric_column)` по очищенным строкам (игнорируя NaN).
4. Если valid values = 0 → `NaN`.

Column naming: `Step1`, `Step2`, …, `StepN` (1-based, 3.0-native). Прямой mapping: `wf_step` → `Step{wf_step}`, без сдвига.

### 3.4 Производные колонки

**Обязательный шаг перед вычислением derived metrics**: каждая Step-колонка проходит `pd.to_numeric(bdf[col], errors="coerce")`. Защита от `object` dtype, как в donor'е (строки 2153, 2258).

Финальный порядок колонок в выходном DataFrame (**20 + N** колонок, где N = `total_steps`):

**Блок A — идентификация (5 фиксированных колонок):**

| # | Колонка | Формула / источник |
|---|---|---|
| 1 | `bucket_param` | Human-readable label: `"ATR 10–12, M 2.0–2.2"` (en-dash U+2013, mult `.1f`) |
| 2 | `bucket_key` | `f"{atr_bucket}_{mult_bucket_ticks}"` |
| 3 | `atr_bucket` | Из bucket assignment |
| 4 | `mult_bucket_ticks` | Из bucket assignment |
| 5 | `bucket_size` | Число физических grid points в бакете (из конфига, через `optimization.multiplier_step`) |

**Блок B — Step-колонки (N колонок):**

| # | Колонка | Формула / источник | Агрегационный слой |
|---|---|---|---|
| 6..5+N | `Step1`…`StepN` | `median(metric_column)` ok-строк шага в бакете, после sentinel-cleaning | ok-only + sentinel-clean |

**Блок C — производные метрики (15 фиксированных колонок):**

| # | Колонка | Формула / источник | Агрегационный слой | Zero-div fallback |
|---|---|---|---|---|
| 6+N | `bucket_presence_steps` | CSV-строка шагов формата `"Step1,Step3,Step5"`, где бакет имеет ≥1 ok+valid строку | ok-only | `""` |
| 7+N | `mean_oos_pnl` | `mean(Step1..StepN)` по non-NaN Step-значениям | ok-only (производная от Step) | `NaN` |
| 8+N | `std_bucket` | Population std (ddof=0) всех raw `metric_column` ok-строк по всем шагам/параметрам в бакете, после sentinel-cleaning | ok-only + sentinel-clean | `NaN` |
| 9+N | `pct_params_positive_pnl` | Доля параметров в бакете, чей mean `metric_column` > 0 по ok-шагам (после sentinel-cleaning). **Параметр = уникальная комбинация `(atr_period, multiplier)` внутри бакета** (как donor: `(atr_p, round(mult_v, 6))`; в 3.0 multiplier уже canonical, дополнительного round не нужно). Параметр без valid ok-строк исключается из знаменателя | ok-only + sentinel-clean | `NaN` |
| 10+N | `wins_count` | Число шагов, где медиана бакета = максимальная среди всех бакетов. **MVP-ограничение**: при tie — один winner (`idxmax`, первый по порядку строк) | ok-only (производная от Step) | `0` |
| 11+N | `win_steps` | CSV-строка выигранных шагов формата `"Step1,Step4"` | ok-only | `""` |
| 12+N | `top3_count` | Число шагов, где медиана бакета в top-3. При tie — `rank(ascending=False, method="min")`, все tied включены | ok-only (производная от Step) | `0` |
| 13+N | `above_median_count` | Число eligible шагов, где медиана бакета >= медианы медиан всех бакетов. **MVP-ограничение**: `>=` (donor-compatible) | ok-only + `min_buckets_for_median` filter | `0` |
| 14+N | `above_median_ratio` | `above_median_count / total_steps` | — | `0.0` (donor) |
| 15+N | `presence_count` | Число шагов с ≥1 ok+valid строкой в бакете | ok-only | `0` |
| 16+N | `above_median_ratio_present` | `above_median_count / presence_count` (diagnostic) | — | `0.0` (donor) |
| 17+N | `eligible_median_steps_count` | Число шагов, где бакет present AND шаг имеет >= `min_buckets_for_median` non-NaN бакетов | ok-only | `0` |
| 18+N | `above_median_ratio_eligible` | `above_median_count / eligible_median_steps_count` (diagnostic) | — | `0.0` (donor) |
| 19+N | `bucket_stability_score` | `round(0.6 * (presence_count / total_steps) + 0.4 * (above_median_count / total_steps), 6)` | — | `0.0` |
| 20+N | `zone_dominance_score` | `round(0.4 * (presence_count / total_steps) + 0.3 * (top3_count / total_steps) + 0.3 * (wins_count / total_steps), 6)` | — | `0.0` |

**Замечания по derived metrics в full-grid:**
- `pct_params_positive_pnl`: в donor — conditional на top-K presence; ключ = `(atr_period, multiplier)`. В 3.0 — conditional на ok-presence (все ok-шаги параметра); ключ = `(atr_period, multiplier)`. Формула не меняется, меняется только входной набор данных.
- `std_bucket`: population std (ddof=0) по **raw** ok+valid значениям `metric_column` (не по Step-медианам). Требует накопления raw values параллельно с median computation.
- `wins_count`: tie-biased в пользу бакета с меньшим `(atr_bucket, mult_bucket_ticks)` из-за `idxmax`. Зафиксировано как MVP-ограничение.
- `above_median_count`: `>=` означает ≥50% бакетов на каждом eligible шаге будут "above median". Зафиксировано как MVP-ограничение; diagnostic columns `above_median_ratio_present` и `above_median_ratio_eligible` доступны для альтернативного анализа.
- **Все scores округляются `round(6)`** (как donor), а не `round(4)`.
- **Zero-division fallback**: diagnostic ratios (`above_median_ratio_present`, `above_median_ratio_eligible`, `above_median_ratio`) → `0.0` при нулевом знаменателе (donor-compatible). Semantic: `NaN` точнее, но donor parity приоритетнее. TODO Phase 2.

### 3.5 Heatmap

- Строки: `atr_bucket` (sorted ASC), с human-readable labels `"12–14"` (en-dash).
- Колонки: `mult_bucket_ticks` (sorted ASC), с labels `"2.0–2.2"` (`.1f`).
- Value: `bucket_stability_score` (hardcoded в MVP).
- Conditional formatting: 3-color scale (white → yellow → green).
- NaN-ячейки **допустимы и ожидаемы** (бакеты с `presence_count = 0`). Не применять `fillna`/`dropna`.
- `heatmap_value_column` в конфиг и API **не добавляется** в MVP.
- `stability_zone` (top-30% zone summary) **не входит в MVP**.

### 3.6 Сортировка основной таблицы

**Primary sort**: `bucket_stability_score` DESC.

**Deterministic tie-break** (secondary): при равных `bucket_stability_score` — `atr_bucket ASC`, затем `mult_bucket_ticks ASC`. Это обеспечивает:
- Воспроизводимый порядок строк при каждом запуске.
- Стабильные golden tests.
- Предсказуемость для пользователя (при прочих равных — меньший ATR, меньший multiplier идёт первым).

Полная сортировка: `sort_values(["bucket_stability_score", "atr_bucket", "mult_bucket_ticks"], ascending=[False, True, True])`.

### 3.7 Truth-table: placeholder vs normal sheet

| Условие на входе `bucket_matrix_median` | Тип листа | Содержимое |
|---|---|---|
| `None` (builder не вызван или бросил исключение) | **Placeholder** | Лист с заголовками + freeze panes, 0 data rows |
| Truly empty DataFrame (0 rows, 0 cols; невозможен при валидном конфиге, только если `step_oos_long` пуст) | **Placeholder** | Лист с заголовками + freeze panes, 0 data rows |
| Full-grid DataFrame с NaN/0 (вход непуст, но 0 ok-строк, или все ok-строки — sentinel) | **Normal sheet** | Полный bucket grid, все Steps = NaN, все scores = 0.0, heatmap с NaN |
| Normal DataFrame (есть ok-данные) | **Normal sheet** | Полная таблица + heatmap |

**Проверка в sheet writer**: `if bucket_matrix_df is None or len(bucket_matrix_df) == 0` → placeholder. Иначе → normal sheet.

## 4. Разрыв между donor-логикой и 3.0

### 4.1 Источник данных: `top_entries` vs pre-gate full-grid + ok-filter + sentinel-clean

| Аспект | Donor | 3.0 |
|---|---|---|
| Источник | `result.steps[i].top_entries` (top-K подвыборка) | Pre-gate `step_oos_long` (полный грид, до step gates) |
| Фильтрация | Нет (`top_entries` уже отфильтрован) | `step_status == "ok"` обязателен; `gate_failed` **не существует** на pre-gate стадии |
| Sentinel-cleaning | `entry.oos_sum_pnl_pct is None` → skip; `-999.0` не фильтруется | `INVALID_METRIC_VALUE → NaN` обязателен после ok-фильтра, через `.copy()` + assign-back |
| `total_steps` | `len(result.steps)` — из WF result object | `step_oos_long["wf_step"].nunique()` — из полного pre-gate DataFrame до ok-фильтра, с continuity guard |
| trade_mode | Не проверяется (single mode предполагается) | Fail-fast guard на полном `step_oos_long` до ok-фильтра |
| Обязательные колонки | Не проверяются (duck typing на entry attributes) | Fail-fast валидация 6 обязательных колонок (§3.1 п.1) |
| Уникальность | Не проверяется (top_entries генерируются детерминировано) | Fail-fast guard на дубликаты `(grid_point_id, wf_step)` (§3.1 п.2) |
| Gate semantics | Gates не применяются к top_entries | Pre-gate: `gate_failed` ещё не существует → ok-filter видит только «настоящий» `ok` |
| Покрытие | Бакет может отсутствовать, если ни один параметр не в top-K | Бакет может «отсутствовать» (presence=0), если все его строки non-ok или sentinel |
| `presence_count` | < total_steps (типично) | < total_steps (возможно) |
| `above_median_ratio` denominator | `total_steps` | `total_steps` (сохранено; known double-penalty при absence, TODO Phase 2) |
| `pct_params_positive_pnl` key | `(atr_period, round(multiplier, 6))` | `(atr_period, multiplier)` — multiplier уже canonical (tick-backed), round не нужен |
| Step naming | `Step0…Step(N-1)` (0-based) | `Step1…StepN` (1-based) |
| CSV label format | `WF0`, `WF1`… (0-based) | `Step1`, `Step2`… (1-based, совпадает с колонками) |
| Score rounding | `round(score, 6)` | `round(score, 6)` — **сохранено** |
| `pd.to_numeric(errors="coerce")` | Применяется перед rank/median/comparison | **Сохранено** |
| Zero-div fallback | `0.0` для diagnostic ratios | `0.0` — **сохранено** (donor-compatible) |
| Formatting | `format_excel_export_df` (round 4 для METRIC_COLS) | Локальная `_format_bucket_df` (round 6 для scores, Int64 для bucket int cols) |

**Адаптация**: формулы **не меняются**. Sentinel-cleaning добавляется как новый обязательный шаг. `total_steps` фиксируется до ok-фильтрации с continuity guard.

### 5.2 Детали по модулям

**`wf_grid/config/schema.py`** — расширение конфига:
- Новый dataclass `BucketConfig`:
  - `atr_bucket_step: int = 2`
  - `mult_bucket_step: float = 0.2`
  - `min_buckets_for_median: int = 5`
- Новое поле `bucket: BucketConfig = field(default_factory=BucketConfig)` в `GridConfig`.
- Loader (`loader.py`) валидирует и парсит `bucket:` секцию из YAML; при отсутствии секции — дефолты. Валидация: `atr_bucket_step > 0`; `mult_bucket_step > 0`; `min_buckets_for_median >= 1`.
- Loader выдаёт **warning** (не error) при несовместимости `mult_bucket_step` и `multiplier_step` (если `mult_bucket_step` не является целым кратным `multiplier_step` с допуском 1e-9).

**`wf_grid/bucket/assignment.py`** — bucket assignment utilities:
- `apply_param_buckets(df, atr_bucket_step, mult_bucket_step) → df` — добавляет `atr_bucket`, `mult_bucket_ticks`.
- `generate_full_bucket_grid(config) → List[Tuple[int, int]]` — полный набор бакетов из search-space.
- `compute_expected_bucket_sizes(config) → Dict[Tuple[int,int], int]` — размер каждого бакета. **Контракт**: перечисляет physical grid points по `config.optimization.atr_period_range`, `config.optimization.multiplier_range`, `config.optimization.multiplier_step`; группирует по `config.bucket.atr_bucket_step`, `config.bucket.mult_bucket_step`. Инвариант: `sum(sizes.values()) == len(enumerate_grid(config))` при single `trade_mode`. Нарушение инварианта → `ValueError`.
- `format_bucket_label(ab, mt, atr_step, mult_step) → str` — контракт: `"ATR {lo}–{hi}, M {mlo}–{mhi}"`, en-dash U+2013.
- `format_atr_range(ab, atr_step) → str` — контракт: `"{ab}–{ab+atr_step}"`, en-dash.
- `format_mult_range(mt, mult_step) → str` — контракт: `"{mt*mult_step:.1f}–{(mt+1)*mult_step:.1f}"`, en-dash, всегда 1 десятичный знак.

**`wf_grid/bucket/median_matrix_builder.py`** — core builder:
- `build_median_bucket_matrix(step_oos_long, config, *, metric_column="sum_pnl_pct") → pd.DataFrame`
- Входы: **pre-gate** `step_oos_long` (полный грид), `GridConfig`.
- Обязательный pipeline внутри (§3.1):
  1. `metric_column != "sum_pnl_pct"` → `NotImplementedError`
  2. Валидация обязательных колонок: `wf_step`, `step_status`, `atr_period`, `multiplier`, `sum_pnl_pct`, `trade_mode` → `ValueError` при отсутствии
  3. Дубликат-guard: `duplicated(subset=["grid_point_id", "wf_step"])` → `ValueError`
  4. trade_mode guard на полном `step_oos_long`
  5. `total_steps = step_oos_long["wf_step"].nunique()` — до ok-фильтра
  6. wf_step continuity guard: множество шагов = `{1, 2, ..., total_steps}` → `ValueError` иначе
  7. `ok_df = step_oos_long[step_oos_long["step_status"] == "ok"].copy()`
  8. Sentinel-cleaning: `ok_df[metric_column] = ok_df[metric_column].replace(INVALID_METRIC_VALUE, np.nan)`
  9. Bucket assignment, full bucket grid generation, median aggregation, derived metrics
  10. `pd.to_numeric(errors="coerce")` на каждую Step-колонку перед derived metrics
  11. Сортировка: `bucket_stability_score DESC, atr_bucket ASC, mult_bucket_ticks ASC`
- Выход: DataFrame с `20 + N` колонками (блоки A + B + C из §3.4).
- **Edge cases**:
  - Пустой вход (0 строк) → пустой DataFrame.
  - Непустой вход, 0 ok-строк → **полный bucket grid** с NaN Steps и 0.0/NaN derived metrics (как donor).
  - Непустой вход, все ok-строки sentinel → **полный bucket grid** с NaN Steps и 0.0/NaN derived metrics.
- Оригинальный `step_oos_long` **не мутируется** (sentinel-cleaning работает на `.copy()`).

**`wf_grid/bucket/heatmap_builder.py`** — heatmap:
- `build_stability_heatmap(bucket_matrix_df, atr_bucket_step, mult_bucket_step) → pd.DataFrame`
- Hardcoded value = `bucket_stability_score`.
- NaN-ячейки допустимы; не применять `fillna`/`dropna`.
- **Параметры обязательны** (не default kwargs) — fix donor'овского бага, где heatmap builder использовал hardcoded defaults вместо config-resolved values.

**`wf_grid/export/bucket_sheet_writer.py`** — Excel sheet writer:
- `write_bucket_matrix_median_sheet(bucket_matrix_df, writer, atr_bucket_step, mult_bucket_step) → None`
- **Локальная функция `_format_bucket_df(df)`** (не `format_excel_export_df`):
  - `_BUCKET_INTEGER_COLS`: `wins_count`, `top3_count`, `presence_count`, `above_median_count`, `eligible_median_steps_count`, `bucket_size` → `Int64` через `_safe_to_int64`.
  - `_BUCKET_SCORE_COLS`: `bucket_stability_score`, `zone_dominance_score` → `round(6)`.
  - `_BUCKET_METRIC_COLS`: `mean_oos_pnl`, `std_bucket`, `pct_params_positive_pnl`, `above_median_ratio`, `above_median_ratio_present`, `above_median_ratio_eligible` → `round(6)`.
  - `atr_bucket`, `mult_bucket_ticks` → `Int64` (уже в глобальном `INTEGER_LIKE_COLS`, но дублируем локально для self-contained).
  - Step-колонки `Step1..StepN` → `round(6)`.
  - Остальное — `format_excel_export_df` по умолчанию.
- Технический контракт записи:
  - Main table: `_format_bucket_df(display_df).to_excel(writer, sheet_name="BucketMatrix_Median", index=False)`.
  - Затем: `ws = writer.sheets["BucketMatrix_Median"]`.
  - Title row + heatmap: через `to_excel(startrow=...)` (как donor — три вызова, не cell-by-cell).
  - `autofilter`: `A1:{last_col}1` — header-only (паттерн 3.0; known divergence от donor'а, UX-only, зафиксировано).
  - Conditional formatting (ColorScaleRule white → yellow → green): применяется **только** к диапазону heatmap.
  - Freeze panes: E2 (freeze блок A первые 4 колонки + header row).
  - Column widths, number formats (Step: `0.000000`, score: `0.000000`, pnl: `0.000000`, int: `0`).
  - **Placeholder mode** (truth-table §3.7): `bucket_matrix_df is None or len(bucket_matrix_df) == 0` → лист с заголовками + freeze panes, 0 data rows.

### 5.3 Интеграция в pipeline

В `orchestrator.py`, **между шагом 8 (collection) и шагом 8a (step gates)**:

1. Сохранить pre-gate DataFrame в явную переменную: `step_oos_long_pre_gate = step_oos_long` (до `apply_step_gates`).
2. Вызов builder'а обёрнут в **изолированный** `try/except`:
   - `try`: `result.bucket_matrix_median = build_median_bucket_matrix(step_oos_long_pre_gate, config)`
   - `except Exception as exc`: `logger.warning("Bucket matrix build failed, continuing pipeline: %s", exc)` → `result.bucket_matrix_median = None`
3. Затем — `step_oos_long = apply_step_gates(step_oos_long_pre_gate, config)` → `result.step_oos_long = step_oos_long` — pipeline продолжается штатно.

В `PipelineResult`: новое поле `bucket_matrix_median: Optional[pd.DataFrame] = None`.

В `export_workbook()`:
1. Новый параметр: `bucket_matrix_median: Optional[pd.DataFrame] = None`.
2. После записи `WF_Train_Trades` вызвать `write_bucket_matrix_median_sheet(...)`.
3. При `None` или пустом DataFrame → placeholder лист (truth-table §3.7).

### 5.4 Не нужен

- Отдельный builder pattern (класс) — достаточно функций.
- Промежуточный dataclass / typed result — DataFrame с фиксированным набором колонок и проверяемый тестами.
- `stability_zone` — исключён из MVP.
- `heatmap_value_column` параметр — исключён из MVP.
- Cell-by-cell openpyxl API для heatmap — donor'овский `to_excel(startrow=...)` корректен и проще.
- Прямой вызов `format_excel_export_df` для bucket sheet — используется локальная `_format_bucket_df`.

## 6. Пошаговый Implementation Plan

### Шаг 1. Расширить `GridConfig` — добавить `BucketConfig`

- **Цель**: определить конфигурационные параметры бакетизации в централизованной схеме.
- **Входы**: `schema.py`, `loader.py`, `config.yaml`.
- **Выходы**: `BucketConfig` dataclass в `schema.py`; парсинг и валидация в `loader.py`; секция `bucket:` в `config.yaml`.
- **Состав `BucketConfig`**: `atr_bucket_step: int = 2`, `mult_bucket_step: float = 0.2`, `min_buckets_for_median: int = 5`.
- **Валидация в loader**: `atr_bucket_step > 0`; `mult_bucket_step > 0`; `min_buckets_for_median >= 1`.
- **Warning в loader**: если `mult_bucket_step` не является целым кратным `multiplier_step` (с допуском 1e-9) → `logger.warning(...)` (не ConfigError). Аналогично если `atr_bucket_step` > `atr_period_range` span.
- **Риски**: имя `bucket` может потенциально конфликтовать с будущими секциями — проверить, что не пересекается.
- **Критерий готовности**: `load_grid_config()` корректно парсит конфиг с и без секции `bucket:`; дефолты работают; невалидные значения отклоняются валидатором; несовместимость step'ов → warning.

### Шаг 2. Реализовать `wf_grid/bucket/assignment.py` — bucket assignment

- **Цель**: чистая реализация bucket assignment без зависимости от donor `scoring.aggregation`.
- **Входы**: `atr_period`, `multiplier`, `atr_bucket_step`, `mult_bucket_step`, `GridConfig`.
- **Выходы**: функции `apply_param_buckets`, `generate_full_bucket_grid`, `compute_expected_bucket_sizes`, `format_*` helpers.
- **Ключевой контракт `compute_expected_bucket_sizes`**: перечисляет grid points по `optimization.multiplier_step`, группирует по `bucket.*_step`. Инвариант: `sum(sizes) == len(enumerate_grid(config))` при single trade_mode. Нарушение → `ValueError`.
- **Контракт format-функций**:
  - Разделитель — en-dash `–` (U+2013), **не** дефис `-`.
  - `format_atr_range(ab, step)` → `"{ab}–{ab+step}"`.
  - `format_mult_range(mt, mult_step)` → `"{mt*mult_step:.1f}–{(mt+1)*mult_step:.1f}"` — всегда 1 десятичный знак.
  - `format_bucket_label(ab, mt, atr_step, mult_step)` → `"ATR {atr_range}, M {mult_range}"`.
- **Риски**: дрейф формулы bucket assignment относительно donor.
- **Критерий готовности**: unit-tests подтверждают parity bucket assignment с donor; полный набор бакетов генерируется; сумма expected sizes == total grid points; format helpers выдают корректные строки с en-dash и `.1f` precision.

### Шаг 3. Реализовать `wf_grid/bucket/median_matrix_builder.py` — core builder

- **Цель**: построить полную медианную матрицу из pre-gate `step_oos_long` по `ok`-строкам с sentinel-cleaning.
- **Входы**: **pre-gate** `step_oos_long: pd.DataFrame`, `GridConfig`, `metric_column: str = "sum_pnl_pct"`.
- **Выходы**: DataFrame с `20 + N` колонками.
- **Обязательный pipeline**:
  1. `metric_column != "sum_pnl_pct"` → `NotImplementedError`.
  2. Валидация обязательных колонок: `wf_step`, `step_status`, `atr_period`, `multiplier`, `sum_pnl_pct`, `trade_mode`. Отсутствие любой → `ValueError` с перечислением missing.
  3. Дубликат-guard: `step_oos_long.duplicated(subset=["grid_point_id", "wf_step"]).any()` → `ValueError`.
  4. trade_mode guard на полном `step_oos_long`: `nunique() > 1` → `ValueError`.
  5. `total_steps = step_oos_long["wf_step"].nunique()` — **до** ok-фильтра.
  6. wf_step continuity guard: `set(wf_step.unique()) != set(range(1, total_steps + 1))` → `ValueError`.
  7. `ok_df = step_oos_long[step_oos_long["step_status"] == "ok"].copy()` — **обязательный `.copy()`**.
  8. Sentinel-cleaning: `ok_df[metric_column] = ok_df[metric_column].replace(INVALID_METRIC_VALUE, np.nan)` — **обязательное присвоение обратно**.
  9. `apply_param_buckets(ok_df, ...)` — добавить `atr_bucket`, `mult_bucket_ticks`.
  10. `generate_full_bucket_grid(config)` — полный набор бакетов.
  11. Для каждого `(bucket, wf_step)`: собрать `metric_column` из ok+valid строк, вычислить median → `Step{wf_step}`. Пусто → NaN.
  12. **Параллельно** с п.11: накопить raw cleaned values по `(bucket, param, step)` для `std_bucket` и `pct_params_positive_pnl`. Ключ группировки «параметр» = `(atr_period, multiplier)`. Raw values тоже проходят sentinel-cleaning (уже применена к `ok_df` в п.8).
  13. `pd.to_numeric(errors="coerce")` на каждую Step-колонку перед derived metrics.
  14. Вычислить derived metrics (Шаг 4). Scores: `round(6)`.
  15. Сортировка: `bucket_stability_score DESC, atr_bucket ASC, mult_bucket_ticks ASC`.
- **Edge cases**:
  - Пустой вход (0 строк) → пустой DataFrame.
  - Непустой вход, 0 ok-строк → **полный bucket grid** из конфига с NaN Steps и 0.0/NaN derived metrics.
  - Непустой вход, все ok-строки sentinel → **полный bucket grid** с NaN Steps и 0.0/NaN derived metrics.
- **Риски**: производительность на больших гридах (приемлемо для Phase A масштабов).
- **Критерий готовности**: на синтетическом DataFrame builder возвращает все `20 + N` колонок; пустой вход → пустой DataFrame; непустой вход без ok-строк → полный grid с NaN/0; sentinel в ok-строках → NaN, не контаминация; оригинальный DataFrame не мутирован; дубликаты → ValueError; отсутствие колонок → ValueError.

### Шаг 4. Реализовать derived metrics

- **Цель**: корректно вычислить все 15 производных колонок поверх Step-матрицы + raw accumulated values.
- **Входы**: матрица с Step-колонками + raw accumulated values + `total_steps`.
- **Выходы**: все колонки блока C из §3.4.
- **Обязательный предшаг**: `pd.to_numeric(bdf[col], errors="coerce")` на каждую Step-колонку.
- **Ключевые формулы (зафиксированы)**:
  - `bucket_stability_score = round(0.6 * (presence_count / total_steps) + 0.4 * (above_median_count / total_steps), 6)`
  - `zone_dominance_score = round(0.4 * (presence_count / total_steps) + 0.3 * (top3_count / total_steps) + 0.3 * (wins_count / total_steps), 6)`
  - `presence_count` = число шагов с ≥1 ok+valid строкой в бакете (НЕ raw row count)
  - `above_median_count` = из eligible шагов (бакет present AND шаг имеет ≥ `min_buckets_for_median` non-NaN бакетов); сравнение `>=` (MVP-ограничение)
  - `wins_count` = один winner через `idxmax` (MVP-ограничение, tie-biased)
  - `pct_params_positive_pnl`: ключ группировки = `(atr_period, multiplier)` внутри бакета.
- **Zero-division fallbacks** (donor-compatible):
  - `above_median_ratio`: `0.0` при `total_steps == 0`
  - `above_median_ratio_present`: `0.0` при `presence_count == 0`
  - `above_median_ratio_eligible`: `0.0` при `eligible_median_steps_count == 0`
- **CSV-лейблы**: `win_steps` = `"Step1,Step4"`; `bucket_presence_steps` = `"Step1,Step2,Step3"` (1-based, совпадает с колонками).
- **`total_steps`**: знаменатель берётся из полного pre-gate `step_oos_long` (определён в Шаге 3 п.5).
- **Критерий готовности**: unit-tests с hand-computed expected values для каждой derived metric; sentinel не влияет на результат; `total_steps` не зависит от ok-фильтра; zero-div cases возвращают `0.0`; `pd.to_numeric` coerce применён.

### Шаг 5. Реализовать `wf_grid/bucket/heatmap_builder.py`

- **Цель**: построить heatmap pivot-таблицу.
- **Входы**: DataFrame из builder, `atr_bucket_step` (обязательный), `mult_bucket_step` (обязательный).
- **Выходы**: pivot DataFrame с `bucket_stability_score` как value; human-readable range labels (en-dash, `.1f` mult).
- **Контракт**: NaN-ячейки допустимы; не применять `fillna`/`dropna`. Параметры `atr_bucket_step` и `mult_bucket_step` **обязательны** (не default kwargs) — fix donor'овского бага.
- **Критерий готовности**: pivot содержит корректные строки/колонки/значения; NaN для бакетов с `presence_count = 0`; пустой вход → пустой DataFrame.

### Шаг 6. Реализовать `wf_grid/export/bucket_sheet_writer.py`

- **Цель**: запись листа `BucketMatrix_Median` в Excel.
- **Входы**: `bucket_matrix_df`, `pd.ExcelWriter`, bucket config params.
- **Локальная функция `_format_bucket_df(df)`**:
  - `_BUCKET_INTEGER_COLS` → `Int64`.
  - `_BUCKET_SCORE_COLS` → `round(6)`.
  - `_BUCKET_METRIC_COLS` → `round(6)`.
  - Step-колонки → `round(6)`.
  - Не вызывает глобальный `format_excel_export_df` напрямую.
- **Технический контракт записи**:
  1. Main table: `_format_bucket_df(display_df).to_excel(writer, sheet_name="BucketMatrix_Median", index=False)`.
  2. `ws = writer.sheets["BucketMatrix_Median"]`.
  3. Title row + heatmap data: через `to_excel(startrow=...)` (как donor — три вызова).
  4. `autofilter`: `A1:{last_col}1` — header-only (паттерн 3.0).
  5. Conditional formatting (ColorScaleRule white → yellow → green): **только** на heatmap range.
  6. Freeze panes: `E2`.
  7. Column widths, number formats.
  8. **Truth-table** (§3.7): `None` или `len() == 0` → placeholder лист с заголовками + freeze panes.
- **Сортировка**: `bucket_stability_score DESC, atr_bucket ASC, mult_bucket_ticks ASC` (уже выполнена builder'ом; sheet writer **не** пересортирует, а проверяет инвариант: первая строка имеет max score).
- **Критерий готовности**: golden-test на структуру XLSX; пустой вход / None → placeholder лист; полный grid с NaN/0 → обычный лист; heatmap не затирает main table; autofilter не покрывает heatmap; integer-колонки = Int64; score-колонки = 6 десятичных знаков.

### Шаг 7. Интегрировать в `orchestrator.py` и `export_workbook()`

  - `PipelineResult`: новое поле `bucket_matrix_median: Optional[pd.DataFrame] = None`.
  - `orchestrator.py`:
    - **Между** collection (шаг 8) и step gates (шаг 8a):
    - `step_oos_long_pre_gate = step_oos_long` — **явное именование** pre-gate DataFrame.
    - Изолированный `try/except`:

      - `except Exception`: `logger.warning(...)` → `result.bucket_matrix_median = None`
    - Затем: `step_oos_long = apply_step_gates(step_oos_long_pre_gate, config)` — post-gate, как сейчас.

### 7.4 Ручные инварианты

| Инвариант | Проверка |
|---|---|
| Pre-gate semantics | Builder получает `step_oos_long_pre_gate`, не `result.step_oos_long` |
| Full-grid vs ok-subset | Число строк в ok-filtered `step_oos_long` ≤ total rows; бакеты могут иметь presence < total_steps |
| Sentinel-cleaning | ok-строка с `INVALID_METRIC_VALUE` → NaN в медиане, в raw std, в pct_params. `.copy()` + assign-back |
| Input immutability | Оригинальный `step_oos_long` не мутирован после builder |
| Ok-only semantics | Бакет с 5 raw строками, из которых 2 ok → median по 2 ok-строкам (после sentinel-cleaning), не по 5 |
| Пустой ok-subset на шаге | Бакет с нулём ok+valid строк → `NaN` в `Step{s}` |
| Non-empty input, zero ok rows | Непустой вход, 0 ok-строк → полный bucket grid с NaN/0, не пустой DataFrame |
| Сортировка строк | Primary: `bucket_stability_score` DESC; secondary: `atr_bucket ASC, mult_bucket_ticks ASC` |
| Deterministic sort | Два вызова с одинаковым входом → побитово идентичный порядок |
| Все колонки | Число колонок = `20 + N`; порядок = блок A + B + C |
| Step naming | Только `Step1…StepN`, отсутствуют `Step0` |
| CSV label format | `win_steps` и `bucket_presence_steps` содержат только `Step{N}` labels (1-based) |
| `total_steps` source | `total_steps` == `step_oos_long["wf_step"].nunique()` (до ok-фильтра), не из `ok_df` |
| `total_steps` with zero-ok step | Шаг полностью без ok → `total_steps` всё равно включает этот шаг |
| wf_step continuity | Множество шагов = {1, 2, ..., N} без пропусков |
| trade_mode guard | Два trade_mode в `step_oos_long` → `ValueError` до любых вычислений |
| Required columns guard | Отсутствие обязательной колонки → `ValueError` с перечислением |
| Duplicate guard | Дубликат (grid_point_id, wf_step) → `ValueError` |
| `bucket_size` invariant | `sum(bucket_sizes) == len(enumerate_grid(config))` |
| `bucket_size` uses grid step | Вычисление по `optimization.multiplier_step`, не по `bucket.mult_bucket_step` |
| Heatmap generation | Pivot shape == `(len(atr_buckets), len(mult_ticks))`; NaN допустимы |
| Пустой вход | `build_median_bucket_matrix(empty_df, config)` → пустой DataFrame, не exception |
| Детерминизм | Два вызова с одинаковым входом → побитово идентичный выход |
| Tie-handling | При 2 бакетах с одинаковой медианой: `wins_count` = 1 у первого (idxmax), `top3_count` = оба |
| `above_median_*` uses `>=` | Бакет == step_median → counted; at least 50% present buckets are "above" |
| Zero-div fallback | Diagnostic ratios → `0.0` при нулевом знаменателе (не NaN) |
| Score precision | `bucket_stability_score` и `zone_dominance_score` → `round(6)` |
| Formatting | Integer cols → Int64, score cols → round(6), не round(4) |
| Heatmap write | Main table не затёрта; autofilter не покрывает heatmap |
| Placeholder truth-table | None → placeholder; empty df → placeholder; full-grid NaN/0 → normal; normal → normal |
| Builder isolation | Exception в builder → warning + None + pipeline продолжается |
| `pct_params_positive_pnl` key | Группировка по `(atr_period, multiplier)`, не по бакету |
| `pd.to_numeric` coerce | Применён перед rank/median/comparison на Step-колонках |

## 8. Риски и зафиксированные проектные решения

### 8.1 Зафиксированные проектные решения

| # | Решение | Обоснование |
|---|---|---|
| D1 | Агрегация только по `step_status == "ok"` на **pre-gate** данных | Pre-gate семантика: ближе к donor; gates ещё не применены; ok-фильтр ≠ ok+gate_failed |
| D2 | Sentinel-cleaning `INVALID_METRIC_VALUE → NaN` после ok-фильтра через `.copy()` + assign-back | Для `sum_pnl_pct` cap-cleaning не требуется; pandas `replace()` без assign-back = no-op |
| D3 | trade_mode guard на полном pre-gate `step_oos_long` | Предотвращает смешивание разнонаправленных стратегий; проверка до ok-фильтра |
| D4 | Step naming: `Step1…StepN` (1-based) | `wf_step` в 3.0 = 1-based; summary wide = `S1…SN`; consistency |
| D5 | CSV labels: `Step1,Step4` (1-based, совпадает с колонками) | Donor `WF0` — 0-based и конфликтует с 3.0 naming |
| D6 | `total_steps` из полного pre-gate `step_oos_long` (`wf_step.nunique()`) с continuity guard | Шаг без ok-строк всё равно входит в знаменатель; guard: множество = {1..N} |
| D7 | `BucketConfig` в `GridConfig` | `GridConfig` — центральная схема; loader валидирует centralized |
| D8 | `bucket_size` через grid resolution + bucket resolution | Перечисление по `optimization.multiplier_step`, группировка по `bucket.mult_bucket_step` |
| D9 | `metric_column != "sum_pnl_pct"` → `NotImplementedError` | Derived metrics жёстко привязаны к PnL-like semantics |
| D10 | Heatmap value = `bucket_stability_score` | MVP; параметризация — follow-up |
| D11 | `stability_zone` не входит в MVP | Scope control |
| D12 | `heatmap_value_column` не добавляется | Scope control |
| D13 | Heatmap через `to_excel(startrow=...)` (как donor) | Donor'овский подход корректен: `to_excel(startrow=N)` пишет в строку N, не затирая предыдущие. Cell-by-cell API не нужен |
| D14 | Число колонок = `20 + N` | 5 фиксированных + N step + 15 фиксированных |
| D15 | Пустые бакеты (presence=0) сохраняются в output | Builder возвращает полный bucket grid; export не фильтрует |
| D16 | NaN допустимы в heatmap | Не применять fillna/dropna |
| D17 | `early_exit` — существующий guardrail | `BacktestConfig.early_exit_enabled = False`; loader отклоняет через `ConfigError` |
| D18 | Builder получает **pre-gate** `step_oos_long` | Pre-gate семантика: donor `top_entries` тоже не проходили gates. Переменная `step_oos_long_pre_gate` в orchestrator для явного отличия от `result.step_oos_long` |
| D19 | Builder изолирован от pipeline try/except | Собственный `try/except`; exception → warning + `None`; pipeline продолжается |
| D20 | Локальная `_format_bucket_df` вместо глобальной `format_excel_export_df` | `METRIC_COLS` содержит `bucket_stability_score` с round(4); нужен round(6). Собственные frozenset'ы |
| D21 | Deterministic tie-break sort | `bucket_stability_score DESC, atr_bucket ASC, mult_bucket_ticks ASC` — golden test stability |
| D22 | Fail-fast валидация обязательных колонок | `ValueError` при отсутствии `wf_step`, `step_status`, `atr_period`, `multiplier`, `sum_pnl_pct`, `trade_mode` |
| D23 | Fail-fast duplicate guard | `ValueError` при дубликатах `(grid_point_id, wf_step)` — предотвращение двойного учёта |
| D24 | `pd.to_numeric(errors="coerce")` перед derived metrics | Donor-compatible защитный слой; Step-колонки с object dtype не ломают rank/comparison |
| D25 | Zero-div fallback = `0.0` (donor-compatible) | Diagnostic ratios при нулевом знаменателе → `0.0`, не `NaN`. Phase 2 TODO: пересмотреть |
| D26 | Непустой вход + 0 ok-строк → полный bucket grid (не empty df) | Donor возвращает полный grid с NaN/0; sheet получает normal лист, не placeholder |

## 9. Definition of Done

Функционал `BucketMatrix_Median` считается завершённым, когда:

1. **Config**: `GridConfig` содержит `BucketConfig` с `atr_bucket_step`, `mult_bucket_step`, `min_buckets_for_median`; конфиг парсится из YAML и валидируется; дефолты работают при отсутствии секции `bucket:`; loader выдаёт warning при несовместимости `mult_bucket_step` / `multiplier_step`.

2. **Builder**: `build_median_bucket_matrix(step_oos_long, config, *, metric_column="sum_pnl_pct")`:
   - Fail-fast: `metric_column != "sum_pnl_pct"` → `NotImplementedError`.
   - Fail-fast: отсутствие обязательных колонок (`wf_step`, `step_status`, `atr_period`, `multiplier`, `sum_pnl_pct`, `trade_mode`) → `ValueError` с перечислением.
   - Fail-fast: дубликаты `(grid_point_id, wf_step)` → `ValueError`.
   - Fail-fast: `step_oos_long["trade_mode"].nunique() > 1` → `ValueError` (на полном df).
   - `total_steps` определяется из полного `step_oos_long` через `wf_step.nunique()` с continuity guard (множество = {1..N}).
   - Агрегирует **только `step_status == "ok"`** строки на **pre-gate** данных.
   - После ok-фильтра выполняет **sentinel-cleaning** через `.copy()` + assign-back.
   - `pd.to_numeric(errors="coerce")` на Step-колонки перед derived metrics.
   - `pct_params_positive_pnl` группирует по `(atr_period, multiplier)`.
   - Zero-div fallback = `0.0` для diagnostic ratios.
   - Scores: `round(6)`.
   - Сортировка: `bucket_stability_score DESC, atr_bucket ASC, mult_bucket_ticks ASC`.
   - Возвращает DataFrame с **`20 + N` колонками** (блоки A + B + C).
   - На пустом входе (0 строк) → пустой DataFrame.
   - На непустом входе без ok-строк / все sentinel → **полный bucket grid** с NaN/0.
   - Оригинальный DataFrame **не мутируется**.

3. **Heatmap**: `build_stability_heatmap()` принимает обязательные `atr_bucket_step`, `mult_bucket_step` (не default kwargs); возвращает корректный pivot с `bucket_stability_score` (hardcoded); human-readable labels (en-dash, `.1f`); NaN-ячейки допустимы.

4. **Excel**: лист `BucketMatrix_Median` записывается в XLSX-книгу с:
   - Основной таблицей через `_format_bucket_df()` → `to_excel()`. Integer cols → Int64. Scores → round(6). **Не** `format_excel_export_df`.
   - Heatmap section (title + pivot + conditional formatting) через `to_excel(startrow=...)` (как donor).
   - `autofilter` header-only; не покрывает heatmap.
   - Freeze panes E2, column widths, number formats.
   - Truth-table: None/empty → placeholder; full-grid NaN/0 → normal; normal → normal.

5. **Pipeline**: `orchestrator.py` вызывает builder на `step_oos_long_pre_gate` (до step gates), в изолированном try/except; exception → warning + `None` + pipeline продолжается. `result.step_oos_long` остаётся post-gate (не регрессия). `export_workbook()` включает лист после `WF_Train_Trades`.

6. **Scope control**: `stability_zone` не реализован; `heatmap_value_column` не добавлен; `metric_column` принимается builder-ом, но единственное допустимое значение = `"sum_pnl_pct"`; cell-by-cell API не используется.

7. **MVP-ограничения зафиксированы**: `wins_count` tie-biased (L1); `above_median_count` через `>=` (L2); пустые бакеты сохраняются (L3).

8. **Тесты**: все unit, integration и golden тесты из §7 проходят; включая:
   - Sentinel-cleaning подтверждена (`.copy()` + assign-back; ok-строка с `-999.0` → NaN, не контаминация; оригинальный df не мутирован).
   - `Step1…StepN` naming (без `Step0`).
   - CSV labels `Step1,Step4` (без `WF0`).
   - `total_steps` из полного pre-gate df с continuity guard.
   - trade_mode guard на полном df.
   - Required columns guard.
   - Duplicate guard.
   - `presence_count` по ok+valid presence.
   - `bucket_size` через `optimization.multiplier_step`.
   - Heatmap не затирает main table (через `to_excel(startrow=...)`).
   - `metric_column="sharpe"` → `NotImplementedError`.
   - `step_oos_long` с двумя trade_mode → `ValueError`.
   - Deterministic sort: tie-break по `atr_bucket ASC, mult_bucket_ticks ASC`.
   - Score precision 6 (не 4).
   - Integer cols → Int64.
   - `pd.to_numeric(errors="coerce")` на Step-колонках.
   - `pct_params_positive_pnl` по `(atr_period, multiplier)`.
   - Zero-div fallback → `0.0`.
   - Непустой вход + 0 ok-строк → полный grid (не empty df).
   - Builder exception → pipeline продолжается; placeholder лист.
   - Pre-gate semantics: `gate_failed` строки участвуют в медиане.
   - Placeholder truth-table: None, empty df, full-grid NaN, normal.

9. **Parity**: для контрольного subset (с идентичными входными данными, все ok, без sentinel) результаты 3.0 совпадают с donor output с точностью ≤ 1e-6 (с учётом Step0→Step1 shift и `WF0`→`Step1` label mapping; scores round(6) match).

10. **Docs**: `xlsx_export_spec.md` обновлён: §2.8 `BucketMatrix_Median`.

11. **Нет регрессий**: существующие тесты проходят (кроме обновлённых assertions на число листов в XLSX); `result.step_oos_long` остаётся post-gate. реализуй план по этапам:  1	BucketConfig в schema.py + парсинг/валидация в loader.py
